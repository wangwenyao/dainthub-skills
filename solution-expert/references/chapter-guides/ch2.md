# ch2 — 需求分析与功能范围

受众：业务方 + 技术方。功能描述用业务语言，非功能需求用量化指标。
读取：diagram-guides.md（生成功能架构图和业务流程图）

## 结构与写作规则

### 2.1 业务场景概述
AS-IS（现状）→ TO-BE（目标）的对比，用具体场景描述，不用抽象词汇。

### 2.2 功能架构
1. 引导段：说明系统按业务域划分为 N 个功能域
2. 功能架构表格：完整定义见 `@references/diagram-guides.md#功能架构图`（5 列标准格式：业务域/功能模块/主要功能点/优先级/本期范围）
3. 各域一句话职责说明

**本章特殊要求：**
- 必须填写"本期范围"列（即使客户未明确分期，也按 P0=本期、P1/P2=本期或后续迭代填写）
- 表格后补充"优先级判断依据"段，说明 P0/P1/P2 的具体判断逻辑（如"按业务流程关键性 + 客户明确指定"）

### 2.3 核心业务流程
仅流程审批类模块画流程图，最多 3 张。
调用 diagram-guides.md 中的业务流程图规则生成 Mermaid flowchart。
每张图后用文字补充说明分支条件和特殊规则。

纯数据管理类系统跳过本节。

### 2.4 非功能性需求

必须覆盖四类：

**性能指标表：**
| 指标 | 目标值 | 说明 |
|---|---|---|
| 并发用户数 | {值} 人 | 同时在线操作的用户数量 |
| 接口响应时间 | P95 < {值}秒 | 普通查询和操作接口 |
| 报表查询响应时间 | P95 < {值}秒 | 复杂统计查询 |

**可用性指标表：**
| 指标 | 目标值 |
|---|---|
| 系统可用性 | {值}%（年停机 < {换算}小时）|
| 故障切换时间 | < {值}秒 |

| 要求 | 说明 |
|---|---|
