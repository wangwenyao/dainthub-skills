# ch2 — 需求分析与业务架构

受众：业务方 + 技术方。功能描述用业务语言，非功能需求用量化指标。
读取：diagram-guides.md（生成功能架构图和业务流程图）

## 结构与写作规则

### 2.1 业务场景概述
AS-IS（现状）→ TO-BE（目标）的对比，用具体场景描述，不用抽象词汇。

### 2.2 功能架构
1. 引导段：说明系统按业务域划分为 N 个功能域
2. 功能架构表格（调用 diagram-guides.md 的功能架构图规则生成）
3. 各域一句话职责说明

优先级判断标准：
- P0：主业务流程上的功能，缺少则系统无法运行
- P1：核心业务功能，一期完成
- P2：辅助/优化类功能，可后续迭代

### 2.3 核心业务流程
仅流程审批类模块画流程图，最多 3 张。
调用 diagram-guides.md 中的业务流程图规则生成 Mermaid flowchart。
每张图后用文字补充说明分支条件和特殊规则。

纯数据管理类系统跳过本节。

### 2.4 非功能性需求
内容来源：客户明确提出的 > 推算的 > nfr-defaults.md 默认值（标注"默认假设"）。

必须覆盖三类：

**性能指标表：**
| 指标 | 目标值 | 说明 |
|---|---|---|
| 并发用户数 | {值} 人 | 同时在线操作的用户数量 |
| 接口响应时间 | P95 < {值}秒 | 普通查询和操作接口 |
| 报表查询响应时间 | P95 < {值}秒 | 复杂统计查询 |

**可用性指标表：**
| 指标 | 目标值 |
|---|---|
| 系统可用性 | {值}%（年停机 < {换算}小时）|
| 故障切换时间 | < {值}秒 |

**安全要求表：**
| 要求 | 说明 |
|---|---|
| 传输安全 | 全站 HTTPS |
| 身份认证 | JWT Token，有效期 {值}小时 |
| 权限控制 | RBAC，粒度到按钮级和数据行级 |
| 操作审计 | 所有增删改操作记录操作人、时间、变更内容，留存 {值}个月 |
| {行业特殊要求} | {如等保级别} |
