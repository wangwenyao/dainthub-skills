# ch1 — 项目概述与价值定位

受众：业务决策层。不出现技术术语。

## 结构与写作规则

### 1.0 方案说明（新增）

在文档最开头生成以下说明表格：

| 项 | 说明 |
|---|---|
| 文档版本 | v1.0 |
| 编制日期 | {日期} |
| 文档定位 | 项目启动阶段技术交底文档，作为开发基准和需求确认依据 |
| 主要受众 | 客户业务负责人、客户技术负责人、项目组成员 |
| 后续更新 | 需求变更需走变更管理流程，重大变更更新文档版本（见附录C）|

### 1.1 项目背景
- 用客户自己的语言描述现状和痛点（2-4 个痛点）
- 痛点要具体：❌"管理效率低下" ✅"采购申请通过邮件传递，审批状态不透明，平均审批周期 3-5 天"
- 如客户有旧系统，说明旧系统的主要问题和本次建设的触发原因
- 从客户输入中提取真实业务背景，不泛泛而谈

### 1.2 建设目标
- 3-5 条目标，与痛点一一对应
- 每条目标必须包含可验收的量化指标或明确的业务变化
- 格式："建立 X，实现 Y，使 Z 从 A 改善到 B"

### 1.3 核心价值（新增）

按以下四类描述系统建成后的核心价值，尽量量化，无法量化时描述具体变化：

| 价值维度 | 描述方式 |
|---|---|
| 效率提升 | {具体场景}从{当前状态}改善到{目标状态}，预计效率提升{百分比或时间} |
| 管理优化 | {具体管理问题}通过{系统能力}得到解决 |
| 数据赋能 | {数据分析能力}支持{决策场景} |
| 风险控制 | {风险点}通过{系统机制}得到管控 |

无法覆盖四类时，只写有实际内容的类别。

### 1.4 建设范围

用表格呈现，明确本次做什么、不做什么：

| 业务域 | 建设方式 | 说明 |
|---|---|---|
| {域名} | 新建 / 改造 / 集成 / 本期不涉及 | {一句话说明} |

**额外说明**：如有客户可能期待但本期不做的功能，在表格下方单独列出：
> 以下功能本期不在范围内，如有需要可在后续迭代中规划：{功能列表}

### 1.5 项目约束
- 时间约束（上线时间、关键里程碑）
- 技术约束（已有系统、对接接口、等保要求）
- 运行环境约束（部署环境、网络）

### 1.6 关键假设与风险（新增）

本方案基于以下关键假设，如假设不成立将影响项目进度或范围：

| 编号 | 假设内容 | 如不成立的影响 |
|---|---|---|
| H1 | {假设，如：客户在第2周提供测试服务器资源} | {影响，如：影响测试环境搭建，可能延期1-2周} |
| H2 | {假设，如：外部系统接口文档在第1周提供} | {影响，如：集成模块工期无法确认} |

假设来源：从输入质量检查的可推断缺失和需求补全的默认值中提取，每条假设在附录 A 中有对应的待确认事项。

## 禁忌
- 不出现：Vue、Spring Boot、JWT、Redis、微服务、分层架构等技术词汇
- 不写："采用先进技术"、"具有良好扩展性"等空洞表述
- 技术术语如必须出现，在附录B中给出业务语言解释
