## 结构与写作规则

### 4.1 应用模块结构

#### 4.1.1 后端组织方式

**设计思路：**

系统按业务域组织代码（按 Step 4 域模型输出的业务域），每个域独立管理自己的业务逻辑和数据访问。

**这样设计的收益：**

| 收益 | 说明 |
|------|------|
| 修改隔离 | 修改某个业务功能时，只需在该域内修改，不会波及其他域 |
| 团队协作 | 不同开发人员可以并行开发不同域，互不干扰 |
| 演进预留 | 后续系统演进时，可按域拆分为独立服务，无需大规模重构 |

**代码边界规则：**

- 一个域不能直接访问另一个域的数据，必须通过该域提供的接口
- 这避免了跨域修改导致的连锁影响

**域组织示例（用 Step 4 输出的实际业务域填充）：**

| 业务域 | 包含模块 | 核心职责 |
|--------|----------|----------|
| {核心业务域1} | {模块1}、{模块2}、{模块3} | {一句话职责} |
| {核心业务域2} | {模块1}、{模块2} | {一句话职责} |
| 系统管理域 | 用户、角色、权限、字典、日志 | 管理系统基础配置 |

---

#### 4.1.2 前端组织方式

**设计思路：**

前端代码按业务域组织页面，公共组件独立管理；用户登录后根据角色权限动态生成菜单和路由。详细组织原则与标准页面模式见 `@references/frontend-constraint.md`。

**这样设计的收益：**

| 收益 | 说明 |
|------|------|
| 界面简洁 | 不同角色的用户只看到自己权限范围内的功能 |
| 权限灵活 | 新增角色或调整权限后，用户刷新页面即可生效 |
| 维护简单 | 菜单配置在后端维护，前端不硬编码路由 |

#### 4.1.3 前后端协作约定

接口技术规范见第 3 章"3.4 横切关注点 - 接口规范"，本节说明开发过程中的协作机制。

**联调机制：**
- 接口文档由后端自动生成，前后端联调以此为准
- 接口变更需提前通知前端，预留联调时间
- 前端可根据接口文档自行生成 Mock 数据，不依赖后端环境

**文件上传处理：**
- 先上传文件获取文件标识，再提交业务数据（避免大文件上传失败影响业务数据）
- 文件存储在文件服务器，数据库只存文件标识和元数据（文件名、大小、类型）

### 4.2 各业务域设计

**每个域的描述结构：**

```
{域名称}（如：采购管理域）
域职责：一句话

[核心复杂模块] → 用设计思路四框架详细描述：
  核心问题定义
  约束下的设计思路（单体事务优势、中间件能力边界）
  关键设计判断（每条判断给出依据）
  边界与假设

  Domain 层判断：
  如模块包含以下复杂业务规则之一，说明是否引入 Domain 层：
  - 状态机（多状态、复杂转换规则）
  - 计费逻辑（多维度计算、复杂公式）
  - 复杂校验（跨字段、跨表、业务规则校验）
  如引入：说明 Domain 层封装的业务规则范围，以及与 Service 层的职责边界
  如不引入：说明业务规则在 Service 层实现的理由（如规则简单、无需独立测试）

[前端实现] → 参考 frontend-constraint.md 的标准页面模式，
  说明该域的主要页面类型，不展开组件细节

[后端实现] → 说明业务逻辑层职责和事务边界，
  与其他域的交互方式（只通过接口，不直接访问对方数据）
  如涉及中间件，说明使用场景（已在第 3 章定义职责，此处只说使用位置）

[标准实现模块] → 一句话描述：
  "{模块名}采用标准数据管理实现，前端对应{页面模式}，
  后端按分层架构组织，无特殊设计"
```

**按复杂模块类型的说明重点（来自 writing.md）：**

| 类型 | 重点说明 |
|---|---|
| 状态流转 | 状态枚举和转换规则；同一事务的操作范围；固定/可配置流程的选择理由 |
| 数据权限 | 权限维度；数据过滤实现方式；接口权限与数据权限的层次关系 |
| 定时任务 | 触发频率依据；幂等性设计；多实例防重机制 |
| 外部集成 | 集成方式选择依据；降级策略；数据一致性保障 |
