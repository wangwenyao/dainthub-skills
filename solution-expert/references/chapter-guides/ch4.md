## 结构与写作规则

### 4.1 应用模块结构

#### 4.1.1 后端组织方式

**设计思路：**

系统按业务域组织代码（按 Step 4 域模型输出的业务域），每个域独立管理自己的业务逻辑和数据访问。

**这样设计的收益：**

| 收益 | 说明 |
|------|------|
| 修改隔离 | 修改某个业务功能时，只需在该域内修改，不会波及其他域 |
| 团队协作 | 不同开发人员可以并行开发不同域，互不干扰 |
| 演进预留 | 后续系统演进时，可按域拆分为独立服务，无需大规模重构 |

**代码边界规则：**

- 一个域不能直接调用另一个域的数据库表，必须通过该域提供的接口
- 这避免了"改一个地方，影响一片"的代码混乱

**域组织示例（用 Step 4 输出的实际业务域填充）：**

| 业务域 | 包含模块 | 核心职责 |
|--------|----------|----------|
| {核心业务域1} | {模块1}、{模块2}、{模块3} | {一句话职责} |
| {核心业务域2} | {模块1}、{模块2} | {一句话职责} |
| 系统管理域 | 用户、角色、权限、字典、日志 | 管理系统基础配置 |

---

#### 4.1.2 前端组织方式

**设计思路：**

前端页面按业务功能组织（按 Step 4 域模型输出的业务域），用户登录后根据角色权限动态显示菜单和路由。

**这样设计的收益：**

| 收益 | 说明 |
|------|------|
| 界面简洁 | 不同角色的用户只看到自己权限范围内的功能 |
| 权限灵活 | 新增角色或调整权限后，用户刷新页面即可生效 |
| 维护简单 | 菜单配置在后端维护，前端不硬编码路由 |

**权限联动机制：**

```
用户登录 → 后端返回该用户的菜单权限 → 前端动态生成路由和菜单
         ↓
角色权限调整 → 用户刷新页面 → 菜单自动更新
```

#### 4.1.3 前后端协作约定

**接口调用规范：**

| 场景 | 约定 | 原因 |
|------|------|------|
| 数据查询 | 统一使用 POST 请求 + JSON 参数 | 支持复杂查询条件，便于扩展 |
| 分页查询 | 固定参数：pageNum、pageSize | 前后端分页逻辑统一 |
| 文件上传 | 先上传文件获取文件ID，再提交业务数据 | 大文件上传失败时不影响业务数据 |
| 错误处理 | 后端统一返回 {code, message, data} | 前端统一处理错误提示，用户体验一致 |

**联调机制：**

- 接口文档由后端自动生成，前后端联调以此为准
- 接口变更需提前通知前端，预留联调时间

### 4.2 各业务域设计

**每个域的描述结构：**

```
{域名称}（如：采购管理域）
域职责：一句话

[核心复杂模块] → 用设计思路四框架详细描述：
  核心问题定义
  约束下的设计思路（单体事务优势、中间件能力边界）
  关键设计判断（每条判断给出依据）
  边界与假设

[前端实现] → 参考 frontend-constraint.md 的标准页面模式，
  说明该域的主要页面类型，不展开组件 API 细节

[后端实现] → 说明 Service 层职责和事务边界，
  与其他域的交互方式（只通过 Service 接口）
  如涉及中间件，说明使用场景（已在第 3 章定义职责，此处只说使用位置）

[标准实现模块] → 一句话描述：
  "{模块名}采用标准 CRUD 实现，前端对应{页面模式}，
  后端按分层架构组织，无特殊设计"
```

**按复杂模块类型的说明重点（来自 writing.md）：**

| 类型 | 重点说明 |
|---|---|
| 状态流转 | 状态枚举和转换规则；同一事务的操作范围；固定/可配置流程的选择理由 |
| 数据权限 | 权限维度；SQL 层实现方式；接口权限与数据权限的层次关系 |
| 定时任务 | 触发频率依据；幂等性设计；Quartz 集群防重 |
| 外部集成 | 集成方式选择依据；降级策略；数据一致性保障 |
