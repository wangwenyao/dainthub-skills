# ch3 — 总体技术架构

受众：技术架构师。每个关键决策给出理由。
读取：backend-constraint.md、middleware-constraints.md、diagram-guides.md

## 结构与写作规则

### 3.1 架构设计原则
4-5 条原则，每条说明"是什么"和"在本项目中如何体现"：
- 业务域边界清晰（域间只通过 Service 接口交互）
- 无状态设计（应用层不保存会话状态，支持多实例扩展）
- 按需引入中间件（由具体业务问题驱动，不默认全部引入）
- 失败可降级（非核心组件故障时核心业务能降级运行）
- 设计留演进路径（单体按域分包，为渐进式服务化保留基础）

### 3.2 系统总体架构
1. 引导段（1-2 句）：前后端分离，整体分为展现层/接入层/应用层/数据层
2. 系统架构图（调用 diagram-guides.md 的系统架构图规则：Mermaid 示意 + 组件表格）
3. 前后端分离说明：接口契约、认证流转、独立部署
4. 如有外部集成：集成清单 + 每个集成的方向和方式

### 3.3 技术架构
1. 技术架构表格（调用 diagram-guides.md 的技术架构图规则）
2. 架构整体说明段（概括技术栈设计逻辑）
3. 单体架构选择说明（必须包含，调用 backend-constraint.md 的"单体架构选择的说明要点"）

> 注意：单体架构说明要点已定义在 backend-constraint.md 中，直接引用，不要重复撰写。

### 3.4 横切关注点
统一描述所有模块共用的基础能力（第 4 章各模块不重复）。
调用 backend-constraint.md 中的"横切关注点标准实现"表格。

每个横切能力按三要素描述（引入原因 + 设计决策 + 边界说明）。

### 3.5 中间件引入说明（如有引入）
调用 middleware-constraints.md 中对应中间件的规则。
每个中间件用三要素表达：引入原因（解决什么业务问题）+ 设计决策（关键判断）+ 边界说明（不负责什么）。
