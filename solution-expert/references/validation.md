# validation

## 规则

全部章节生成完成后执行，发现问题直接修正，不对用户暴露自查过程。

---

### 跨章节对应关系检查

**检查一：功能模块覆盖**

第 2 章功能架构表格中的每个功能模块，第 4 章应有对应描述。
- 核心复杂模块：必须有详细设计思路
- 标准实现模块：可合并描述（"以下模块均为标准 CRUD 实现，前端采用列表管理页模式"）
- 不允许完全不提及某个功能模块

**检查二：非功能需求响应**

第 2 章每条非功能需求，在第 3、4、5 章中应有对应的技术响应。
逐条确认：性能指标 → 部署方案是否匹配；可用性要求 → 是否有高可用设计；安全要求 → 是否有对应技术措施。

**检查三：中间件引用一致性**

第 3 章提到的每个中间件：
- 第 4 章的使用描述是否符合第 3 章定义的职责（Redis 不存主数据，Quartz 不处理实时触发）
- 第 5 章的部署方案是否包含了对应的服务器资源

**检查四：复杂模块说明完整性**

Step 4 标注的每个"核心复杂模块"，第 4 章是否有三要素（引入原因 + 设计决策 + 边界说明）的完整描述。

**检查五：规模与部署一致性**

第 2 章的并发用户数和可用性要求，与第 3 章"3.6 部署架构"的实例数量、负载均衡方案是否匹配。

---

**检查六：第 1 章假设与附录 A 一致性**

第 1 章"1.6 关键假设与风险"中的每条假设，必须在附录 A"待客户确认事项清单"中有对应条目。

**检查七：第 2 章数据留存与第 3 章备份策略一致性**

第 2 章"2.4 非功能性需求 - 数据留存要求表"中的留存期限，第 3 章"3.6.4 数据备份策略"的保留时间必须覆盖（不得短于留存期限）。

**检查八：第 2 章功能规模与第 5 章工期估算一致性**

第 5 章"5.1 整体交付计划 - 工期估算依据"中的模块数、复杂模块数必须引用第 2 章功能架构表格，不能重新估算。

**检查九：术语转化检查**

第 1、5 章、附录 A 中不应出现未转化的技术术语（JWT、RBAC、Redis、Nginx、MySQL、Quartz 等）。
如必须出现，在附录 B"技术术语表"中给出业务语言解释。

---

### 逐项自查清单

**技术约束自洽：**
- [ ] 单体架构描述中未出现分布式事务（Saga、TCC）
- [ ] 未引入消息队列，但没有描述基于消息的异步处理
- [ ] Redis 未被描述为存储业务主数据
- [ ] Quartz 未被描述为处理实时触发的业务逻辑
- [ ] 前端超出 Ant Design 能力范围的组件已单独说明引入了额外库

**量化表达：**
- [ ] 性能指标已用 P95 + 具体数值表达（不是"响应快"）
- [ ] 可用性已用百分比 + 年停机时间表达
- [ ] 安全要求已转化为具体技术措施

**设计说明完整性：**
- [ ] 每个技术选择有三要素（引入原因 + 设计决策 + 边界说明）
- [ ] 每个核心复杂模块有设计推理框架（问题定义 + 约束 + 判断 + 边界）

**章节完整性：**
- [ ] 5 个章节（第 1-5 章）+ 附录全部生成
- [ ] 第 2 章包含功能架构表格、业务流程图（流程审批类系统）、非功能需求三类内容
- [ ] 第 3 章包含系统架构图、技术架构图、横切关注点描述

**Mermaid 格式正确性（语法规则定义见 `@references/diagram-guides.md#Mermaid-语法规范`）：**
- [ ] 所有 Mermaid 图均符合 diagram-guides.md 规定的语法规范（节点 ID/形状/数量/标注/subgraph/前后说明）
- [ ] 流程图节点命名遵循动宾结构（操作）和疑问句（判断）的语义约定
- [ ] 单图节点数 ≤ 15，超出时已分拆为多张图
- [ ] 每张图前有引导语，图后有文字补充
