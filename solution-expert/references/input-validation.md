# input-validation

## 规则

### A 类：致命缺失（必须澄清，停止后续步骤）

触发条件 → 停止动作：

| 触发条件 | 澄清方向 |
|---|---|
| 输入过于抽象，无法提取任何具体功能点 | 请描述核心业务对象和主要操作是什么 |
| 业务流程描述存在明显逻辑矛盾 | 指出具体矛盾点，请客户确认哪种是正确的 |
| 提到外部系统集成但方向和方式完全不明 | 问清数据流方向、同步方式、对方是否有接口 |

存在任何 A 类缺失时：整理所有问题**一次性提出**（不多次来回），等待回复后继续。

提问格式规范：
- 给出选项（选择题优于开放题）
- 说明这个问题影响什么设计决策
- 所有问题在一次回复中提完

### B 类：可推断缺失（记录默认值，文档中标注）

| 缺失信息 | 处理方式 |
|---|---|
| 性能指标、可用性要求 | 使用 nfr-defaults.md 的默认值，文档标注"默认假设，请确认" |
| 部署环境 | 默认私有化部署 |
| 审批流程是否可配置 | 默认固定流程，文档标注并说明可配置的额外成本 |
| 是否需要移动端 | 默认仅 PC 端浏览器 |
| 数据留存期限 | 使用行业默认值 |

### C 类：歧义（做出选择，文档中说明另一种选择的差异）

| 歧义类型 | 处理方式 |
|---|---|
| 功能边界不清晰（如"统计报表"深度不明） | 选择合理默认解读，在文档中说明另一种解读的成本差异 |
| 功能优先级未明确 | 按功能类型和业务逻辑推断 P0/P1/P2，在第 2 章说明判断依据 |
