# integration-patterns

## 规则

当客户提及外部系统集成需求时，按本文档细化设计要点。集成模块是方案中最容易延期和出问题的环节，需在方案中提前暴露风险。

---

## 一、集成方式选择

### 同步 vs 异步调用

| 场景特征 | 选择方式 | 理由 |
|---|---|---|
| 调用结果直接影响后续业务判断 | 同步 HTTP 调用 | 用户等待结果，必须立即返回 |
| 调用结果仅用于通知/记录 | 异步调用（Spring 事件或定时任务）| 不阻塞主流程，失败可重试 |
| 外部系统响应慢（>3秒）| 异步调用 + 回调或轮询 | 避免用户长时间等待 |
| 高频批量数据同步 | 定时任务批量同步 | 控制调用频率，避免冲击外部系统 |

### API 集成 vs 消息队列集成

本技术栈（Spring Boot 单体）默认推荐 HTTP API 集成，消息队列超出边界需专项评估。

| 集成方式 | 适用场景 | 本技术栈支持情况 |
|---|---|---|
| HTTP API | 请求-响应模式、实时查询、单次调用 | 默认支持（RestTemplate/Feign）|
| 消息队列 | 高吞吐异步、削峰填谷、解耦强 | 超出边界（需专项评估 RabbitMQ/Kafka）|
| 共享数据库 | 内部系统间、紧耦合、数据量大 | 不推荐（违反系统边界原则）|
| 文件交换 | 批量数据导入导出、离线同步 | 支持（Excel/CSV 文件处理）|

## 二、常见外部系统集成要点

### ERP 系统集成

| 集成点 | 典型场景 | 设计要点 |
|---|---|---|
| 物料/商品数据同步 | 本系统物料编码来自 ERP | 定时拉取 ERP 物料主数据；增量同步（按更新时间）；物料编码作为唯一标识 |
| 订单推送 | 本系统订单推送到 ERP | 订单状态变更时触发推送；推送失败记录重试队列；ERP 返回单号回写 |
| 库存查询 | 实时库存来自 ERP | 本系统不存库存，实时调用 ERP 接口查询；接口超时降级为显示"库存查询失败" |
| 采购申请对接 | 采购流程与 ERP 审批联动 | 申请提交后推送到 ERP；审批结果回调或轮询获取 |

### OA 系统集成

| 集成点 | 典型场景 | 设计要点 |
|---|---|---|
| 统一认证 | 与 OA 单点登录 | OAuth2.0 或 CAS 集成；OA 作为认证源，本系统信任 OA Token |
| 待办同步 | OA 待办与本系统待办互通 | 本系统审批推送到 OA 待办；OA 待办点击跳转回本系统 |
| 消息通知 | OA 作为通知渠道 | 审批结果推送到 OA 消息；OA 用户 ID 与本系统用户 ID 映射 |

### 支付系统集成

| 集成点 | 典型场景 | 设计要点 |
|---|---|---|
| 支付发起 | 本系统发起支付请求 | 生成支付订单号；调用支付接口；前端跳转或扫码 |
| 支付回调 | 支付结果通知 | 支付平台异步回调；回调验签；订单状态更新（幂等设计）|
| 支付查询 | 主动查询支付状态 | 回调未到达时主动查询；定时轮询未完成订单 |
| 对账 | 支付数据核对 | 每日下载支付平台账单；与本系统支付订单比对；差异记录告警 |

### 第三方地图服务

| 集成点 | 典型场景 | 设计要点 |
|---|---|---|
| 地址解析 | 地址转经纬度 | 调用地图 API 解析；解析失败降级为用户手动选择 |
| 地图展示 | 位置在地图上显示 | 前端集成地图 SDK；坐标从后端接口获取 |
| 路线规划 | 多点路线计算 | 调用路线规划 API；结果返回前端展示 |

## 三、容错与降级策略

### 失败处理层次

| 层次 | 处理方式 | 适用场景 |
|---|---|---|
| 立即重试 | 同一请求重试 1-2 次 | 网络抖动、瞬时故障 |
| 延迟重试 | 记录失败，定时任务重试 | 外部系统临时不可用、业务允许延迟 |
| 人工介入 | 记录失败，告警通知人工处理 | 关键业务、多次重试失败 |
| 业务降级 | 替代方案继续业务流程 | 非关键集成、业务可容忍缺失 |

### 降级策略模板

**实时查询降级（如库存查询）：**

```markdown
设计：
- 正常：实时调用外部接口获取库存
- 降级：接口超时或失败，显示"库存信息暂时不可用，请稍后刷新"
- 恢复：接口恢复后自动切换回正常模式
```

**数据推送降级（如订单推送）：**

```markdown
设计：
- 正常：订单提交时同步推送外部系统
- 降级：推送失败，订单本地先保存，记录推送队列，定时任务重试
- 告警：重试 3 次仍失败，发送告警通知人工处理
- 恢复：推送成功后清除队列记录
```

## 四、数据一致性保障

### 主从关系判断

| 问题 | 判断依据 |
|---|---|
| 以谁的数据为准？| 数据产生方是主，接收方是从；修改权在产生方 |
| 本系统是否可修改外部数据？| 一般不可，只能推送变更请求由外部系统处理 |
| 外部数据是否可缓存？| 可缓存，但需明确更新机制（定时刷新或推送通知）|

### 对账机制

必须对账的集成：
- 涉及金额的支付集成
- 涉及库存的关键数据
- 涉及订单状态的核心流程

对账设计要点：
| 要点 | 说明 |
|---|---|
| 对账周期 | 每日凌晨或每小时（按业务重要性）|
| 对账方式 | 下载外部系统账单/数据，与本系统记录比对 |
| 差异处理 | 差异记录写入对账差异表，告警通知人工核对 |
| 差异归因 | 数据延迟、推送失败、外部系统异常等 |

## 五、凭证与鉴权管理

### 凭证管理原则

| 原则 | 说明 |
|---|---|
| 不硬编码 | API Key、Secret 不写在代码里，存储在配置或数据库 |
| 定期更换 | 凭证设置有效期，到期前更换 |
| 权限最小化 | 只申请必要的 API 权限 |
| 调用追踪 | 记录每次调用的凭证使用情况，异常告警 |

### 外部调用本系统的鉴权

| 鉴权方式 | 适用场景 |
|---|---|
| API Key + IP 白名单 | 内部系统间调用、信任度高 |
| OAuth2.0 Client Credentials | 标准化鉴权、外部 SaaS 平台 |
| HMAC 签名 | 高安全要求、防篡改防重放 |
