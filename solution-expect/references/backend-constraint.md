# backend-constraint

## 规则

### 分层职责（用于第 3、4 章描述）

| 层次 | 职责 | 关键约束 |
|---|---|---|
| Controller | 接收请求、参数校验、调用 Service、返回响应 | 不含业务逻辑；只依赖 Service 接口 |
| Service | 业务逻辑编排、事务边界定义、跨模块协调 | 不直接处理 HTTP；通过 Repository 接口访问数据 |
| Domain | 业务规则和核心计算逻辑 | 不依赖 Spring/MyBatis 等框架；可独立单元测试 |
| Repository | 封装数据库操作，提供面向业务的数据访问接口 | 对上层屏蔽 SQL 细节 |

### 按域分包结构（用于第 4 章描述）

```
com.company.project
├── {domain1}/          业务域1（如 purchase/）
│   ├── controller/
│   ├── service/
│   ├── repository/
│   ├── domain/
│   └── dto/
├── {domain2}/          业务域2
└── common/             横切关注点
    ├── security/       认证鉴权
    ├── exception/      统一异常处理
    ├── config/         公共配置
    └── util/           工具类
```

### 域边界纪律（第 4 章域描述时必须说明）

- 一个域不能直接调用另一个域的 Repository，只能调用其 Service
- 域间调用通过接口而不是直接依赖实现类
- 违反此约定，单体会退化为大泥球

### 横切关注点标准实现（第 3 章统一描述，第 4 章各模块不重复）

| 关注点 | 标准实现 |
|---|---|
| 认证 | Spring Security + JWT；Access Token 2h + Refresh Token 7天 |
| 鉴权 | RBAC 三层（用户→角色→权限）；接口级 @PreAuthorize + 菜单级动态路由 + 数据级行过滤 |
| 异常 | @ControllerAdvice 全局捕获；业务异常携带错误码；系统异常脱敏返回 |
| 审计 | AOP 切面拦截；记录操作人/时间/类型/参数摘要/结果 |
| 响应 | 统一结构 `{code, message, data, timestamp}` |

### 单体架构选择的说明要点（第 3 章必须主动说明）

**为什么选单体（回应技术评审的潜在质疑）：**
- 当前业务域数量和团队规模在单体适用范围内
- 本地事务简化了跨模块数据一致性处理
- 单一部署产物降低运维复杂度

**演进路径（体现技术前瞻性）：**
- 当前：单体，按业务域严格分包
- 成长期：识别独立扩展需求最强的域，优先拆分
- 成熟期：核心域微服务化，非核心域保持单体

**水平扩展的设计前提（第 5 章部署方案需体现）：**

| 前提 | 设计要求 |
|---|---|
| 无状态设计 | Session 不存在本地内存，使用 JWT 或 Redis Session |
| 文件不存本地 | 上传文件存储到 MinIO/OSS |
| 定时任务防重 | Quartz 使用数据库存储（集群模式）|
| 避免本地缓存 | 慎用 Guava Cache 等本地缓存，多实例间不同步 |
