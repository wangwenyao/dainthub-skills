## 规则

### 一、受众表达规范

**业务方（第 1、6 章，以及第 2 章的功能描述部分）**

规则：
- 只说"做什么"和"为什么"，不说"怎么做"
- 不出现技术术语；必须出现时，用括号给出业务语言解释
- 结论先行，用收益语言描述价值

转化示例：

| 不能这样写 | 应该这样写 |
|---|---|
| "系统采用 JWT 无状态认证" | "用户登录后获得访问凭证，2 小时内无需重复登录，持续操作时系统自动续期" |
| "基于 RBAC 的权限控制模型" | "系统支持按角色分配功能权限，财务人员只能看到费用相关功能，业务人员只能操作自己部门的数据" |
| "Quartz 定时扫描待审批记录" | "审批超过 24 小时未处理，系统自动发送催办提醒" |

**技术架构师（第 3、4、5 章）**

规则：
- 说清楚"怎么做"和"为什么这样做"
- 显式描述设计权衡，不回避局限性
- 用可验证指标，不用模糊形容词
- 主动说明边界和演进路径

转化示例：

| 弱表达 | 强表达 |
|---|---|
| "后端采用 Spring Boot 框架" | "后端基于 Spring Boot 3.x 构建单体应用，按业务域分包，通过包访问控制维持域边界；选择单体的理由是当前规模在适用范围内，且按域分包为后续渐进式拆分保留了路径" |
| "系统具有良好的高可用性" | "生产环境部署两个应用实例，Nginx 负载均衡，单实例故障时流量自动切换，可用性目标 99.5%（年停机<44小时）" |

### 二、量化规范

扫描全文，将以下模糊表达替换为量化描述：

| 模糊表达 | 量化方向 |
|---|---|
| "响应速度快" | 给出 P95 响应时间指标 |
| "支持高并发" | 给出并发用户数或 QPS |
| "系统稳定可靠" | 给出可用性百分比和年停机时间 |
| "数据安全有保障" | 说明具体安全措施（HTTPS、加密字段、操作审计等）|
| "操作可追溯" | 说明记录哪些内容、保留多久 |

### 三、论证结构规范

**章节级结构（SCQA）：**

每个主要章节按此顺序组织：现状（Situation）→ 冲突（Complication）→ 答案（Answer）。
不需要显式标注这三个词，但写作时要意识到逻辑顺序。

**设计点级三要素（每个技术选择或设计决策必须包含）：**

| 要素 | 回答的问题 | 缺少时的问题 |
|---|---|---|
| 引入原因 | 为什么在这里需要这个设计 | 读者无法评估这个设计是否合适 |
| 设计决策 | 做了什么关键判断，为什么没选其他方案 | 看不出技术判断力，像是随意选择 |
| 边界说明 | 这个设计负责什么，不负责什么 | 读者不知道边界在哪，容易产生误解 |

**什么值得写，什么不值得写：**

| 值得写 | 不值得写 |
|---|---|
| 模块核心职责和边界 | 具体注解用法（@Transactional 的语法）|
| 关键数据的一致性保证方式 | 框架的 API 调用方式 |
| 设计选择的权衡理由 | 数据库字段的具体命名和 DDL |
| 哪些地方做了简化及代价 | 代码实现的具体步骤 |
| 技术风险及应对 | — |

### 四、实现思路表达规范

对每个核心复杂模块，按以下框架描述（而不是直接给技术答案）：

```
1. 核心问题定义
   这个模块最关键的设计挑战是什么？

2. 约束下的设计思路
   在固定技术栈约束下，这个问题有什么天然优势和限制？

3. 关键设计判断
   做了哪几个关键决策？每个决策的依据是什么？
   （只说影响全局的判断，不展开所有实现细节）

4. 边界与假设
   这个设计在什么条件下是合适的？
   超出哪个边界后设计会失效？
```

按模块类型的侧重点：

| 模块类型 | 重点说明 |
|---|---|
| 状态流转类 | 状态枚举和转换规则；哪些操作在同一事务内；固定/可配置流程的选择 |
| 数据权限类 | 权限维度；SQL 层实现方式；与接口权限的层次关系 |
| 定时任务类 | 触发频率的依据；幂等性设计；多实例防重机制 |
| 外部集成类 | 集成方式的选择依据；降级策略；数据一致性保障 |
