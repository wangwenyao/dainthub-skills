# middleware-constraints

## 规则

### MySQL

**数据主权原则：每张表只归属于一个业务域。**

| 规则 | 说明 |
|---|---|
| 表命名体现所属域 | `purchase_apply`（采购域）、`sys_user`（系统域）|
| 跨域只通过 Service 访问 | 采购域不直接查 sys_user 表，只能调用 UserService |
| 跨域通过外键 ID 关联 | 不做跨域 JOIN 查询 |

**事务边界规则：**
- 事务边界在 Service 层定义，覆盖一个完整业务操作的所有数据变更
- 事务内不调用外部 HTTP 接口（超时会导致长时间持锁）
- 事务内不发送通知（通知在事务提交后异步执行）

**公共字段规范（所有业务表必含）：**
`id`（雪花算法）、`create_by`、`create_time`、`update_by`、`update_time`、`deleted`（软删除标记）

**能力边界 → 触发其他方案：**

| 问题 | 触发的方案 |
|---|---|
| 高频读取的热点数据（字典、配置、权限菜单）| 引入 Redis 缓存 |
| 全表聚合统计（数据量大时）| Quartz 定时预聚合写入汇总表 |
| 全文检索需求 | 引入 Elasticsearch |
| 复杂多条件组合查询（非结构化数据）| 引入 Elasticsearch |

### Elasticsearch

**定位：检索能力层，MySQL 不擅长全文检索和复杂组合查询。**

**触发条件：**
- 客户明确提出"全文检索"、"关键词搜索"、"智能搜索"需求
- 数据量超过百万行且需要多字段组合检索
- 搜索响应要求秒级而非分钟级

**适用场景：**

| 场景 | Elasticsearch 角色 | MySQL 角色 |
|---|---|---|
| 全文检索 | 存储索引字段，提供检索能力 | 存储完整业务数据 |
| 多条件组合查询 | 复杂条件筛选和排序 | 简单条件查询 |
| 聚合统计 | 实时聚合分析 | 精确统计（数据量小时）|

**设计规则：**

1. **数据同步策略：**
   - 同步双写：业务数据写入 MySQL 同时写入 ES（适用于数据一致性要求高的场景）
   - 异步同步：MySQL 数据变更后通过定时任务同步到 ES（适用于一致性要求较低的搜索场景）
   - CDC 同步：通过 MySQL Binlog 监听变更同步到 ES（适用于实时性要求高的场景，需额外组件如 Canal）

2. **索引设计：**
   - 只索引需要检索的字段，不全量同步
   - 索引结构按业务域设计（如 `purchase_order`、`contract`）
   - 索引字段与 MySQL 字段映射关系文档化

3. **查询策略：**
   - 搜索请求先查 ES，获取 ID 列表
   - 根据 ID 从 MySQL 获取完整数据（避免 ES 存储冗余）
   - ES 查询失败降级为 MySQL LIKE 查询（性能下降但业务不中断）

4. **能力边界：**
   | 不适合的场景 | 原因 | 正确处理方式 |
   |---|---|---|
   | 作为主数据存储 | ES 数据可能丢失，不是可信数据源 | MySQL 是主存储，ES 是索引 |
   | 事务性操作 | ES 不支持事务 | 事务操作只在 MySQL |
   | 精确统计（如财务对账）| ES 数据可能不实时一致 | 精确统计用 MySQL |

**引入说明模板（第 3 章）：**

> 本系统引入 Elasticsearch 用于全文检索场景。MySQL 作为业务主数据存储，Elasticsearch 作为检索索引层。数据同步采用异步定时同步方式（每天凌晨同步增量数据），搜索请求先查 ES 获取候选 ID，再从 MySQL 获取完整数据。ES 查询失败时降级为 MySQL LIKE 查询，保证业务不中断。

### Redis

**定位：旁路能力层，不存储业务主数据。Redis 故障后系统能降级运行。**

**三类适用场景：**

**场景一：热点数据缓存**

触发条件：读多（分钟内大量重复读）+ 写少（小时/天级才变更）+ 数据量小

设计规则：
- 策略：Cache-Aside（写操作先更新 MySQL，成功后删除缓存；读操作先查 Redis，Miss 后查 MySQL 并回填）
- 选择"删除缓存"而非"更新缓存"：删除是幂等操作，避免并发写场景下的不一致
- TTL 设置：字典数据 30 分钟 + 数据修改时主动删除；权限数据跟随 Token 生命周期

**场景二：幂等控制（防重复提交）**

触发条件：用户可能重复点击、网络重试、资金类操作

设计规则：
- 前端生成幂等键（UUID），随请求携带
- `SET {幂等键} "processing" NX EX {超时时间}` → 成功则执行，失败则返回"处理中"
- 超时时间 = 业务预期最长执行时间 × 2

**场景三：Token 与会话管理**

触发条件：多实例部署 / 需要主动失效 Token / 需要强制踢人

设计规则：
- JWT 本身无状态，主动失效通过 Redis 黑名单实现
- 登出/踢人 → Token 加入黑名单，TTL 对齐 Token 过期时间，到期自动清理
- Redis 存储在线用户信息，支持查看在线人数和强制踢出

**降级原则：**

| Redis 故障时 | 降级处理 |
|---|---|
| 热点缓存失效 | 直接查 MySQL，性能下降但业务不中断 |
| 幂等控制失效 | 临时关闭幂等校验，接受极低概率重复提交 |
| Token 管理失效 | 降级为纯 JWT 校验，主动失效能力临时不可用 |

### Quartz

**定位：定时触发层，解决"在特定时间自动执行业务逻辑"的问题。**

**场景识别规则：**

| 客户描述关键词 | 隐含的定时任务 | 建议频率 |
|---|---|---|
| "审批超时自动提醒" | 扫描超时待审批单据，触发通知 | 每小时 |
| "合同/设备到期预警" | 扫描即将到期记录，提前 N 天推送 | 每日凌晨 |
| "每月自动生成报表" | 月末触发数据聚合，归档报表 | 月度 Cron |
| "数据定期与外部同步" | 按周期拉取或推送数据 | 按集成需求 |
| "统计数据预聚合" | 预聚合明细数据写入汇总表 | 小时/天级 |

**四条设计规则：**

1. **幂等性**：业务表增加处理状态字段，任务每次执行前检查，已处理的记录不再处理
2. **多实例防重**：配置 Quartz 集群模式（JDBCJobStore），通过数据库行锁保证同一任务只有一个实例执行；需在 MySQL 中创建 Quartz 元数据表
3. **分批处理**：大批量数据每次处理固定数量（如 100 条），避免单次执行时间过长、大事务锁表
4. **任务可观测性**：每次执行写入任务日志（触发时间、开始/结束时间、处理数量、执行状态、错误信息）

**能力边界：**

| 不适合的场景 | 原因 | 正确处理方式 |
|---|---|---|
| 用户操作后立即触发 | Quartz 是定时触发，不是事件触发 | Service 同步调用或 Spring 事件 |
| 高并发任务队列处理 | 超出 Quartz 设计场景 | 超出本技术栈边界，需专项评估 |

### 中间件组合规则

**MySQL + Redis（带缓存的数据访问）**
- 职责：MySQL 是唯一事实来源，Redis 是读加速层
- 一致性：写先更 MySQL，再删 Redis；读先查 Redis，Miss 再查 MySQL 并回填
- 判断"值不值得缓存"：高频变更的数据缓存收益低，引入一致性成本得不偿失

**MySQL + Quartz（数据驱动的定时处理）**
- 职责：Quartz 触发扫描，MySQL 存储数据和任务状态
- 原则：任务处理结果必须写回 MySQL；在 MySQL 记录中设处理状态，保证幂等
- 事务：每条记录的处理和状态更新在同一事务内

**三者组合（综合业务管理系统的标准配置）**
