# ch5 — 基础设施与部署架构

受众：技术架构师 + 运维。资源规格量化，部署结构清晰。

## 结构与写作规则

### 5.1 部署架构选择
根据第 2 章的非功能需求，按以下规则选择部署方式并说明理由：

| 可用性 | 并发用户 | 部署方式 |
|---|---|---|
| <99% | <50 人 | 单机部署 |
| 99%~99.5% | 50-200 人 | 双实例 + Nginx 负载均衡，数据库独立服务器 |
| 99.5%~99.9% | 200-1000 人 | 多实例集群 + Nginx，数据库主从读写分离 |
| >99.9% | >1000 人 | 超出本方案标准范围，需专项规划 |

### 5.2 各环境资源规划

三个环境（开发/测试/生产）的服务器配置表：

| 服务器 | 环境 | 规格建议 | 数量 | 备注 |
|---|---|---|---|---|
| 应用服务器 | 生产 | 4核8G / 8核16G | {1-2}台 | Spring Boot + Nginx |
| 数据库服务器 | 生产 | 8核16G，SSD | {1-2}台 | MySQL，独立部署 |
| 缓存服务器 | 生产 | 2核4G | {0-1}台 | Redis，小规模可与应用同机 |
| 应用服务器 | 测试 | 4核8G | 1台 | 单机部署 |
| 数据库服务器 | 测试 | 4核8G | 1台 | MySQL |
| 开发服务器 | 开发 | 4核8G | 1台 | 全组件单机 |

标注：*以上规格为参考值，请根据实际预算和业务量确认。*

### 5.3 高可用设计
按实际部署方式选择描述内容：
- 应用层：Nginx 健康检查 + 无状态多实例 → 单实例故障流量自动切换
- 数据库：主从复制（如需）→ 说明主库故障切换方式和预期时间
- Redis：故障降级策略（直接查 MySQL）
- Quartz：集群模式，实例故障后任务由其他实例继续执行（需在部署时配置 JDBCJobStore）

### 5.4 数据备份策略

| 备份类型 | 频率 | 保留时间 | 存储位置 |
|---|---|---|---|
| MySQL 全量备份 | 每日凌晨 | 30 天 | 独立备份服务器或对象存储 |
| MySQL 增量（Binlog）| 实时 | 7 天 | 本地 + 异地 |
| 应用配置 | 每次变更 | 永久 | 版本控制系统 |

RPO：< 24 小时（全量）/ < 5 分钟（Binlog）
RTO：< 2 小时

### 5.5 网络与安全边界
- 数据库只允许应用服务器内网 IP 访问，不暴露公网
- Nginx 配置 HTTPS，HTTP 自动重定向
- 防火墙只开放必要端口（80/443 对外，数据库端口只对内网）
- 生产数据库账号最小权限（应用账号只有 DML，无 DDL）
