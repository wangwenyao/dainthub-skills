# complexity-detection

## 规则

符合以下任一标准的模块，标注为"核心复杂模块"，在第 4 章详细说明设计推理；其余模块简要描述。

### 识别标准

| 标准 | 识别信号 | 第 4 章重点说明的内容 |
|---|---|---|
| 状态流转复杂 | 出现"审批/工单/流程/驳回/撤回"；存在条件分支或回退路径 | 状态枚举、转换规则、事务边界；固定/可配置流程的选择和理由 |
| 数据权限复杂 | "只能看到自己的"、"部门数据"、"按权限查看"；涉及组织层级隔离 | 权限维度、SQL 层实现方式、与接口权限的层次关系 |
| 性能要求特殊 | 大量历史数据查询；全表聚合统计；高频访问的配置数据 | Redis 缓存策略、分页和索引思路、实时/预聚合的选择 |
| 外部集成关键路径 | 业务流程中有"调用外部系统"；外部返回结果影响后续业务判断 | 降级策略、超时重试、数据一致性保障 |
| 定时任务驱动 | "自动"、"定期"、"到期"、"超时"；时间驱动的状态变更 | 触发频率、幂等性设计、多实例防重（Quartz 集群模式）|

### 复杂度等级

- **核心复杂**：符合上述任一标准 → 第 4 章用设计思路三要素详细说明
- **标准实现**：标准 CRUD，无特殊复杂度 → 第 4 章简要说明模块职责和页面结构

### 输出格式

```
[核心复杂模块]
- {模块名}（原因：{复杂类型}）

[标准实现模块]
- {模块名列表}
```
