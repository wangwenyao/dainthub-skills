# Product Manager Skill

B端产品经理能力体系，从需求发现到PRD交付（含页面线框图）。

---

## ⚠️ 执行约束（MANDATORY）

**加载本 skill 后必须遵守以下规则，违反即为 skill 执行失败**：

### 0. 前置约束

以下三项约束为流程执行前的必要判断，必须在其他约束之前应用。

#### 0.1 自适应文档结构约束

**根据端划分自动选择文档结构**：

| 端数量 | 文档结构 | 适用场景 |
|--------|----------|----------|
| ≥ 2 个端 | **三层结构** | 小程序 + 管理后台 |
| = 1 + 模块 ≥ 3 | **两层结构** | 管理后台多模块 |
| = 1 + 模块 ≤ 2 | **单层结构** | 简单管理后台 |

**结构选择流程**：
```
端划分确认（需求分析输出）
    ├── 端数量 ≥ 2 → 三层结构
    ├── 端数量 = 1 → 检查模块数量
    │   ├── 模块 ≥ 3 → 两层结构
    │   └── 模块 ≤ 2 → 单层结构
    └── 无端划分 → 默认两层结构
```

**模板引用**：`@references/prd-template.md#自适应文档结构`

---

#### 0.2 输入完整性判断约束

**追问策略根据输入覆盖率自动决定**：

| 覆盖率 | 追问策略 | 说明 |
|--------|----------|------|
| ≥ 80% | **跳过追问** | 输入足够完整，直接进入需求结构化 |
| 50-80% | **精简追问** | 只追问未明确维度 |
| < 50% | **完整追问** | 按追问清单逐项确认 |

**覆盖率计算**（9 个维度）：
```
覆盖率 = Σ(维度得分 × 权重)
├── 核心业务场景 (15%) + 用户角色 (12%) + 数据实体 (12%)
├── 业务流程 (15%) + 异常场景 (10%) + 外部系统 (8%)
├── 数据来源 (8%) + 权限需求 (10%) + 端划分 (10%)
└── 维度得分：有明确说明 = 100%，部分说明 = 50%，未说明 = 0%
```

**约束规则**：
- 覆盖率 ≥ 80% → 必须跳过追问，不得强制追问
- 覆盖率 50-80% → 只追问未明确维度，不得完整追问
- 覆盖率 < 50% → 按完整追问清单逐项确认

#### 0.3 功能复杂度约束

**复杂度决定描述方式，与优先级独立**：

| 复杂度 | 定义 | 描述方式 |
|--------|------|----------|
| 简单 | 单一步骤，无分支 | 一句话 |
| 标准 | 3-5 步骤，有校验 | 引用模板 + 差异 |
| 复杂 | ≥5 步骤，复杂规则 | 完整描述 |

**通用功能模板引用**：
| 模板ID | 适用功能 |
|--------|----------|
| T01-T06 | 查看/搜索/新增/编辑/删除/导出 |

**禁止行为**：
- ❌ 通用功能不引用模板
- ❌ 不判断复杂度直接完整描述
- ❌ 简单功能完整描述

### 1. 流程执行约束

| 规则 | 说明 |
|------|------|
| **禁止跳步骤** | 必须按 workflow 定义步骤顺序执行，不得跳过任何步骤 |

| **必须读取引用** | workflow 中 `@xxx.md` 引用必须实际读取，不得凭记忆推断 |

### 2. 全链路数量传递约束（核心约束）

**从用户输入到线框图输出的完整数量传递链路**：

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│ 需求分析    │───→│ 业务建模    │───→│ 功能架构    │───→│ PRD编写     │───→│ 线框图生成  │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
       │                  │                  │                  │                  │
       ↓                  ↓                  ↓                  ↓                  ↓
  N 个需求点       →    N 个业务活动    →    M 个功能点     →    M 个章节     →    P 个页面

       └──────────────────────────────────────────────────────────────────────┘
                                    数量传递约束（强制）
```

**每个环节必须输出映射表，验证数量传递**：

| 环节 | 输入 | 输出 | 强制映射表 |
|------|------|------|------------|
| 需求分析 | 用户输入（追问清单 9 项） | N 个需求点 | 追问确认表 |
| 业务建模 | N 个需求点 | N 个业务活动 | 需求点→活动映射表 |
| 功能架构 | N 个业务活动 | M 个功能点 | 业务活动→功能点映射表 |
| PRD编写 | M 个功能点 | M 个章节 | 功能点→章节覆盖表 |
| 线框图生成 | M 个功能点 | P 个页面 | 功能点→页面映射验证表 |

**约束规则**：
- 任何环节的映射表缺失 → skill 执行失败
- 任何输入项无对应输出 → skill 执行失败
- 数量不匹配（输出数量 < 输入数量） → skill 执行失败

### 3. 逐项处理约束

**每个环节必须逐项处理，不得批量处理**：

```
逐项处理清单格式：
├── ☐ 输入项 1 → 正在处理...
│   └── 输出：输出项 1 ✓
├── ☐ 输入项 2 → 正在处理...
│   └── 输出：输出项 2 ✓
├── ☐ 输入项 3 → 正在处理...
│   └── 输出：输出项 3 ✓

完成确认：所有输入项已处理完毕
```

### 4. 输出格式约束

| 输出类型 | 强制要求 |
|----------|----------|
| **PRD 功能清单** | 必须使用统一功能清单表格式（一级合并、二级合并、三级按行展示，含「对应业务活动」列） |
| **业务↔功能关联** | 功能点必须有对应业务活动，实现业务活动清单↔功能点清单双向追溯 |
| **PRD 功能详细描述** | P0 和 P1 功能必须完整填写：前置条件、权限矩阵、字段约束、业务规则、验收标准 |
| **线框图 HTML** | 必须使用模板定义的 class，不得自创样式；灰色系配色，不得使用真实颜色 |
| **线框图 YAML** | 必须包含完整的 pages 配置，每个页面必须有 type、route、config |
| **Mermaid 图表** | 必须使用正确的 Mermaid 语法，确保可渲染 |

### 5. 质量验证约束

| 验证时机 | 必须动作 |
|----------|----------|
| **生成前** | 确认已读取所有需要的 references 文件 + 接收上游输出 |
| **生成中** | 逐项处理 + 每完成一项更新映射表状态 |
| **生成后** | 输出映射表（最终状态） + 检查无缺失项 |

### 6. 禁止行为

| 禁止 | 原因 |
|------|------|
| ❌ 凭记忆生成内容 | skill 文件可能已更新，必须实际读取 |
| ❌ 输出不完整的章节 | P0 章节必须完整，不得"后面补充" |
| ❌ 自创格式或样式 | 必须使用 skill 定义的模板和格式 |
| ❌ 跳过质量检查 | 生成后必须验证，不得直接结束 |
| ❌ 批量处理（不逐项） | 必须逐项处理，不得批量跳过 |
| ❌ 不输出映射表 | 每个环节必须输出数量传递映射表 |
| ❌ 数量传递断裂 | 上游输出与下游输入数量不匹配 |

### 7. 执行信号

以下行为表明 skill 执行失败：
- **数量传递约束违反**：
  - 映射表缺失（追问确认表/需求点→活动映射表/活动→功能点映射表/功能点→章节覆盖表/功能点→页面映射验证表）
  - 输入项无对应输出项
  - 输出数量 < 输入数量
- **逐项处理约束违反**：
  - 逐项处理清单未逐项输出
  - 批量处理跳过部分项
- **输出格式约束违反**：
  - 功能清单没有使用统一表格格式
  - 功能清单缺少「对应业务活动」列
  - 功能详细描述遗漏 P0/P1 功能
  - 线框图使用真实颜色或图标
  - YAML 缺少 pages 配置或 config 不完整
  - Mermaid 图表语法错误无法渲染
- **基础约束违反**：
  - 没有读取引用文件就直接生成内容

## 方法论速查

### 需求优先级方法

| 场景 | 用哪个 | 详见 |
|------|--------|------|
| 功能分类 | KANO | `@references/requirement-analysis.md#KANO模型` |
| 排优先级 | MoSCoW | `@references/requirement-analysis.md#MoSCoW优先级法` |

### 验收标准格式

```
Given [前置条件]
When [用户操作]
Then [预期结果]
```

> 验收标准在功能详细描述中定义，无需单独写用户故事

### PRD必需章节（P0）

```
1. 执行摘要          6. 业务流程矩阵
2. 成功标准          7. 业务活动与规则
3. 产品范围          8. 功能架构（含三级功能点）
4. 业务角色模型      9. 业务实体模型（ER图）
5. 用户旅程          10. 非功能需求
```

> 完整模板见 `@references/prd-template.md`

## 协同Skill

**页面线框图生成**：不需要 UI 设计 skill。
- 线框图只表达结构，不涉及 UI 视觉
- 前端 skill（vben-frontend-dev）直接读取 YAML → 代码

**验收标准转化**：
- 验收标准可转化为测试场景
- 用户旅程 → E2E 测试流程
- 业务规则 → Service 层单元测试

详细转化指引 → `@references/workflows/prd-writing.md#验收标准与测试协作`

**偏差反馈处理**：与 tech-manager 协作
- 接收 tech-manager 的偏差反馈报告
- 评估产品影响 → 做出产品决策 → 更新 PRD

详细流程 → `@references/workflows/deviation-handling.md`
