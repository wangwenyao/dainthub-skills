# Product Manager Skill 改进建议

> 基于对全部 13 个文件的深度评估，聚焦目标：**同时兼顾产品功能设计质量与 PRD 对开发的指导精度**。

---

## 背景与评估结论

当前 skill 在需求分析、业务建模、功能架构设计、PRD 编写、线框图生成五条主链路上均已达到资深 B 端产品经理水准（综合评分 4.8/5）。以下改进项不是对现有体系的否定，而是针对"设计→文档→开发交付"这条链路上的具体断点进行修补。

---

## P1 改进项（高优先级，直接影响开发可执行性）

### 1. 验收标准扩展为四象限测试矩阵

**问题根源**

现有 `prd-writing-guide.md` 的验收标准只要求 1-3 条正向 `Given/When/Then`。开发和测试面对的 80% 问题集中在边界值和异常场景，这些在当前 PRD 里完全空白，靠开发自行揣摩。

**改进方案**

对 P0/P1 功能点，将验收标准从单条格式升级为四类场景的完整矩阵：

```markdown
**验收标准**

| 场景类型 | Given | When | Then |
|----------|-------|------|------|
| 正向用例 | 用户已登录且有新增权限 | 填写完整表单点击确定 | 数据保存成功，列表刷新，Toast 提示"新增成功" |
| 边界值 | 名称字段输入 50 个字符（最大值） | 点击确定 | 校验通过，正常保存 |
| 负向用例 | 名称字段为空 | 点击确定 | 名称输入框下方显示"请输入名称"，表单不提交 |
| 权限异常 | 用户无新增权限 | 进入列表页 | "新增"按钮不渲染（不是置灰，而是不显示） |
```

**修改位置**

- `references/prd-writing-guide.md` → `#五、用户故事与验收标准`：将格式说明从单条示例升级为四类场景说明
- `references/workflows/prd-writing.md` → `#4. 生成中必须动作`：加入约束项"P0/P1 功能点验收标准必须覆盖四类场景"
- `references/workflows/prd-writing.md` → `#5. 生成后必须动作`：加入检查项"☐ 验收标准是否覆盖负向用例和权限异常？"

### 2. 表单字段约束从"模板可选"升为"P0/P1 强制"

**问题根源**

`prd-template.md` 的 T03/T04 模板中已有字段约束表，但 `workflows/prd-writing.md` 的执行约束没有强制要求填写。实际执行时，字段约束经常只有"名称、必填"两列，缺少"错误提示文案"、"校验层级（前端/后端）"等开发必需信息。

**改进方案**

在现有字段约束表基础上扩充两列，并将其升级为硬约束：

```markdown
**字段约束**

| 字段名 | 类型 | 必填 | 约束规则 | 错误提示文案 | 校验层级 |
|--------|------|------|----------|-------------|----------|
| 商品名称 | 文本 | 是 | 长度 1-50 字符，不可重复 | "请输入商品名称"（空） / "名称已存在"（重复） | 前端（空） + 后端（重复） |
| 价格 | 数字 | 是 | ≥0，最多2位小数 | "请输入有效价格" | 前端 |
| 库存 | 整数 | 是 | ≥0，≤99999 | "库存数量不合法" | 前端 |
| 商品图片 | 文件 | 否 | jpg/png，单张≤2MB，最多5张 | "图片格式不支持" / "图片超过大小限制" | 前端 |
```

**修改位置**

- `references/prd-template.md` → `#5.3 T03 / 5.5 T04 模板`：字段约束表加入"错误提示文案"和"校验层级"两列
- `references/workflows/prd-writing.md` → `#1. 数量传递约束`：增加约束"含表单的 P0/P1 功能点，字段约束表必须填写错误提示文案和校验层级，缺失视为 PRD 不完整"
- `references/workflows/prd-writing.md` → `#5. 生成后必须动作`：加入检查项"☐ 新增/编辑功能的字段约束表是否包含错误提示文案？"

### 3. 状态机补充转换矩阵（仅有 Mermaid 图不够）

**问题根源**

`stateDiagram-v2` 的流程图给了状态流转的可视化路径，但无法表达：谁有权触发这个转换、触发的前置条件是什么、转换时关联实体如何联动。开发实现时最常出现的"这个状态能不能回退""哪个角色能审批"等问题，都需要补充一张表才能说清楚。

**改进方案**

在每个状态流转图之后，强制输出配套的转换矩阵：

```markdown
**状态转换矩阵**

| 当前状态 | 触发操作 | 执行角色 | 前置条件 | 目标状态 | 关联联动 |
|----------|----------|----------|----------|----------|----------|
| 草稿 | 提交审批 | 运营人员 | 必填字段已填写 | 待审核 | 通知审批人（系统消息） |
| 待审核 | 审批通过 | 管理员 | 无 | 已上架 | 商品对外可见；库存锁定 |
| 待审核 | 审批驳回 | 管理员 | 须填写驳回原因 | 草稿 | 通知提交人（系统消息） |
| 已上架 | 下架 | 管理员/运营 | 无 | 已下架 | 商品对外不可见 |
| 已下架 | 重新上架 | 管理员/运营 | 无 | 已上架 | 商品对外可见 |
| 草稿/待审核 | 删除 | 管理员 | 状态为草稿或待审核 | —（已删除） | 关联订单不受影响 |
```

**修改位置**

- `references/business-modeling.md` → `#四、业务规则体系`：在"状态规则"类型下新增"状态转换矩阵"模板，作为有状态实体的必要补充
- `references/prd-template.md` → `#6. 业务流程矩阵`：在 `6.4 状态流转图` 下注明"有状态实体必须同时输出状态转换矩阵"
- `references/workflows/business-modeling-flow.md` → `#步骤五：定义业务规则`：增加约束"含状态流转的实体必须输出状态转换矩阵，Mermaid 图为辅助可视化"

## P2 改进项（中优先级，显著提升功能设计质量）

### 4. 功能点依赖关系声明

**问题根源**

统一功能清单表目前无依赖列。开发在排期时不知道哪些功能可以并行开发、哪些必须串行，这类信息完全依赖口头沟通，容易在迭代过程中丢失。

**改进方案**

在统一功能清单表的列定义中新增"前置依赖"列：

```markdown
| 一级模块 | 二级功能 | 三级功能点 | 优先级 | 页面/组件 | 用户角色 | 对应业务活动 | 前置依赖 |
|----------|----------|------------|--------|-----------|----------|--------------|----------|
| M01 商品管理 | F01 商品列表 | F0103 新增商品 | P0 | P002 | 管理员 | BA02 | F0201（分类管理）|
| | F02 商品分类 | F0201 查看分类树 | P0 | P003 | 管理员 | BA05 | — |
```

规则说明：
- 填写被依赖的功能点 ID，多个用 `/` 分隔
- 若无依赖填 `—`
- 外部系统依赖用文字描述，如"支付接口联调完成"

**修改位置**

- `references/prd-writing-guide.md` → `#2.2 功能清单格式`：表格列定义中新增"前置依赖"列，说明填写规则
- `references/workflows/function-architecture-design.md` → `#步骤七：验证完整性`：加入检查项"☐ 功能点间依赖关系是否声明？"

### 5. 复杂功能点增加设计决策记录

**问题根源**

PRD 记录了 WHAT（做什么），但不记录 WHY（为什么这么设计）。开发遇到细节问题时无从理解设计意图，产生不必要的沟通来回；产品迭代时也难以判断原始约束是否还成立。

**改进方案**

对复杂度 = 复杂的功能点，在现有模板末尾增加"设计决策"可选项（P0 复杂功能必填，P1 建议填写）：

```markdown
**设计决策**

- 备选方案：A. 实时搜索（输入即搜索）B. 点击按钮搜索
- 选择方案：B（点击搜索）
- 选择原因：数据量大时实时搜索对后端压力过大；B 端用户习惯点击确认
- 接受的权衡：用户需要多一次点击操作，但可接受
- 约束条件：若后续引入 ElasticSearch，可评估改为实时搜索
```

**修改位置**

- `references/prd-writing-guide.md` → `#2.5 功能详细描述模板（三级功能点）`：在验收标准之后新增"设计决策"模板（标注：仅复杂功能，P0 必填）
- `references/workflows/prd-writing.md` → `#步骤二：功能复杂度判断`：在复杂功能的处理路径中加入"需输出设计决策"的提示

### 6. 产品设计流程（流程二）输出约束实质化

**问题根源**

`references/workflows/product-design.md` 目前只有 4 步骨架，是对 `product-design.md` 的简单路由。Design Thinking 和 UX 五要素的输出物没有强制映射到 PRD 章节，导致产品设计过程的思考在 AI 执行中经常被跳过。

**改进方案**

在 `workflows/product-design.md` 中为每个 UX 层级增加强制输出约束：

```markdown
## 强制输出映射

| UX 要素层 | 必须输出的内容 | 对应 PRD 章节 |
|-----------|---------------|---------------|
| 战略层 | 商业目标 + 用户核心需求 + 成功指标 | 一、执行摘要 + 二、成功标准 |
| 范围层 | In Scope / Out of Scope + MVP 功能筛选矩阵 | 三、产品范围 |
| 结构层 | 信息架构树 + 核心流程路径 | 五、用户旅程 + 八、功能菜单层级 |
| 骨架层 | 关键页面布局说明（非线框图，是布局决策） | 六、业务流程 + 线框图设计说明 |

**执行失败信号**：产品设计流程执行后，上述 4 层均无对应产出 → 视为流程未实质执行
```

**修改位置**

- `references/workflows/product-design.md`：在现有 4 步流程后新增"强制输出映射"表格和执行失败信号
- `SKILL.md` → `#方法论 → 输出映射`：补充 UX 各层的 PRD 章节对应关系
