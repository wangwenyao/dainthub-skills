# 偏差反馈处理流程

**何时使用**: 接收 tech-manager 提交的实现偏差反馈报告，进行产品决策并更新 PRD。

---

## 输入来源

偏差反馈报告来自 tech-manager 的实现验证阶段，格式见 tech-manager skill 的 `@change-templates.md#实现偏差反馈`。

---

## 流程步骤

```
1. 接收偏差反馈报告
   ├── 来源：tech-manager 实现验证后的"不通过"项
   ├── 格式：偏差反馈报告模板
   └── 内容：偏差描述、分类、严重等级、PRD要求、实际实现、影响评估、建议方案

2. 理解偏差分类
   ├── 技术限制型 → 技术上无法按 PRD 实现，需调整方案
   ├── 逻辑缺陷型 → PRD 存在逻辑矛盾或遗漏
   ├── 优化建议型 → 技术侧发现更优实现方式
   └── 约束新增型 → 实现中发现新的限制条件

3. 评估产品影响
   ├── 用户体验影响 → 高/中/低/无
   ├── 业务价值影响 → 是否影响核心功能
   ├── MVP 影响 → 是否影响 MVP 范围
   └── 用户接受度 → 用户是否能接受变化

4. 产品决策
   ├── 接受偏差 → 同意技术侧方案，更新 PRD
   ├── 坚持原需求 → 说明理由，技术侧需重新实现
   └── 协商调整 → 与技术侧共同确定新方案

5. 更新 PRD
   ├── 标注偏差决策结果
   ├── 更新相关章节内容
   └── 添加修订记录

6. 生成决策通知
   └── 输出偏差决策通知给 tech-manager
```

---

## 偏差分类理解

### 技术限制型

```
场景：PRD 要求实时计算库存，但数据量大无法在 500ms 内完成
理解要点：
- 这是技术上的客观限制，不是开发能力问题
- 需评估替代方案的用户体验影响
- 判断是否可以接受性能妥协

决策倾向：
- 若用户体验影响低 → 通常接受替代方案
- 若用户体验影响高 → 可能需要优化技术方案或调整需求
```

### 逻辑缺陷型

```
场景：PRD 定义的状态机缺少"撤回"流转，用户提交后无法修改
理解要点：
- 这是 PRD 本身的问题，非技术侧原因
- 需承认需求设计遗漏
- 补充完整逻辑

决策倾向：
- 通常接受补充逻辑的建议
- 需更新 PRD 补充缺失流程
```

### 优化建议型

```
场景：PRD 要求手动选择地址，技术侧增加了自动定位功能
理解要点：
- 这是用户体验提升，无负面影响
- 需评估是否符合产品方向
- 判断是否与现有设计冲突

决策倾向：
- 若符合产品方向 → 通常接受优化
- 若与设计方向冲突 → 可能坚持原需求
```

### 约束新增型

```
场景：第三方支付 SDK 不支持 PRD 要求的"组合支付"
理解要点：
- 这是外部依赖的限制
- 需评估对 MVP 的影响
- 判断是否延后实现

决策倾向：
- 若影响 MVP 核心功能 → 可能需要更换方案
- 若非核心功能 → 通常接受延后实现
```

## 影响评估维度

| 维度 | 评估内容 | 影响等级 |
|------|---------|---------|
| **用户体验** | 用户是否能感知变化？变化是否影响使用？ | 高/中/低/无 |
| **业务价值** | 是否影响核心业务目标？是否影响营收？ | 高/中/低/无 |
| **MVP 范围** | 偏差是否导致 MVP 功能缺失？ | 是/否 |
| **用户接受度** | 目标用户是否能接受变化？ | 高/中/低 |

### 影响等级判断

```
用户体验影响：
- 高：用户操作习惯改变、核心体验下降
- 中：用户需适应新方式、体验有变化但不严重
- 低：用户几乎无感知、细节调整
- 无：用户完全无感知

业务价值影响：
- 高：影响核心业务目标、营收影响
- 中：影响次要业务、有替代路径
- 低：影响边缘功能、影响小
- 无：无业务影响

用户接受度：
- 高：用户完全接受或更满意
- 中：用户能接受但需适应
- 低：用户不满意、需解释说明
```

## 产品决策

### 决策选项

| 决策 | 适用场景 | 后续行动 |
|------|---------|---------|
| **接受偏差** | 技术限制型/优化建议型，用户体验影响低 | 更新 PRD，技术侧按新方案继续 |
| **坚持原需求** | 业务价值影响高，核心功能不能妥协 | 说明理由，技术侧需优化方案 |
| **协商调整** | 双方都有合理诉求，需共同确定方案 | 与 tech-manager 讨论，确定折中方案 |

### 决策判断矩阵

| 偏差类型 | 用户体验影响 | 业务价值影响 | 推荐决策 |
|---------|-------------|-------------|---------|
| 技术限制型 | 低 | 低 | 接受偏差 |
| 技术限制型 | 高 | 高 | 协商调整（可能坚持） |
| 逻辑缺陷型 | - | - | 接受偏差（补充逻辑） |
| 优化建议型 | 低/无 | 低/无 | 接受偏差 |
| 优化建议型 | 与设计冲突 | - | 坚持原需求或协商 |
| 约束新增型 | 低 | 低 | 接受偏差（延后） |
| 约束新增型 | 高 | 高 | 协商调整（更换方案） |

## PRD 更新规范

### 更新位置

偏差决策后，需更新 PRD 对应章节：

| 偏差类型 | 更新章节 | 更新内容 |
|---------|---------|---------|
| 技术限制型 | 非功能需求 / 功能详细描述 | 调整性能要求 / 调整实现方式说明 |
| 逻辑缺陷型 | 业务流程 / 状态流转 | 补充缺失流程 / 补充状态流转 |
| 优化建议型 | 功能详细描述 | 补充新增功能点说明 |
| 约束新增型 | 产品范围 / 时间规划 | 调整 MVP 范围 / 延后功能 |

### 修订记录格式

```markdown
## 修订记录

| 版本 | 日期 | 修订人 | 修订内容 |
|------|------|--------|----------|
| 1.1 | 2026-04-15 | 产品经理 | 偏差#1决策：库存展示改为异步同步，标注"仅供参考" |
```

### 章节标注格式

```markdown
### 3.2 商品列表

**商品库存展示**（偏差决策）：
- 原需求：实时显示库存数量
- 调整：异步同步（每5分钟），页面标注"库存仅供参考"
- 原因：技术性能限制，实时计算超 500ms
- 决策时间：2026-04-15
```


## 严重等级响应时效

| 严重等级 | 定义 | 产品经理响应时效 |
|---------|------|----------------|
| **P0-阻塞** | 核心功能无法按 PRD 实现 | 立即确认 |
| **P1-重要** | 非核心功能偏差，影响用户体验 | 1 个工作日内 |
| **P2-一般** | 细节偏差，不影响核心流程 | 3 个工作日内 |
| **P3-建议** | 优化建议，提升体验 | 下次迭代评审 |

## 协同模式：状态更新

**仅在协同开发模式下执行**（检测条件：项目根目录存在 `docs/status.md`）。

### 偏差决策后的状态更新

决策完成后，需更新 `docs/status.md`：

**更新内容**：
1. 将偏差反馈项从"待处理项汇总"移除
2. 在"下一步任务"中添加 tech-manager 的执行项

**更新模板**（参考 `../team-collaboration/references/12-status-update-templates.md`）：

```markdown
## 待处理项汇总（更新后）
| 类型 | 模块 | 文件路径 | 处理角色 | 状态 |
|------|------|---------|---------|------|
| 偏差决策 | product | docs/reports/deviation-product.md | product-manager | ✅ 已决策 |

## 下一步任务（新增）
| 角色 | 任务 | 输入文件 | 输出文件 |
|------|------|---------|---------|
| tech-manager | 通知开发者执行 | docs/decisions/decision-product.md | 执行确认 |
```

### 状态更新流程

```
决策完成
    ↓
读取 ../team-collaboration/references/05-status-mechanism.md
    ↓
更新 docs/status.md
    ├── 移除偏差反馈项
    ├── 标记决策已完成
    ├── 添加 tech-manager 下一步任务
    ↓
git commit -m "偏差决策完成: {模块名}"
```
