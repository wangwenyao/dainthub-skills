# 业务建模流程

业务建模专项流程，从角色定义到业务规则的完整建模。

---

## ⚠️ 硬约束（MANDATORY）

**必须遵守以下规则，违反即为流程执行失败**：

### 1. 数量传递约束（需求点→业务活动）

**前置输入验证**：
```
需求分析输出：N 个需求点
    ↓ 强制接收
业务建模输入：必须处理这 N 个需求点（不得遗漏）
```

**需求点→业务活动映射表（强制输出）**：

| 需求点ID | 需求名称 | 对应业务活动 | 是否处理 | 缺失说明 |
|----------|----------|--------------|----------|----------|
| R01 | 商品管理 | BA01 商品管理活动 | ✓ | |
| R02 | 订单管理 | BA02 订单管理活动 | ✓ | |
| R03 | 库存管理 | BA03 库存管理活动 | ✓ | |
| R04 | 用户角色定义 | BA04 角色定义 | ✓ | |
| R05 | 数据实体定义 | BA05 实体定义 | ✓ | |
| ... | ... | ... | ... | ... |

**约束规则**：
- 任何需求点无对应业务活动 → 流程暂停，需补充
- 需求点数量 N → 业务活动数量 ≥ N（每个需求点至少1个活动）
- 用户明确说"不需要"的需求点 → 标记 ✗ 并记录原因

---

### 2. 逐项处理清单（必须逐项输出）

**步骤一执行时（定义业务角色）**：

输入：从需求分析传递的角色需求

```
逐项处理清单（必须逐项输出）：
├── ☐ 角色需求 R01 → 正在处理...
│   └── 输出：角色定义 R01（管理员）
├── ☐ 角色需求 R02 → 正在处理...
│   └── 输出：角色定义 R02（运营人员）
├── ☐ 角色需求 R03 → 正在处理...
│   └── 输出：角色定义 R03（客服人员）

完成确认：所有角色需求已处理完毕
```

---

### 3. 四要素输出约束

| 规则 | 说明 |
|------|------|
| **禁止跳步骤** | 必须按顺序执行，不得跳过任何步骤 |
| **必须输出四要素** | 业务角色 + 业务流程 + 业务活动 + 业务规则 |
| **必须引用规范** | 详细格式见 `@business-modeling.md` |
| **必须覆盖异常** | 每个流程必须包含异常场景 |

---

## 输入与输出

| 输入 | 来源 |
|------|------|
| 业务需求 | 需求分析阶段输出（N 个需求点） |
| 用户画像 | PRD 第四章 |

| 输出 | 说明 |
|------|------|
| 业务角色模型 | 角色定义 + 权限矩阵 + 组织架构 |
| 业务流程矩阵 | 流程清单 + 主流程 + 异常流程 |
| 业务活动清单 | 活动 + 流程关联 + 功能点对应 |
| 业务规则清单 | 规则定义 + 适用场景 + 实体关联 |
| **需求点→活动映射表** | **数量传递验证** |

---

## 执行步骤

### 步骤一：接收需求点清单并验证

**目标**：接收需求分析的输出，验证数量传递。

**输入验证**：
```
接收需求分析输出的 N 个需求点
验证：是否所有需求点都有对应业务活动？
    ├── ✓ 有对应 → 继续
    └── ❌ 无对应 → 暂停，补充业务活动
```

**强制输出**：需求点→业务活动映射表

---

### 步骤二：定义业务角色

**目标**：识别所有业务相关角色，定义职责和权限。

**方法**：
1. 从用户画像中提取角色
2. 区分角色类型：系统角色、业务角色、外部角色
3. 定义每个角色的职责和操作范围
4. 构建权限矩阵

**逐项处理**（必须逐项输出）：
```
输入角色需求 →逐项定义 → 输出角色定义
```

**输出格式**（见 `@business-modeling.md#业务角色模型`）：

| 角色ID | 角色名称 | 角色类型 | 角色职责 | 操作范围 |
|--------|----------|----------|----------|----------|
| R01 | 管理员 | 系统角色 | 系统管理、数据维护 | 全局 |
| R02 | 运营人员 | 业务角色 | 商品管理、订单处理 | 商品模块、订单模块 |

**权限矩阵**：

| 角色 | 商品管理 | 订单管理 | 用户管理 |
|------|----------|----------|----------|
| 管理员 | 全权限 | 全权限 | 全权限 |
| 运营人员 | 查看/新增/编辑 | 查看/处理 | 无 |

**验证**：
- 所有用户画像角色都已定义
- 权限矩阵覆盖所有功能模块
- 角色职责与功能架构对应

---

### 步骤三：定义业务流程

**目标**：识别核心业务流程，定义主流程和异常流程。

**方法**：
1. 从业务需求中提取关键业务场景
2. 识别所有核心业务流程（按实际复杂度定义，不做上限限制）
3. 每个流程包含≥3节点时，必须给出 Mermaid 流程图
4. 定义每个流程的触发条件、步骤、参与者
5. 必须定义异常流程

**流程类型**：

| 流程类型 | 数量规则 | 说明 |
|---------|---------|------|
| 核心业务流程 | ≥3节点必须给出，按实际复杂度定义 | 主线业务操作 |
| 审批流程 | 按需 | 多角色协作流程 |
| 异常流程 | 必须覆盖 | 错误处理、回退流程 |

**输出格式**（见 `@business-modeling.md#业务流程矩阵`）：

| 流程ID | 流程名称 | 流程类型 | 涉及角色 | 优先级 |
|--------|----------|----------|----------|--------|
| BF01 | 商品发布流程 | 核心流程 | 管理员、运营 | P0 |
| BF02 | 订单处理流程 | 核心流程 | 管理员、客服 | P0 |
| BF03 | 审批驳回处理 | 异常流程 | 审批人、申请人 | P0 |

**Mermaid 流程图**：
- 主流程：使用 `flowchart TD`
- 状态流转：使用 `stateDiagram-v2`

**验证**：
- 流程数量 ≤ 5 个（避免过度细化）
- 每个流程有触发条件
- 异常流程已定义

---

### 步骤四：定义业务活动

**目标**：将流程拆解为具体业务活动，关联功能点。

**方法**：
1. 每个流程拆解为业务活动
2. 活动类型：触发活动、执行活动、决策活动、结束活动
3. 关联每个活动对应的功能点

**输出格式**（见 `@business-modeling.md#业务活动定义`）：

| 活动ID | 活动名称 | 所属流程 | 执行角色 | 对应功能点 |
|--------|----------|----------|----------|------------|
| BA01 | 提交商品信息 | BF01 商品发布 | 运营人员 | F0103 新增商品 |
| BA02 | 审核商品信息 | BF01 商品发布 | 管理员 | F0107 商品审核 |

**验证**：
- 每个流程的活动已完整定义
- 每个活动有对应的功能点
- 活动执行角色已明确

---

### 步骤五：定义业务规则

**目标**：识别业务规则，分类定义。

**规则类型**：

| 规则类型 | 说明 | 示例 |
|---------|------|------|
| 数据规则 | 字段约束、格式校验 | 价格 ≥ 0 |
| 流程规则 | 流程流转条件 | 审批通过 → 发布 |
| 计算规则 | 业务计算逻辑 | 总价 = 单价 × 数量 |
| 约束规则 | 业务约束条件 | 库存充足才能下单 |
| 状态规则 | 状态流转条件 | 待审核 → 已发布 |

> **⚠️ 状态规则硬约束**：含状态流转的实体必须输出状态转换矩阵
> 
> `stateDiagram-v2` Mermaid 图为辅助可视化，状态转换矩阵为必要实现依据。
> 
> 状态转换矩阵格式见 `@business-modeling.md#4.1.1 状态转换矩阵模板`

**输出格式**（见 `@business-modeling.md#业务规则体系`）：

| 规则ID | 规则名称 | 规则类型 | 适用场景 | 适用实体 |
|--------|----------|----------|----------|----------|
| BR01 | 商品名称唯一 | 数据规则 | 新增/编辑商品 | 商品 |
| BR02 | 库存校验 | 约束规则 | 下单 | 订单 |

**验证**：
- P0 流程的业务规则已定义
- 规则类型分类正确
- 规则适用实体已明确

---

### 步骤六：验证完整性

**检查清单**：

| 检查项 | 要求 |
|--------|------|
| **需求点覆盖** | 所有 N 个需求点都有对应业务活动 |
| 角色覆盖 | 所有用户画像角色已定义 |
| 流程覆盖 | 核心业务场景有对应流程 |
| 异常覆盖 | 每个流程有异常处理 |
| 活动对应 | 每个活动有功能点对应 |
| 规则覆盖 | P0 流程有业务规则 |

---

## 与其他流程关系

| 前置流程 | 说明 |
|---------|------|
| 需求分析 | 提供业务需求输入（N 个需求点） |

| 后续流程 | 说明 |
|---------|------|
| 功能架构设计 | 提供角色、流程、活动输入（数量传递） |

## 执行信号

以下行为表明流程执行失败：
- 需求点→业务活动映射表缺失
- 需求点无对应业务活动
- 逐项处理清单未逐项输出
- 四要素输出不完整
