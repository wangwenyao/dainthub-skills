# PRD编写流程

**何时使用**: 编写产品需求文档、功能规格说明书时

---

## ⚠️ 硬约束（MANDATORY）

**必须遵守以下规则，违反即为流程执行失败**：

### 0. 自适应文档结构约束

**根据端划分自动选择文档结构**：

```
端划分确认（需求分析输出）
    ├── 端数量 ≥ 2 → 三层结构（总览 PRD + 各端独立 PRD）
    ├── 端数量 = 1 + 模块 ≥ 3 → 两层结构（单一 PRD 按模块分章节）
    ├── 端数量 = 1 + 模块 ≤ 2 → 单层结构（精简 PRD）
    └── 无端划分 → 默认两层结构

文档结构选择规则：
├── 三层结构：先写总览 PRD，再写各端独立 PRD
├── 两层结构：完整 PRD，按模块分章节
└── 单层结构：精简 PRD，合并章节

模板引用 → @../prd-template.md#自适应文档结构
```

---

### 0.1 功能复杂度判断约束

**每个功能必须先判断复杂度，再决定描述方式**：

```
复杂度判断流程（必须逐项输出）：
├── ☐ F01 [功能名称] → 判断复杂度...
│   ├── 是否通用功能（查看/搜索/新增/编辑/删除/导出）？
│   │   ├── 是 → 通用功能
│   │   │   ├── 有特殊业务规则 → 标准（引用模板 + 差异）
│   │   │   └── 无特殊业务规则 → 简单（一句话）
│   │   └── 否 → 业务特定功能
│   │   │   ├── 步骤 ≤ 3 无分支 → 简单
│   │   │   ├── 步骤 3-5 有校验 → 标准（完整描述）
│   │   │   └── 步骤 ≥ 5 或复杂规则 → 复杂（完整描述）
│   └── 输出：[复杂度] + [描述方式] ✓
├── ☐ F02 [功能名称] → 判断复杂度...
│   └── 输出：[复杂度] + [描述方式] ✓
...

复杂度统计：
├── 简单功能：X 个（一句话描述）
├── 标准功能：Y 个（引用模板或完整描述）
└── 复杂功能：Z 个（完整描述）
```

### 0.2 通用模板引用约束

**通用功能必须优先引用模板**：

| 模板ID | 适用功能 | 引用方式 |
|--------|----------|----------|
| T01 | 查看列表 | 模板 + 字段差异 |
| T02 | 搜索筛选 | 模板 + 搜索条件差异 |
| T03 | 新增表单 | 模板 + 字段差异 |
| T04 | 编辑表单 | 模板 + 可编辑字段差异 |
| T05 | 删除确认 | 模板 + 删除条件差异 |
| T06 | 导出功能 | 模板 + 导出格式差异 |

**模板库位置**：总览 PRD 第五章「共享通用功能模板库」

**引用格式**（必须使用）：
```markdown
#### F01 [功能名称]

**模板引用**: T01 查看列表模板 @总览PRD#5.1

**差异标注**:
- 列表字段: [字段1, 字段2, 字段3] (差异)
- 排序: [创建时间降序] (差异)

**前置条件**: [用户已登录且有权限]
**权限矩阵**: [角色A: ✓, 角色B: ✗]
```

**禁止行为**：
- ❌ 通用功能不引用模板，完整描述
- ❌ 复杂业务功能引用模板但不补充规则 → 描述不完整
- ❌ 不判断复杂度直接完整描述
- ❌ 引用模板不标注来源（缺少@总览PRD#章节）

### 0.3 功能清单表格格式约束

**⚠️ 强制格式：一级合并、二级合并、三级按行**

必须按以下格式输出功能清单表格：

```
正确格式示例：
| 一级模块 | 二级功能 | 三级功能点 | 优先级 | 页面/组件 | 用户角色 | 业务活动 |
|----------|----------|------------|--------|-----------|----------|----------|
| M01 登录认证 | F01 登录页 | F0101 账号密码登录 | P0 | P001 登录页 | 会员 | BA01 |
| | | F0102 微信授权登录 | P0 | P001 登录页 | 会员 | BA01 |
| M02 首页服务 | F02 首页 | F0201 园区介绍查看 | P0 | P002 首页 | 会员 | BA02 |
| | | F0202 服务入口导航 | P0 | P002 首页 | 会员 | BA02 |
```

**格式规则**：
- 一级模块：只在模块第一行填写，后续行留空（视觉合并）
- 二级功能：只在功能第一行填写，后续行留空（视觉合并）
- 三级功能点：每行独立填写，不合并
- 对应业务活动：每行必须标注，实现业务↔功能双向追溯

**违例检测**：
- 每行都填写完整一级模块名 → ❌ 格式错误
- 每行都填写完整二级功能名 → ❌ 格式错误
- 缺少"对应业务活动"列 → ❌ 格式错误

**未按此格式输出 → 流程执行失败**

### 0.4 功能详细描述完整性约束

**P0/P1 功能必须完整填写以下9项**：

| 序号 | 必填项 | 说明 | 适用复杂度 |
|------|--------|------|------------|
| 1 | 功能描述 | 一句话说明功能目的 | 全部 |
| 2 | 功能点清单 | 表格列出三级功能点 | 全部 |
| 3 | 前置条件 | 用户登录、权限、数据依赖 | 全部 |
| 4 | 权限矩阵 | 各角色的操作权限表格 | 全部 |
| 5 | 字段约束 | 表单字段的校验规则（含错误提示文案+校验层级） | 标准/复杂 |
| 6 | 业务规则 | 引用BR编号 | 标准/复杂 |
| 7 | 验收标准 | Given-When-Then格式，四象限测试矩阵 | 标准/复杂 |
| 8 | 引用标注来源 | 标注@文件名#章节 | 全部 |
| 9 | 设计决策 | 备选方案、选择原因（仅复杂功能） | 复杂（P0必填） |

**完整模板**：
```markdown
#### F01 [二级功能名称]

**功能描述**
[一句话说清楚这个功能做什么、解决什么问题]

**功能点清单**
| 功能点ID | 功能点名称 | 页面/组件 | 优先级 | 对应业务活动 |
|----------|------------|-----------|--------|--------------|
| F0101 | 查看列表 | P001 列表页 | P0 | A01 |
| F0102 | 搜索筛选 | P001 搜索区 | P0 | A01 |

**前置条件**
- [用户已登录且有权限]

**权限矩阵**
| 角色 | 查看 | 新增 | 编辑 | 删除 |
|------|------|------|------|------|
| 管理员 | ✓ | ✓ | ✓ | ✓ |
| 运营 | ✓ | ✓ | ✓ | ✗ |

**字段约束** @../prd-writing-guide.md#字段约束规范
| 字段名 | 类型 | 必填 | 约束规则 | 错误提示文案 | 校验层级 |
|--------|------|------|----------|-------------|----------|
| 名称 | 文本 | 是 | 长度1-50 | "请输入名称" | 前端 |

**业务规则** @../business-modeling.md#业务规则体系
- BR01: 名称不可重复

**验收标准**
| 场景类型 | Given | When | Then |
|----------|-------|------|------|
| 正向用例 | 用户已登录且有权限 | 进入列表页 | 显示数据列表 |
| 边界值 | 列表数据超过100条 | 进入列表页 | 只显示前100条，提示分页 |
| 负向用例 | 用户未登录 | 进入列表页 | 跳转登录页 |
| 权限异常 | 用户无查看权限 | 进入列表页 | 提示"无权限访问" |

**设计决策**（仅复杂功能，P0 必填）
- 备选方案：[方案A / 方案B]
- 选择方案：[最终选择]
- 选择原因：[为什么选择]
- 接受的权衡：[牺牲了什么]
- 约束条件：[未来可能改变的条件]
```

**未完整填写9项 → P0/P1功能描述不完整 → 流程执行失败**

### 0.5 引用标注来源约束

**⚠️ 强制标注：所有引用必须标注来源**

引用规范、模板、业务规则时，必须标注来源：

| 引用类型 | 标注格式 | 示例 |
|----------|----------|------|
| 模板引用 | **模板引用**: T01 @总览PRD#5.1 | T01 @总览PRD#5.1 |
| 规范引用 | > 规范见 @文件名#章节 | > 字段约束见 @../prd-writing-guide.md#2.5 |
| 业务规则引用 | - BR01 @../business-modeling.md#业务规则体系 | - BR03 @../business-modeling.md#业务规则体系 |

**禁止凭记忆推断**：
- ❌ "模板引用就是简写" → 未标注来源，凭记忆推断
- ✓ 实际读取规范文件 → 输出：**字段约束** @../prd-writing-guide.md#字段约束规范

**未标注来源 → 引用无效**
**凭记忆推断 → 流程执行失败**

### 1. 数量传递约束（功能点→PRD章节）

**前置输入验证**：
```
功能架构输出：N 个二级功能 + M 个三级功能点
    ↓ 强制接收
PRD编写输入：必须处理这 N 个二级功能 + M 个三级功能点（不得遗漏）
```

**功能点→章节覆盖表（强制输出）**：

| 二级功能ID | 二级功能名称 | 三级功能点数量 | 详细描述状态 | 缺失说明 |
|------------|--------------|----------------|--------------|----------|
| F01 | 商品列表 | 5 | ✓ 已描述 | |
| F02 | 商品分类 | 4 | ✓ 已描述 | |
| F03 | 订单列表 | 6 | ❌ 未描述 | 需补充 |

**约束规则**：
- P0/P1 二级功能必须详细描述，任何缺失 → PRD 不完整
- P2 功能可简要描述或省略
- **含表单的 P0/P1 功能点，字段约束表必须填写"错误提示文案"和"校验层级"两列，缺失视为 PRD 不完整**

### 2. 逐项填充约束（验证式输出）

**⚠️ 必须输出实际执行内容，不能只声明"正在处理"**

编写功能详细描述时，必须按以下验证式格式逐项输出：

```
逐项填充清单（验证式输出）：
├── ☐ F01 商品列表 → 正在编写详细描述...
│   ├── 步骤1：编写功能描述 → "查看商品列表，支持分页浏览"
│   ├── 步骤2：编写功能点清单 → 表格输出F0101-F0105（5行）
│   ├── 步骤3：编写前置条件 → "用户已登录且有查看权限"
│   ├── 步骤4：编写权限矩阵 → 表格输出管理员/运营权限（3行）
│   ├── 步骤5：编写字段约束 → 表格输出字段校验规则（4行）@../prd-writing-guide.md#2.5
│   ├── 步骤6：编写业务规则 → "BR01: 名称不可重复" @../business-modeling.md#业务规则体系
│   ├── 步骤7：编写验收标准 → "Given用户已登录When进入列表页Then显示列表"
│   ├── 步骤8：标注引用来源 → 模板引用@总览PRD#5.1 ✓
│   └── 验证：实际输出 = 功能描述1行 + 功能点清单表5行 + 前置条件1行 + 权限矩阵表3行 + 字段约束表4行 + 业务规则1行 + 验收标准1行 ✓
├── ☐ F02 商品分类 → 正在编写详细描述...
│   ├── 步骤1：编写功能描述 → "管理商品分类，支持新增编辑删除"
│   ├── 步骤2：编写功能点清单 → 表格输出F0201-F0204（4行）
│   ├── 步骤3：编写前置条件 → "用户已登录且有管理权限"
│   ├── 步骤4：编写权限矩阵 → 表格输出管理员权限（2行）
│   ├── 步骤5：编写字段约束 → 表格输出字段校验规则（3行）
│   ├── 步骤6：编写业务规则 → "BR02: 分类名称不可重复"
│   ├── 步骤7：编写验收标准 → "Given用户有权限When点击新增Then打开表单弹窗"
│   ├── 步骤8：标注引用来源 → @../prd-writing-guide.md#功能详细描述模板 ✓
│   └── 验证：实际输出行数统计 ✓
...

完成确认：所有 P0/P1 二级功能已详细描述完毕
验证统计：
├── 已执行：F01-F23（23项）
├── 实际输出内容：功能描述段落 + 功能点清单表格 + 前置条件 + 权限矩阵表格 + 字段约束表格 + 业务规则 + 验收标准
└── 每项都有可检查的输出内容 ✓
```

**禁止行为**：
- ❌ 只输出"正在处理... ✓"不输出实际内容 → 声明式，无证据
- ❌ 批量输出多个"✓"不逐项执行 → 批量声明，可能跳过
- ✓ 必须输出实际执行内容（功能描述、表格等具体内容）
- ✓ 必须输出验证证据（行数统计、内容检查）

### 3. 生成前必须动作

```
1. 确认已读取以下文件（不得凭记忆推断）：
   ├── @../prd-template.md       # PRD模板（必须读取）
   ├── @../prd-writing-guide.md  # PRD编写规范（必须读取）
   └── @../business-modeling.md  # 业务建模规范（必须读取）
   
2. 确认用户需求类型：
   ├── 新产品 PRD → 完整执行全部章节
   ├── 功能变更 → 重点更新功能架构 + 验收标准
   └── 补充文档 → 只更新指定章节

3. 接收功能架构输出：
   ├── 二级功能数量：N 个
   ├── 三级功能点数量：M 个
   └── 数量传递验证：是否所有功能点都有详细描述？
```

### 4. 生成中必须动作

| 章节 | 必须检查 |
|------|----------|
| 功能清单 | 对照 prd-writing-guide.md#功能架构规范，确保三级功能点完整 |
| 功能清单表格格式 | 一级合并（只填第一行）、二级合并（只填第一行）、三级按行 |
| 功能清单对应业务活动 | 每行必须有对应业务活动ID |
| 功能详细描述 | **逐项填充清单（验证式）**，P0/P1 功能必须完整填写9项 |
| 验收标准四类场景 | **P0/P1 功能验收标准必须覆盖：正向用例 + 边界值 + 负向用例 + 权限异常**，使用四象限测试矩阵格式 |
| 业务角色模型 | 使用 business-modeling.md 模板定义角色、权限矩阵 |
| 业务流程矩阵 | 使用 business-modeling.md 模板定义流程清单 + 主流程 + 异常流程 |
| 业务活动与规则 | 使用 business-modeling.md 模板定义活动清单 + 规则清单 |
| Mermaid 图表 | 必须语法正确可渲染，不得使用错误语法 |
| 非功能需求 | 必须引用基准值，不得写模糊描述 |
| 引用标注来源 | 所有引用必须标注@文件名#章节 |

### 5. 生成后必须动作

**⚠️ 必须输出检查结果验证表（验证式，不能只声明）**

```
检查结果验证表（基于实际输出验证）：
| 检查项 | 规范要求 | 实际验证方式 | 验证结果 | 状态 |
|--------|----------|--------------|----------|------|
| PRD必需章节 | 10个P0章节 | 实际读取PRD → 检查章节标题 | 共10章 | ✓/❌ |
| 功能清单表格格式 | 一级合并、二级合并 | 实际读取表格 → 检查一级模块出现次数 | M01出现次数应为1 | ✓/❌ |
| 功能清单对应业务活动 | 每行必须有对应活动 | 实际读取表格 → 检查最后一列 | 每行都有业务活动ID | ✓/❌ |

| **验收标准四类场景** | **正向+边界值+负向+权限异常** | **实际读取验收标准 → 检查四象限矩阵是否完整** | **四类场景均有覆盖** | **✓/❌** |

| 逐项填充清单 | 必须逐项输出证据 | 实际读取清单 → 检查是否有实际内容 | 每项有功能描述+表格等 | ✓/❌ |
| 业务角色模型完整 | 角色定义+权限矩阵 | 实际读取内容 → 检查是否有完整表格 | 有角色定义表+权限矩阵表 | ✓/❌ |

违规项汇总：
├── ❌ [违规项1]: [规范要求] vs [实际执行]
├── ❌ [违规项2]: [规范要求] vs [实际执行]
状态：如有违规项，需修正后重新验证
```

**禁止行为**：
- ❌ 只输出"✓ 已完成"不提供验证证据
- ❌ 批量标注多个检查项为✓不实际验证
- ✓ 必须输出实际验证方式和具体结果

---

### 6. 禁止行为

| 禁止 | 原因 |
|------|------|
| ❌ 不读取模板文件就开始生成 | 可能使用错误格式 |
| ❌ 功能清单只有二级功能 | 缺少三级功能点定义 |
| ❌ 功能清单表格每行填写完整一级/二级名 | 未遵循合并规则 |
| ❌ 功能清单缺少对应业务活动列 | 无法追溯业务↔功能 |
| ❌ 批量处理功能详细描述 | 必须逐项填充 |
| ❌ 只输出"正在处理... ✓"不输出实际内容 | 声明式，无证据 |
| ❌ P0/P1功能描述不完整（缺失9项中任一项） | 描述不完整 |
| ❌ 跳过 P0/P1 功能详细描述 | 必须完整展开 |
| ❌ 引用不标注来源（缺少@文件名#章节） | 凭记忆推断 |
| ❌ 业务角色模型简化 | 必须包含角色定义+权限矩阵 |
| ❌ 业务流程只有主流程 | 必须包含异常流程 |
| ❌ 非功能需求写模糊描述 | 必须量化，引用基准值 |
| ❌ 生成后不输出检查结果验证表 | 必须验证 |
| ❌ 不输出功能点→章节覆盖表 | 必须输出数量传递验证 |

---

### 7. 违规处理机制

**⚠️ 发现违规必须立即停止，不能继续执行**

```
违规处理流程：
├── 步骤1：立即停止当前流程 ← 强制停止
├── 步骤2：输出违规说明表 ← 必须输出表格
├── 步骤3：不继续执行后续步骤 ← 禁止掩盖
├── 步骤4：等待用户确认后修正 ← 不能自动修正
└── 步骤5：修正后重新验证 ← 重新输出验证表

违规说明表格式：
| 违规项 | 规范要求 | 实际执行 | 差异说明 | 修正方案 |
|--------|----------|----------|----------|----------|
| 功能清单表格格式 | 一级模块只填第一行 | 每行都填写完整模块名 | M01重复出现11次 | 删除后续行的一级模块名，改为空 |
| 功能详细描述完整性 | P0功能9项必填 | F02只有3项 | 缺失字段约束、业务规则、验收标准 | 补充缺失项 |
| 引用标注来源 | 必须标注@总览PRD#章节 | 只写"模板引用T01" | 缺少来源标注 | 补充@总览PRD#5.1 |

当前状态：流程暂停，等待用户确认修正方案
禁止行为：继续执行后续步骤、不输出违规说明表、自动修正不等待确认
```

**发现违规后继续执行 → 流程执行失败**

## 流程步骤

### 步骤零：确定文档结构

**目标**：根据端划分确定 PRD 文档结构。

**输入**：需求分析输出的端划分确认

**判断流程**：
```
端数量 ≥ 2？
├── 是 → 三层结构
│   ├── 输出总览 PRD（包含模板库）
│   └── 输出各端独立 PRD
└── 否 → 检查模块数量
    ├── 模块 ≥ 3 → 两层结构
    │   └── 输出完整 PRD
    └── 模块 ≤ 2 → 单层结构
        └── 输出精简 PRD
```

**输出**：文档结构决定 + 模板选择

---

### 步骤一：接收功能架构输出并验证

**目标**：接收功能架构的输出，验证数量传递。

**输入验证**：
```
接收功能架构输出的 N 个二级功能 + M 个三级功能点
验证：是否所有 P0/P1 功能都有详细描述？
    ├── ✓ 有描述 → 继续
    └── ❌ 无描述 → 暂停，需补充详细描述
```

**强制输出**：功能点→章节覆盖表（初始状态，标记待描述）

---

### 步骤二：功能复杂度判断

**目标**：逐项判断每个功能的复杂度，决定描述方式。

**输入**：功能架构输出的二级功能清单

**判断流程**（必须逐项输出）：
```
逐项判断清单：
├── ☐ F01 [功能名称] → 判断复杂度...
│   ├── 功能类型：[通用/业务特定]
│   ├── 步骤数量：[X 步]
│   ├── 业务规则：[简单/标准/复杂]
│   └── 输出：[复杂度] + [描述方式] ✓
├── ☐ F02 [功能名称] → 判断复杂度...
│   └── 输出：[复杂度] + [描述方式] ✓
...

复杂度统计输出：
| 复杂度 | 功能数量 | 描述方式 |
|--------|----------|----------|
| 简单 | X 个 | 一句话 |
| 标准 | Y 个 | 引用模板/完整描述 |
| 复杂 | Z 个 | 完整描述 |
```

**模板引用判断**：
```
通用功能（查看/搜索/新增/编辑/删除/导出）？
├── 是 → 引用模板 + 差异标注
│   ├── 差异简单 → 一句话差异
│   └── 差异复杂 → 列表差异 + 补充规则
└── 否 → 业务特定功能
    ├── 标准 → 完整描述（使用模板结构）
    └── 复杂 → 完整描述 + 详细规则 + **设计决策** ← P0 复杂功能必填
```

> **设计决策提示**：复杂功能（P0）必须输出设计决策
> - 备选方案 + 选择方案 + 选择原因 + 接受的权衡 + 约束条件
> - 帮助开发理解设计意图，便于后续迭代判断原始约束是否还成立

---

### 步骤三：编写核心章节

```
执行摘要 → 产品愿景、核心价值、目标用户
成功标准 → 可量化指标、验收标准、时间节点
产品范围 → In Scope / Out of Scope / MVP定义
详细模板 → @../prd-template.md
```

---

### 步骤四：编写业务角色模型

```
角色定义 → 角色ID、名称、类型、职责、操作范围
权限矩阵 → 各角色的操作权限矩阵
组织架构 → 角色层级关系
详细规范 → @../business-modeling.md#业务角色模型
```

---

### 步骤五：编写用户旅程

```
核心用户流程路径 → 用户从入口到完成目标的页面路径
异常路径 → 用户操作失败时的路径
说明 → 用户旅程是用户视角，业务流程是业务视角，两者互补
```

---

### 步骤六：编写业务流程矩阵

```
流程清单 → 流程ID、名称、类型、涉及角色
主流程 → ≥3节点必须给出（Mermaid），按实际复杂度定义
异常流程 → 必须覆盖
状态流转图 → 如有需要
详细规范 → @../business-modeling.md#业务流程矩阵
```

---

### 步骤七：编写业务活动与规则

```
业务活动清单 → 活动ID、名称、所属流程、对应功能点
业务规则清单 → 规则ID、名称、类型、适用场景
关键规则详细描述 → P0规则需展开
详细规范 → @../business-modeling.md#业务活动定义 + #业务规则体系
```

---

### 步骤八：编写功能架构（逐项填充 + 验证式输出）

**目标**：按逐项填充清单，逐个二级功能编写详细描述。

**逐项填充**（验证式输出）：
```
输入二级功能清单 → 逐项编写详细描述 → 输出详细描述章节
每项完成后：
├── 输出实际执行内容（功能描述、表格等）
├── 输出验证证据（行数统计、内容检查）
└── 更新功能点→章节覆盖表状态
```

```
功能清单：
├── 二级功能清单 → 功能ID、名称、优先级、模块、描述
├── 三级功能点清单 → 功能点ID、名称、页面/组件、用户角色、对应业务活动
├── 功能菜单层级结构 → 树形结构
└── 功能详细描述 → P0/P1功能需展开（逐项填充，验证式输出）

详细规范 → @../prd-writing-guide.md#功能架构规范
```

---

### 步骤九：设计业务实体模型

```
实体定义 → 识别核心业务实体
实体属性 → 字段定义
实体关系 → ER图（Mermaid erDiagram）
详细规范 → @../prd-writing-guide.md#业务实体模型规范
```

---

### 步骤十：编写非功能需求

```
性能 → 响应时间、并发、分页上限
安全 → 认证、权限、数据隔离、审计
可用性 → SLA、备份、浏览器兼容
基准值参考 → @../prd-writing-guide.md#非功能需求量化规范
```

---

### 步骤十一：质量检查

**目标**：输出检查结果验证表，验证完整性。

**强制输出**（验证式）：
```
检查结果验证表（基于实际输出验证）：
| 检查项 | 规范要求 | 实际验证方式 | 验证结果 | 状态 |
...

功能点→章节覆盖表（最终状态）：
├── 所有 P0/P1 二级功能状态为 ✓ 已描述
└── 无缺失项

如有违规项 → 输出违规说明表 → 等待用户确认修正
```

## PRD章节清单

```
必需章节（P0）：
1. 执行摘要
2. 成功标准
3. 产品范围
4. 业务角色模型（角色定义+权限矩阵+组织架构）
5. 用户旅程（核心流程路径+异常路径）
6. 业务流程矩阵（流程清单+主流程+异常流程）
7. 业务活动与规则（活动清单+规则清单）
8. 功能架构（二级+三级功能清单+功能详细描述）
9. 业务实体模型（ER图，Mermaid）
10. 非功能需求（性能/安全/可用性）

可选章节（P1）：
11. 风险与依赖
12. 页面原型（HTML）
13. 时间规划

总览PRD特有章节：
5. 共享通用功能模板库（T01-T06完整定义）
```

## 执行信号

以下行为表明流程执行失败：
- 功能点→章节覆盖表缺失
- P0/P1 二级功能无详细描述
- 逐项填充清单只声明不输出实际内容
- 功能点数量与功能架构输出不匹配
- 功能清单表格格式错误（一级/二级未合并）
- 功能详细描述不完整（缺失9项中任一项）
- 引用未标注来源（缺少@文件名#章节）
- 发现违规后继续执行
