# 功能架构设计流程

功能架构设计专项流程，从业务分析到三级功能点拆解。

---

## ⚠️ 硬约束（MANDATORY）

**必须遵守以下规则，违反即为流程执行失败**：

### 1. 数量传递约束（业务活动→功能点）

**前置输入验证**：
```
业务建模输出：N 个业务活动
    ↓ 强制接收
功能架构输入：必须处理这 N 个业务活动（不得遗漏）
```

**业务活动→功能点映射表（强制输出）**：

| 业务活动ID | 业务活动名称 | 对应功能点 | 是否处理 | 缺失说明 |
|------------|--------------|------------|----------|----------|
| BA01 | 商品管理活动 | F01 商品列表 | ✓ | |
| BA02 | 订单管理活动 | F03 订单列表 | ✓ | |
| BA03 | 库存管理活动 | F05 库存管理 | ✓ | |
| ... | ... | ... | ... | ... |

**约束规则**：
- 任何业务活动无对应功能点 → 流程暂停，需补充
- 业务活动数量 N → 功能点数量 ≥ N（每个活动至少1个功能点）

---

### 2. 最小覆盖约束（三级功能点数量）

**每个二级功能的三级功能点数量约束**：

| 二级功能类型 | 最少三级功能点 | 强制包含 | 说明 |
|--------------|----------------|----------|------|
| 数据管理类 | ≥ 4 个 | 查看 + 搜索 + 新增 + 编辑 | 基础 CRUD |
| 数据管理类（含删除） | ≥ 5 个 | 查看 + 搜索 + 新增 + 编辑 + 删除 | 完整 CRUD |
| 流程审批类 | ≥ 3 个 | 查看 + 详情 + 审批操作 | 流程操作 |
| 统计报表类 | ≥ 2 个 | 查看 + 筛选 | 报表查看 |
| 配置管理类 | ≥ 2 个 | 查看 + 编辑 | 配置维护 |

**约束规则**：
- 二级功能点数量 < 最少要求 → 流程执行失败
- 缺少强制包含的功能点 → 流程执行失败

---

### 3. 逐项处理清单（必须逐项输出）- 验证式输出

**步骤三执行时（拆解三级功能点）**：

输入：二级功能清单

**⚠️ 禁止声明式输出，必须输出实际内容**：

```
错误示例（声明式，禁止）：
├── ☐ F01 商品列表 → 正在处理... ✓
│   └── 输出：F0101 查看、F0102 搜索、F0103 新增、F0104 编辑、F0105 删除

正确示例（验证式，必须）：
├── F01 商品列表
│   ├── 三级功能点拆解（实际输出）：
│   │   | 功能点ID | 功能点名称 | 页面/组件 | 优先级 | 对应业务活动 |
│   │   |----------|------------|-----------|--------|--------------|
│   │   | F0101    | 查看列表   | P001 列表页 | P0 | BA01 商品管理 |
│   │   | F0102    | 搜索筛选   | P001 搜索区 | P0 | BA01 商品管理 |
│   │   | F0103    | 新增商品   | P002 表单弹窗 | P0 | BA02 商品上架 |
│   │   | F0104    | 编辑商品   | P002 表单弹窗 | P0 | BA03 商品修改 |
│   │   | F0105    | 删除商品   | 确认弹窗 | P1 | BA04 商品下架 |
│   └── 验证：功能点数量=5 ≥ 最少要求(5) ✓、强制包含齐全 ✓

├── F02 商品分类
│   ├── 三级功能点拆解（实际输出）：
│   │   | F0201 | 查看分类树 | P003 分类树页 | P0 | BA05 分类管理 |
│   │   | F0202 | 新增分类 | P004 分类表单 | P0 | BA05 分类管理 |
│   │   | F0203 | 编辑分类 | P004 分类表单 | P0 | BA05 分类管理 |
│   │   | F0204 | 删除分类 | 确认弹窗 | P1 | BA05 分类管理 |
│   └── 验证：功能点数量=4 ≥ 最少要求(4) ✓、强制包含齐全 ✓

完成确认：所有二级功能已拆解完毕
验证：每个二级功能的三级功能点数量 ≥ 最少要求
```

**验证要求**：
- 每个二级功能必须输出实际的三级功能点表格内容
- 禁止只声明"正在处理... ✓"而不输出实际表格
- 每个二级功能必须有具体验证项（功能点数量、强制包含）

### 4. 基础约束

| 规则 | 说明 |
|------|------|
| **禁止跳步骤** | 必须按顺序执行，不得跳过任何步骤 |
| **必须输出三级功能点** | 功能清单必须包含二级功能 + 三级功能点两层 |

## 执行步骤

### 步骤一：接收业务活动清单并验证

**目标**：接收业务建模的输出，验证数量传递。

**输入验证**：
```
接收业务建模输出的 N 个业务活动
验证：是否所有业务活动都有对应功能点？
    ├── ✓ 有对应 → 继续
    └── ❌ 无对应 → 暂停，补充功能点
```

**强制输出**：业务活动→功能点映射表

---

### 步骤二：识别一级模块（业务领域）

**目标**：从业务需求中识别核心业务领域，划分一级模块。

**方法**：
1. 分析业务需求中的核心业务场景
2. 按业务领域划分模块（如：商品管理、订单管理、用户管理）
3. 每个模块对应一个一级分类

**输出**：
```
M01 商品管理
M02 订单管理
M03 用户管理
...
```

**验证**：
- 模块数量 ≤ 10 个（B端典型规模）
- 模块划分符合业务语义
- 模块之间边界清晰

---

### 步骤三：定义二级功能（业务功能单元）

**目标**：每个模块下定义二级功能，对应菜单项。

**方法**：
1. 每个模块识别核心业务功能
2. 二级功能 = 用户可执行的完整业务操作单元
3. 对应菜单项（用户可见的功能入口）

**输出**：

| 功能ID | 功能名称 | 优先级 | 模块 | 描述 | 用户角色 |
|--------|----------|--------|------|------|----------|
| F01 | 商品列表 | P0 | M01 商品管理 | 商品数据的查询、管理 | 管理员/运营 |
| F02 | 商品分类 | P0 | M01 商品管理 | 商品分类的维护 | 管理员 |
| F03 | 订单列表 | P0 | M02 订单管理 | 订单数据的查询、处理 | 管理员/客服 |

**编号规则**：
- 功能ID：F01-F99（全局唯一）
- 模块归属在"模块"列标注

---

### 步骤四：拆解三级功能点（CRUD操作）

**目标**：每个二级功能拆解为三级功能点。

**方法**：
1. 每个二级功能按 CRUD 操作拆解
2. 基本功能点：查看列表、搜索、新增、编辑、删除
3. 特殊功能点：审批、分配、导出、批量操作等（按业务需求）

**逐项处理**（必须逐项输出）：
```
输入二级功能 → 逐项拆解 → 输出三级功能点
验证：三级功能点数量 ≥ 最少要求
```

**功能点类型 → 页面/组件映射**：

| 功能点类型 | 页面/组件 | 说明 |
|-----------|-----------|------|
| 查看列表 | P00X 列表页 | 数据展示 |
| 搜索筛选 | P00X 搜索区 | 列表页内组件 |
| 新增 | 表单弹窗 | 点击按钮触发 |
| 编辑 | 表单弹窗 | 与新增共用组件 |
| 删除 | 确认弹窗 | 二次确认 |
| 批量操作 | 确认弹窗 | 批量选择后触发 |
| 导出 | 导出按钮 | 列表页内操作 |
| 审批 | 审批弹窗 | 特殊业务操作 |
| 分配 | 分配弹窗 | 特殊业务操作 |
| 多步配置 | wizard 分步向导 | 复杂配置流程 |
| 任务管理 | kanban 看板 | 任务/工单流转 |
| 数据对比 | comparison 对比视图 | 版本/数据对比 |
| 帮助引导 | help 帮助页 | Onboarding/FAQ |

**输出**：

| 功能点ID | 功能点名称 | 优先级 | 所属功能 | 页面/组件 | 用户角色 | 对应业务活动 |
|----------|------------|--------|----------|-----------|----------|--------------|
| F0101 | 查看商品列表 | P0 | F01 商品列表 | P001 列表页 | 管理员/运营 | BA01 商品管理 |
| F0102 | 搜索商品 | P0 | F01 商品列表 | P001 搜索区 | 管理员/运营 | BA01 商品管理 |
| F0103 | 新增商品 | P0 | F01 商品列表 | P002 表单弹窗 | 管理员 | BA01 商品上架 |
| F0104 | 编辑商品 | P0 | F01 商品列表 | P002 表单弹窗 | 管理员 | BA01 商品修改 |
| F0105 | 删除商品 | P1 | F01 商品列表 | 确认弹窗 | 管理员 | BA01 商品下架 |

**编号规则**：
- 功能点ID：F0101-F0199（二级功能编号 + 功能点序号）
- 对应业务活动：关联到业务活动清单中的活动ID，实现业务↔功能双向追溯

---

### 步骤五：构建功能菜单层级结构

**目标**：输出功能菜单的树形结构。

**格式**：
```
系统名称
├── M01 商品管理（一级模块）
│   ├── F01 商品列表（二级菜单项）
│   │   ├── F0101 查看列表（三级功能点）
│   │   ├── F0102 搜索（三级功能点）
│   │   ├── F0103 新增（三级功能点）
│   │   └── F0104 编辑（三级功能点）
│   └── F02 商品分类（二级菜单项）
│       ├── F0201 查看列表
│       └── F0202 新增分类
└── M02 订单管理
    └── F03 订单列表
        └── F0301 查看列表
        └── F0302 处理订单
```

**注意**：
- 三级功能点不体现在菜单中（通过页面内按钮实现）
- 菜单层级 = 一级模块 + 二级功能

---

### 步骤六：编写功能详细描述

**目标**：P0/P1 功能展开详细描述。

**方法**：
1. 读取 `@prd-writing-guide.md#功能详细描述模板`
2. 每个二级功能（P0/P1）填写：
   - 功能描述
   - 功能点清单
   - 前置条件
   - 权限矩阵
3. 复杂功能点（新增/编辑/审批）展开：
   - 触发位置
   - 交互流程
   - 字段约束
   - 业务规则
   - 验收标准

**验证**：
- P0/P1 功能必须完整展开
- 每个功能点有对应的页面/组件
- 业务规则引用 `@business-modeling.md#业务规则体系`

---

### 步骤七：验证完整性

**检查清单**：

| 检查项 | 要求 |
|--------|------|
| **业务活动覆盖** | 所有 N 个业务活动都有对应功能点 |
| 功能清单完整 | 二级功能 + 三级功能点两层都有 |
| **最小覆盖满足** | 每个二级功能的三级功能点数量 ≥ 最少要求 |
| 编号规范 | 功能ID和功能点ID符合编号规则 |
| 页面/组件映射 | 每个功能点有对应的页面/组件 |
| 权限覆盖 | 每个功能点有用户角色标注 |
| 优先级分布 | Must ≤ 60%，不能全是 P0 |
| **业务↔功能关联** | 每个功能点有对应业务活动，无遗漏、无多余 |
| **双向追溯** | 业务活动清单 ↔ 功能点清单 可双向查询 |
| **依赖关系声明** | 功能点间依赖关系已声明（前置依赖列填写完整） |

## 执行信号

以下行为表明流程执行失败：
- 业务活动→功能点映射表缺失
- 业务活动无对应功能点
- 二级功能的三级功能点数量 < 最少要求
- 缺少强制包含的功能点（如数据管理类缺少"新增"）
- 逐项处理清单未逐项输出
