# 功能架构设计流程

功能架构设计专项流程，从业务分析到三级功能点拆解。

---

## ⚠️ 执行约束（MANDATORY）

**必须遵守以下规则，违反即为流程执行失败**：

| 规则 | 说明 |
|------|------|
| **禁止跳步骤** | 必须按顺序执行，不得跳过任何步骤 |
| **必须输出三级功能点** | 功能清单必须包含二级功能 + 三级功能点两层 |
| **必须编号** | 功能ID和功能点ID必须符合编号规则 |
| **必须引用规范** | 功能详细描述必须引用 `@prd-writing-guide.md#功能架构规范` |

---

## 输入与输出

| 输入 | 来源 |
|------|------|
| 业务需求 | 需求分析阶段输出 |
| 业务角色 | `@business-modeling.md#业务角色模型` |
| 业务流程 | `@business-modeling.md#业务流程矩阵` |

| 输出 | 说明 |
|------|------|
| 二级功能清单 | 功能ID + 功能名称 + 优先级 + 模块 + 描述 |
| 三级功能点清单 | 功能点ID + 功能点名称 + 页面/组件 + 用户角色 + 对应业务活动 |
| 功能菜单层级结构 | 一级模块 → 二级功能 → 三级功能点 |
| 功能详细描述 | P0/P1 功能的完整展开 |
| 业务↔功能关联矩阵 | 业务活动清单 ↔ 功能点清单 双向追溯 |

---

## 执行步骤

### 步骤一：识别一级模块（业务领域）

**目标**：从业务需求中识别核心业务领域，划分一级模块。

**方法**：
1. 分析业务需求中的核心业务场景
2. 按业务领域划分模块（如：商品管理、订单管理、用户管理）
3. 每个模块对应一个一级分类

**输出**：
```
M01 商品管理
M02 订单管理
M03 用户管理
...
```

**验证**：
- 模块数量 ≤ 10 个（B端典型规模）
- 模块划分符合业务语义
- 模块之间边界清晰

---

### 步骤二：定义二级功能（业务功能单元）

**目标**：每个模块下定义二级功能，对应菜单项。

**方法**：
1. 每个模块识别核心业务功能
2. 二级功能 = 用户可执行的完整业务操作单元
3. 对应菜单项（用户可见的功能入口）

**输出**：

| 功能ID | 功能名称 | 优先级 | 模块 | 描述 | 用户角色 |
|--------|----------|--------|------|------|----------|
| F01 | 商品列表 | P0 | M01 商品管理 | 商品数据的查询、管理 | 管理员/运营 |
| F02 | 商品分类 | P0 | M01 商品管理 | 商品分类的维护 | 管理员 |
| F03 | 订单列表 | P0 | M02 订单管理 | 订单数据的查询、处理 | 管理员/客服 |

**编号规则**：
- 功能ID：F01-F99（全局唯一）
- 模块归属在"模块"列标注

---

### 步骤三：拆解三级功能点（CRUD操作）

**目标**：每个二级功能拆解为三级功能点。

**方法**：
1. 每个二级功能按 CRUD 操作拆解
2. 基本功能点：查看列表、搜索、新增、编辑、删除
3. 特殊功能点：审批、分配、导出、批量操作等（按业务需求）

**功能点类型 → 页面/组件映射**：

| 功能点类型 | 页面/组件 | 说明 |
|-----------|-----------|------|
| 查看列表 | P00X 列表页 | 数据展示 |
| 搜索筛选 | P00X 搜索区 | 列表页内组件 |
| 新增 | 表单弹窗 | 点击按钮触发 |
| 编辑 | 表单弹窗 | 与新增共用组件 |
| 删除 | 确认弹窗 | 二次确认 |
| 批量操作 | 确认弹窗 | 批量选择后触发 |
| 导出 | 导出按钮 | 列表页内操作 |
| 审批 | 审批弹窗 | 特殊业务操作 |
| 分配 | 分配弹窗 | 特殊业务操作 |
| 多步配置 | wizard 分步向导 | 复杂配置流程 |
| 任务管理 | kanban 看板 | 任务/工单流转 |
| 数据对比 | comparison 对比视图 | 版本/数据对比 |
| 帮助引导 | help 帮助页 | Onboarding/FAQ |

**输出**：

| 功能点ID | 功能点名称 | 优先级 | 所属功能 | 页面/组件 | 用户角色 | 对应业务活动 |
|----------|------------|--------|----------|-----------|----------|--------------|
| F0101 | 查看商品列表 | P0 | F01 商品列表 | P001 列表页 | 管理员/运营 | A01 商品查询 |
| F0102 | 搜索商品 | P0 | F01 商品列表 | P001 搜索区 | 管理员/运营 | A01 商品查询 |
| F0103 | 新增商品 | P0 | F01 商品列表 | P002 表单弹窗 | 管理员 | A02 商品上架 |
| F0104 | 编辑商品 | P0 | F01 商品列表 | P002 表单弹窗 | 管理员 | A03 商品修改 |
| F0105 | 删除商品 | P1 | F01 商品列表 | 确认弹窗 | 管理员 | A04 商品下架 |

**编号规则**：
- 功能点ID：F0101-F0199（二级功能编号 + 功能点序号）
- 对应业务活动：关联到业务活动清单中的活动ID，实现业务↔功能双向追溯

---

### 步骤四：构建功能菜单层级结构

**目标**：输出功能菜单的树形结构。

**格式**：
```
系统名称
├── M01 商品管理（一级模块）
│   ├── F01 商品列表（二级菜单项）
│   │   ├── F0101 查看列表（三级功能点）
│   │   ├── F0102 搜索（三级功能点）
│   │   ├── F0103 新增（三级功能点）
│   │   └── F0104 编辑（三级功能点）
│   └── F02 商品分类（二级菜单项）
│       ├── F0201 查看列表
│       └── F0202 新增分类
└── M02 订单管理
    └── F03 订单列表
        └── F0301 查看列表
        └── F0302 处理订单
```

**注意**：
- 三级功能点不体现在菜单中（通过页面内按钮实现）
- 菜单层级 = 一级模块 + 二级功能

---

### 步骤五：编写功能详细描述

**目标**：P0/P1 功能展开详细描述。

**方法**：
1. 读取 `@prd-writing-guide.md#功能详细描述模板`
2. 每个二级功能（P0/P1）填写：
   - 功能描述
   - 功能点清单
   - 前置条件
   - 权限矩阵
3. 复杂功能点（新增/编辑/审批）展开：
   - 触发位置
   - 交互流程
   - 字段约束
   - 业务规则
   - 验收标准

**验证**：
- P0 功能必须完整展开
- 每个功能点有对应的页面/组件
- 业务规则引用 `@business-modeling.md#业务规则体系`

---

### 步骤六：验证完整性

**检查清单**：

| 检查项 | 要求 |
|--------|------|
| 功能清单完整 | 二级功能 + 三级功能点两层都有 |
| 编号规范 | 功能ID和功能点ID符合编号规则 |
| 页面/组件映射 | 每个功能点有对应的页面/组件 |
| 权限覆盖 | 每个功能点有用户角色标注 |
| 优先级分布 | Must ≤ 60%，不能全是 P0 |
| **业务↔功能关联** | 每个功能点有对应业务活动，无遗漏、无多余 |
| **双向追溯** | 业务活动清单 ↔ 功能点清单 可双向查询 |
