# 需求评审流程

**何时使用**: 评审PRD文档、需求清单、产品方案时

---

## 评审维度

```
1. 完整性评审
   ├── 章节完整性 → 是否包含所有必需章节
   ├── 功能完整性 → 是否覆盖核心业务场景
   ├── 数据完整性 → 是否覆盖核心数据实体
   └── 检查清单 → @../review-checklist.md#完整性检查

2. 合理性评审
   ├── 业务逻辑合理性 → 流程是否闭环、状态是否完整
   ├── 优先级合理性 → 是否合理划分P0/P1/P2
   ├── 技术可行性 → 是否可实现、有无技术障碍
   └── 检查清单 → @../review-checklist.md#合理性检查

3. 清晰度评审
   ├── 需求清晰度 → 是否存在歧义描述
   ├── 可测试性 → 是否有明确验收标准
   └── 检查清单 → @../review-checklist.md#清晰度检查

4. 风险识别
   ├── 技术风险 → 算法方案、性能瓶颈
   ├── 业务风险 → 业务规则变更、用户接受度
   └── 检查清单 → @../review-checklist.md#风险检查

5. 输出
   └── 评审报告（问题清单、改进建议、风险清单）
```

---

## 常见问题识别

| 问题类型 | 表现 | 建议 |
|----------|------|------|
| 需求模糊 | "系统要快" | 添加具体数值："页面加载<2秒" |
| 验收缺失 | "支持搜索" | 添加Given-When-Then |
| 优先级膨胀 | 全是P0 | 使用MoSCoW重新分级 |
| 异常缺失 | 只有正常流程 | 添加异常场景表 |
| 预警规则缺失 | 算法未定义 | 标注TBD，明确输入输出 |
