# 需求分析流程

**何时使用**: 发现需求、定义需求、确定优先级时

---

## ⚠️ 硬约束（MANDATORY）

**必须遵守以下规则，违反即为流程执行失败**：

### 1. 输入完整性判断机制

**在开始追问前，先评估用户输入的完整性，决定追问策略**：

#### 1.1 输入覆盖率计算

**评估维度（9 项）**：

| 序号 |评估维度 |权重 |判断标准 |
|------|----------|------|----------|
| 1 | 核心业务场景描述 | 15% | 是否明确描述了核心业务场景？ |
| 2 | 用户角色定义 | 12% | 是否列出了主要用户角色？ |
| 3 | 数据实体说明 | 12% | 是否说明了主要数据实体？ |
| 4 | 业务流程描述 | 15% | 是否描述了主要业务流程？ |
| 5 | 异常场景覆盖 | 10% | 是否提到了异常处理？ |
| 6 | 外部系统对接 | 8% | 是否说明外部系统需求？ |
| 7 | 数据来源说明 | 8% | 是否说明数据来源？ |
| 8 | 权限需求说明 | 10% | 是否说明权限需求？ |
| 9 | 端划分确认 | 10% | 是否明确哪些端需要开发？ |

**覆盖率计算公式**：
```
覆盖率 = Σ(维度得分 × 权重)
- 维度得分：有明确说明 = 100%，部分说明 = 50%，未说明 = 0%
```

#### 1.2 追问策略判断

| 覆盖率 | 追问策略 | 说明 |
|--------|----------|------|
| ≥ 80% | **跳过追问** | 输入已足够完整，直接进入需求结构化 |
| 50-80% | **精简追问** | 只追问未明确维度（得分<100%的维度） |
| < 50% | **完整追问** | 按完整追问清单逐项确认 |

**判断流程**：
```
1. 读取用户输入
2. 逐维度评估得分（有/部分/无）
3. 计算覆盖率
4. 判断追问策略：
   ├── ≥ 80% → 输出"输入完整，跳过追问" → 进入步骤三
   ├── 50-80% → 输出"输入部分完整，精简追问以下维度：[列表]" → 只追问未明确维度
   └── < 50% → 输出"输入不完整，完整追问" → 按追问清单逐项确认
```

---

### 2. 需求点追问清单（9 项）

**当覆盖率 < 80% 时，必须追问未明确维度**：

| 序号 |追问项 |追问内容 |权重 |确认状态 |
|------|--------|----------|------|----------|
| 1 | 核心业务场景 | 系统主要解决什么业务问题？有哪些核心业务场景？ | 15% | ☐ 必问 |
| 2 | 用户角色 | 系统有哪些用户角色？每个角色做什么？ | 12% | ☐ 必问 |
| 3 | 数据实体 | 系统管理哪些数据实体？每个实体有哪些属性？ | 12% | ☐ 必问 |
| 4 | 业务流程 | 每个业务场景的完整流程是什么？从开始到结束？ | 15% | ☐ 必问 |
| 5 | 异常场景 | 流程中可能出现的异常情况？如何处理？ | 10% | ☐ 必问 |
| 6 | 外部系统 | 是否需要对接外部系统？对接方式是什么？ | 8% | ☐ 必问 |
| 7 | 数据来源 | 数据从哪来？初始数据如何导入？ | 8% | ☐ 必问 |
| 8 | 权限需求 | 不同角色能看到什么数据？能做什么操作？ | 10% | ☐ 必问 |
| 9 | 端划分确认 | 系统需要哪些端？如小程序端、管理后台、H5端等 | 10% | ☐ 必问 |

**追问规则**：
- 覆盖率 ≥ 80% → 跳过追问，直接进入需求结构化
- 覆盖率 50-80% → 只追问未明确维度（得分<100%的维度）
- 覆盖率 < 50% → 按完整追问清单逐项确认
- 用户明确说"不需要" → 标记 ✗ 并记录原因
- 用户说"不确定" → 标记 ? 并继续追问直到明确

---

### 3. 端划分确认规则

**端划分决定 PRD 文档结构**：

| 端数量 | PRD 结构 | 说明 |
|--------|----------|------|
| ≥ 2 个端 | **三层结构** | 总览 PRD + 各端独立 PRD（如小程序端 PRD、管理端 PRD） |
| = 1 个端 + 模块 ≥ 3 | **两层结构** | 单一 PRD，按模块分章节 |
| = 1 个端 + 模块 ≤ 2 | **单层结构** | 精简 PRD，直接描述功能 |

**端划分判断流程**：
```
1. 用户输入中是否有明确的端划分？
   ├── 有 → 使用用户指定的端划分
   └── 无 → 默认为"管理后台"单端
2. 输出端划分确认：
   ├── 端列表：[小程序端、管理后台、H5端...]
   ├── 端数量：N 个
   └── PRD 结构建议：[三层/两层/单层]
```

### 5. 数量传递约束

**需求点清单 → 后续流程（业务建模）的输入约束**：

```
需求分析输出：N 个需求点
    ↓ 强制传递
业务建模输入：必须处理这 N 个需求点（不得遗漏）

传递表（必须在需求清单末尾输出）：
├── R01 商品管理 → 待转化为业务活动
├── R02 订单管理 → 待转化为业务活动
├── R03 库存管理 → 待转化为业务活动
├── ... → ...
└── R19 权限管理 → 待转化为业务活动

传递约束：
- 后续流程必须处理所有 N 个需求点
- 任何需求点无对应业务活动 → 需求遗漏
```

## 流程步骤

### 步骤一：输入完整性判断

**目标**：评估用户输入完整性，决定追问策略。

**方法**：
1. 读取用户输入
2. 按 9 个维度评估得分（有明确说明/部分说明/未说明）
3. 计算覆盖率 = Σ(维度得分 × 权重)
4. 判断追问策略：
   - ≥ 80% → 跳过追问，直接进入步骤三
   - 50-80% → 精简追问，只追问未明确维度
   - < 50% → 完整追问，按追问清单逐项确认

**输出**：
- 输入覆盖率评估表（9 个维度得分）
- 覆盖率数值
- 追问策略决定（跳过/精简/完整）

---

### 步骤二：追问确认需求边界

**目标**：通过追问清单，确认需求完整性（仅当覆盖率 < 80% 时执行）。

**方法**：
1. 根据追问策略（精简/完整）决定追问范围
2. 只追问未明确维度（得分<100%的维度）
3. 记录用户回答和确认状态
4. 确认端划分（决定 PRD 结构）

**输出**：
- 追问确认表（追问结果）
- 端划分确认（端列表 + PRD 结构建议）
- 确认状态：所有项必须 ✓ 或 ✗（明确不需要）

---

### 步骤三：需求挖掘

**方法**：
```
用户访谈 → @../requirement-analysis.md#用户访谈
问卷调查 → @../requirement-analysis.md#问卷调查
数据分析 → @../requirement-analysis.md#数据分析
```

---

### 步骤四：需求分类（KANO模型）

```
基本需求（必须有，无则不满）
期望需求（越好越满意）
兴奋需求（意想不到的惊喜）

详细方法 → @../requirement-analysis.md#KANO模型
```

---

### 步骤五：需求优先级（MoSCoW）

```
Must have（必须有，MVP核心）
Should have（应该有，重要非紧急）
Could have（可以有，资源允许时）
Won't have（不会有，未来考虑）

详细方法 → @../requirement-analysis.md#MoSCoW优先级法
```

---

### 步骤六：需求结构化

**目标**：将需求转化为结构化的需求点清单。

**输出要求**：
1. 每个需求点有唯一 ID（R01-R99）
2. 每个需求点标注来源追问项
3. 输出需求点总数 N

---

### 步骤七：输出需求点清单

**强制输出**：
- 输入覆盖率评估表（9 个维度得分）
- 追问策略决定（跳过/精简/完整）
- 追问确认表（追问结果，如覆盖率 < 80%）
- 端划分确认（端列表 + PRD 结构建议）
- 需求点清单（含 ID、来源、优先级）
- 需求点总数 N
- 数量传递表（传递给后续流程）

---

## 执行信号

以下行为表明流程执行失败：
- 追问清单有未回答项（☐ 状态）
- 需求点清单无总数统计
- 需求点无唯一 ID 编号
- 未输出数量传递表
