# 需求分析流程

**何时使用**: 发现需求、定义需求、确定优先级时

---

## ⚠️ 硬约束（MANDATORY）

**必须遵守以下规则，违反即为流程执行失败**：

### 1. 需求点追问清单（8 项必问）

**在开始需求分析前，必须逐项追问用户，确认需求边界**：

| 序号 |追问项 |追问内容 |确认状态 |
|------|--------|----------|----------|
| 1 | 核心业务场景 | 系统主要解决什么业务问题？有哪些核心业务场景？ | ☐ 必问 |
| 2 | 用户角色 | 系统有哪些用户角色？每个角色做什么？ | ☐ 必问 |
| 3 | 数据实体 | 系统管理哪些数据实体？每个实体有哪些属性？ | ☐ 必问 |
| 4 | 业务流程 | 每个业务场景的完整流程是什么？从开始到结束？ | ☐ 必问 |
| 5 | 异常场景 | 流程中可能出现的异常情况？如何处理？ | ☐ 必问 |
| 6 | 外部系统 | 是否需要对接外部系统？对接方式是什么？ | ☐ 必问 |
| 7 | 数据来源 | 数据从哪来？初始数据如何导入？ | ☐ 必问 |
| 8 | 权限需求 | 不同角色能看到什么数据？能做什么操作？ | ☐ 必问 |

**追问规则**：
- 用户明确说"不需要" → 标记 ✗ 并记录原因
- 用户说"不确定" → 标记 ? 并继续追问直到明确
- 用户未回答 → 标记 ☐ 并暂停流程，必须追问

---

### 2. 需求点输出约束

**需求分析完成后，必须输出结构化的需求点清单**：

```markdown
# 求点清单

## 追问确认表

|追问项 |用户回答 |确认状态 |纳入需求 |
|--------|----------|----------|----------|
| 核心业务场景 | 商品管理、订单管理、库存管理 | ✓ 已确认 | R01-R03 |
| 用户角色 | 管理员、运营、客服 | ✓ 已确认 | R04 |
| 数据实体 | 商品、订单、库存、用户 | ✓ 已确认 | R05-R08 |
| 业务流程 | 商品上架→销售→发货→售后 | ✓ 已确认 | R09-R12 |
| 异常场景 | 库存不足、支付失败、退货 | ✓ 已确认 | R13-R15 |
| 外部系统 | 支付系统、物流系统 | ✓ 已确认 | R16-R17 |
| 数据来源 | 手动录入、批量导入 | ✓ 已确认 | R18 |
| 权限需求 | 角色权限矩阵 | ✓ 已确认 | R19 |

## 需求点清单

| ID | 需求名称 | 来源追问项 | 类型 | 优先级 | 用户价值 | 业务价值 |
|----|----------|------------|------|--------|----------|----------|
| R01 | 商品管理 | 核心业务场景 | 功能 | M | 商品信息维护 | 商品运营基础 |
| R02 | 订单管理 | 核心业务场景 | 功能 | M | 订单处理 | 销售闭环 |
| ... | ... | ... | ... | ... | ... | ... |

## 需求点统计
- 总需求点数量：N 个
- Must have：X 个 (X%)
- Should have：Y 个 (Y%)
- Could have：Z 个 (Z%)
```

---

### 3. 数量传递约束

**需求点清单 → 后续流程（业务建模）的输入约束**：

```
需求分析输出：N 个需求点
    ↓ 强制传递
业务建模输入：必须处理这 N 个需求点（不得遗漏）

传递表（必须在需求清单末尾输出）：
├── R01 商品管理 → 待转化为业务活动
├── R02 订单管理 → 待转化为业务活动
├── R03 库存管理 → 待转化为业务活动
├── ... → ...
└── R19 权限管理 → 待转化为业务活动

传递约束：
- 后续流程必须处理所有 N 个需求点
- 任何需求点无对应业务活动 → 需求遗漏
```

## 流程步骤

### 步骤一：追问确认需求边界

**目标**：通过追问清单，确认需求完整性。

**方法**：
1. 按追问清单逐项询问用户
2. 记录用户回答和确认状态
3. 用户未回答的项，暂停并追问

**输出**：
- 追问确认表（8 项追问结果）
- 确认状态：所有项必须 ✓ 或 ✗（明确不需要）

---

### 步骤二：需求挖掘

**方法**：
```
用户访谈 → @../requirement-analysis.md#用户访谈
问卷调查 → @../requirement-analysis.md#问卷调查
数据分析 → @../requirement-analysis.md#数据分析
```

---

### 步骤三：需求分类（KANO模型）

```
基本需求（必须有，无则不满）
期望需求（越好越满意）
兴奋需求（意想不到的惊喜）

详细方法 → @../requirement-analysis.md#KANO模型
```

---

### 步骤四：需求优先级（MoSCoW）

```
Must have（必须有，MVP核心）
Should have（应该有，重要非紧急）
Could have（可以有，资源允许时）
Won't have（不会有，未来考虑）

详细方法 → @../requirement-analysis.md#MoSCoW优先级法
```

---

### 步骤五：需求结构化

**目标**：将需求转化为结构化的需求点清单。

**输出要求**：
1. 每个需求点有唯一 ID（R01-R99）
2. 每个需求点标注来源追问项
3. 输出需求点总数 N

---

### 步骤六：输出需求点清单

**强制输出**：
- 追问确认表（8 项）
- 需求点清单（含 ID、来源、优先级）
- 需求点总数 N
- 数量传递表（传递给后续流程）

---

## 执行信号

以下行为表明流程执行失败：
- 追问清单有未回答项（☐ 状态）
- 需求点清单无总数统计
- 需求点无唯一 ID 编号
- 未输出数量传递表
