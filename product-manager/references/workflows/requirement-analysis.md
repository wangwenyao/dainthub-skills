# 需求分析流程

**何时使用**: 发现需求、定义需求、确定优先级时

---

## 流程步骤

```
1. 需求挖掘
   ├── 用户访谈 → @../requirement-analysis.md#用户访谈
   ├── 问卷调查 → @../requirement-analysis.md#问卷调查
└── 数据分析 → @../requirement-analysis.md#数据分析

2. 需求分类（KANO模型）
   ├── 基本需求（必须有，无则不满）
   ├── 期望需求（越好越满意）
   ├── 兴奋需求（意想不到的惊喜）
   └── 详细方法 → @../requirement-analysis.md#KANO模型

3. 需求优先级（MoSCoW）
   ├── Must have（必须有，MVP核心）
   ├── Should have（应该有，重要非紧急）
   ├── Could have（可以有，资源允许时）
   ├── Won't have（不会有，未来考虑）
   └── 详细方法 → @../requirement-analysis.md#MoSCoW优先级法

4. 需求结构化
└── 用户故事地图 → @../requirement-analysis.md#用户故事地图

5. 输出
   └── 需求清单（含优先级、用户价值、业务价值）
```

## 常见问题

### 所有需求都是P0怎么办？

```
解决方案：
1. 使用MoSCoW重新分类
2. 应用MVP原则：去掉该功能产品是否还能工作？
3. 按业务价值排序：营收影响、用户影响、竞争优势
4. 按技术依赖排序：哪些功能是其他功能的前置条件
```

### 需求描述模糊怎么办？

```
解决方案：
1. 添加具体数值（如"≤2秒"而非"快速"）
2. 添加使用场景（如"在购物车页面点击结算"）
3. 使用Given-When-Then格式明确验收标准
```
