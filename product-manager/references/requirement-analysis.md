# 需求分析方法论

需求分析核心方法论，从需求挖掘到优先级排序。

---

## 一、需求挖掘技术

### 1.1 用户访谈

| 类型 | 目的 | 适用阶段 |
|------|------|----------|
| 探索性访谈 | 了解用户行为和动机 | 需求发现 |
| 验证性访谈 | 验证假设和方案 | 需求验证 |

**访谈技巧**：

```
DO：
✓ 使用开放式问题
✓ 关注过去行为而非未来意愿
✓ 使用"最近一次"技巧获取具体案例
✓ 追问"为什么"（至少5次）

DON'T：
✗ 问"你会用这个功能吗？"
✗ 引导性问题
✗ 询问未来行为
```

### 1.2 问卷调查

**问卷结构**：
```
开场白 → 篮选问题 → 核心问题 → 人口统计学 → 感谢语
```

**KANO问卷设计**：
每个功能设计正向+反向问题，使用5级量表：
- 我喜欢这样 / 我期望这样 / 我无所谓 / 我可以忍受 / 我不喜欢这样

### 1.3 数据分析

| 来源 | 数据类型 | 分析价值 |
|------|----------|----------|
| 系统日志 | 操作行为数据 | 功能使用率、操作瓶颈 |
| 客服反馈 | 问题工单 | 痛点聚类、优先级 |
| 业务报表 | 业务指标数据 | 流程效率、异常率 |

## 二、需求分类方法

### 2.1 KANO模型

**五类需求**：

| 类型 | 定义 | 优先级 | 示例 |
|------|------|--------|------|
| **基本需求(M)** | 必须具备 | 最高 | 登录功能、支付功能 |
| **期望需求(O)** | 越好越满意 | 高 | 搜索速度、加载速度 |
| **兴奋需求(Q)** | 意想不到 | 中 | 个性化推荐 |
| **无差异需求(I)** | 客户不在乎 | 低 | 皮肤更换 |
| **反向需求(R)** | 客户不喜欢 | 避免 | 过度推送 |

**核心规则**：
- 基本需求必须满足，无则不满
- 期望需求线性影响满意度
- 兴奋需求无则无感，有则惊喜

### 2.2 MoSCoW优先级法

| 级别 | 含义 | 占比 | 特征 |
|------|------|------|------|
| **M** | Must have | ≤60% | MVP核心，无则产品失败 |
| **S** | Should have | ~20% | 重要但非紧急，有替代方案 |
| **C** | Could have | ~20% | 期望功能，资源允许时实现 |
| **W** | Won't have | 记录 | 本次不做 |

**判断规则**：
```
Must have：
1. 没有这个功能，产品还能正常工作吗？→ 不能 = M
2. 这是核心业务流程必需的吗？→ 是 = M
3. 用户会因为这个功能缺失而投诉吗？→ 会 = M
```

## 三、需求结构化方法

### 3.1 用户故事地图

**结构**：
```
骨架层 → 活动层 → 任务层 → 发布切片
```

**构建步骤**：
1. 定义用户画像和目标
2. 构建叙事骨架（主要步骤）
3. 分解用户活动（详细步骤）
4. 添加任务细节（用户故事）
5. 切割发布版本（MVP定义）

### 3.2 业务流程走查

**核心概念**：在需求阶段模拟完整业务操作路径，验证需求闭环。

**走查步骤**：
1. 选取一个核心业务场景
2. 从触发者开始，逐步走完整个流程
3. 检查每个步骤是否有对应的功能支撑
4. 检查异常路径是否有处理方案
5. 输出走查问题清单

## 四、需求验证方法

### 4.1 原型走查验证

| 验证方式 | 适用场景 | 参与角色 |
|---------|---------|---------|
| 线框图走查 | 页面结构确认 | 产品经理 + 业务方 |
| 流程模拟 | 业务流程闭环验证 | 产品经理 + 业务方 + 技术 |
| 功能清单评审 | 功能覆盖完整性 | 产品经理 + 技术经理 |

**验证标准**：
- 核心业务路径无断点
- 异常场景有处理方案
- 每个业务活动有对应功能点支撑
- 权限矩阵覆盖所有操作
