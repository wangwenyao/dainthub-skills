## 一、需求挖掘技术

### 1.1 用户访谈

**访谈类型**:

| 类型 | 目的 | 适用阶段 |
|------|------|----------|
| **探索性访谈** | 了解用户行为和动机 | 需求发现阶段 |
| **验证性访谈** | 验证假设和方案 | 需求验证阶段 |
| **JTBD访谈** | 深挖真实需求动机 | 产品创新阶段 |

**访谈技巧**:

```
DO（应该做）：
✓ 使用开放式问题（"能描述一下您最近一次...的经历吗？"）
✓ 关注过去行为而非未来意愿
✓ 使用"最近一次"技巧获取具体案例
✓ 记录情绪和痛点
✓ 追问"为什么"（至少5次）
✓ 在用户环境中进行访谈

DON'T（不应该做）：
✗ 问"你会用这个功能吗？"（用户会为了取悦你而说"会"）
✗ 引导性问题（"这个功能对您有帮助吗？"）
✗ 询问未来行为（"你未来会..."）
✗ 使用专业术语
✗ 打断用户叙述
```

**访谈问题模板**:

```markdown
## 开场问题（5分钟）
1. 请简单介绍一下您的工作职责？
2. 您日常工作中最常做的事情是什么？

## 行为探索（15分钟）
1. 您最近一次遇到[问题场景]是什么时候？
2. 当时是什么情况？能详细描述一下吗？
3. 您当时是怎么处理的？
4. 结果如何？满意吗？

## 痛点深挖（10分钟）
1. 在这个过程中，最让您困扰的是什么？
2. 您尝试过哪些解决方案？效果如何？
3. 如果有一个理想方案，您希望它是什么样的？

## 结束问题（5分钟）
1. 还有其他想分享的吗？
2. 如果后续有相关问题，可以再联系您吗？
```

### 1.2 问卷调查

**问卷设计原则**:

```
1. 问题设计
   ├── 简洁明确，避免歧义
   ├── 避免引导性问题
   ├── 封闭式+开放式问题结合
   └── 逻辑顺序合理

2. 问卷结构
   ├── 开场白（说明目的、预计时长）
   ├── 筛选问题（确定目标用户）
   ├── 核心问题（需求、痛点、偏好）
   ├── 人口统计学问题（年龄、职业等）
   └── 感谢语

3. 问卷测试
   ├── 预测试（5-10人）
   ├── 检查理解偏差
   └── 优化问题表述
```

**KANO问卷设计**:

```
每个功能设计两个问题：
1. 正向问题：如果产品有这个功能，您感觉如何？
   - 我喜欢这样
   - 我期望这样
   - 我无所谓
   - 我可以忍受
   - 我不喜欢这样

2. 反向问题：如果产品没有这个功能，您感觉如何？
   - 我喜欢这样
   - 我期望这样
   - 我无所谓
   - 我可以忍受
   - 我不喜欢这样

分类规则表：
| 正向问题 | 反向问题 | 需求类型 |
|----------|----------|----------|
| 喜欢 | 喜欢 | 兴奋需求(Q) |
| 喜欢 | 期望 | 兴奋需求(Q) |
| 喜欢 | 无所谓 | 兴奋需求(Q) |
| 喜欢 | 可以忍受 | 兴奋需求(Q) |
| 期望 | 不喜欢 | 期望需求(O) |
| 无所谓 | 不喜欢 | 基本需求(M) |
| 可以忍受 | 不喜欢 | 基本需求(M) |
```

### 1.3 数据分析

**数据来源**:

| 来源 | 数据类型 | 分析价值 |
|------|----------|----------|
| **产品埋点** | 用户行为数据 | 功能使用率、转化漏斗 |
| **用户日志** | 操作记录 | 用户路径、异常行为 |
| **客服反馈** | 问题工单 | 痛点聚类、优先级 |
| **社交媒体** | 舆情数据 | 用户情绪、口碑 |
| **竞品数据** | 市场数据 | 市场份额、定价策略 |

**分析方法**:

```
1. 漏斗分析
   - 定义关键步骤
   - 计算转化率
   - 识别流失点
   - 优化关键环节

2. 用户分群
   - 按行为分群（活跃/沉默/流失）
   - 按属性分群（新用户/老用户）
   - 对比不同群体需求差异

3. A/B测试
   - 定义假设
   - 设计实验
   - 收集数据
   - 分析结果
   - 做出决策

4. 趋势分析
   - 时间序列分析
   - 异常检测
   - 预测建模
```

### 1.4 竞品分析

**分析框架**:

```
1. 竞品选择
   ├── 直接竞品（解决相同问题）
   ├── 间接竞品（替代方案）
   └── 潜在竞品（跨界进入者）

2. 分析维度
   ├── 产品定位（目标用户、核心价值）
   ├── 功能对比（功能矩阵、差异化）
   ├── 用户体验（交互流程、视觉风格）
   ├── 商业模式（收费方式、盈利路径）
   └── 市场表现（用户量、口碑、增长）

3. 输出格式
   ┌────────────────────────────────────────┐
   │  竞品对比矩阵                          │
   ├──────────┬──────────┬──────────┬───────┤
   │  功能    │  竞品A   │  竞品B   │ 我们  │
   ├──────────┼──────────┼──────────┼───────┤
   │  功能1   │    ✓     │    ✓     │   ✗   │
   │  功能2   │    ✓     │    ✗     │   ✓   │
   │  功能3   │    ✗     │    ✓     │   ✓   │
   └──────────┴──────────┴──────────┴───────┘

4. 洞察提炼
   - 竞品优势学习点
   - 竞品劣势突破点
   - 差异化竞争策略
```

## 二、需求分类方法

### 2.1 KANO模型

**理论基础**: Noriaki Kano (1984) - 东京理工大学教授

**五类需求**:

```
满意度
  ↑
  │         ╱ 兴奋需求(Q)
  │        ╱
  │       ╱
  │──────╱──────── 期望需求(O)
  │     ╱
  │────╱─────────── 基本需求(M)
  │   ╱
  │──╱───────────── 无差异需求(I)
  │ ╱
  ╱──────────────── 反向需求(R)
  └──────────────────→ 实现程度
```

| 类型 | 定义 | 客户满意度影响 | 优先级 | 示例 |
|------|------|----------------|--------|------|
| **基本需求(M)** | 必须具备 | 无则不满，有则无感 | 最高 | 登录功能、支付功能 |
| **期望需求(O)** | 越好越满意 | 线性相关 | 高 | 搜索速度、加载速度 |
| **兴奋需求(Q)** | 意想不到 | 无则无感，有则惊喜 | 中 | 个性化推荐、智能提醒 |
| **无差异需求(I)** | 客户不在乎 | 无影响 | 低 | 皮肤更换、字体选择 |
| **反向需求(R)** | 客户不喜欢 | 越多越不满 | 避免 | 过度推送、强制评分 |

**实施步骤**:

```
1. 设计问卷
   - 每个功能设计正向+反向问题
   - 使用5级量表

2. 收集数据
   - 样本量建议≥100
   - 覆盖目标用户群体

3. 分类统计
   - 使用KANO分类表
   - 计算Better-Worse系数

4. 绘制KANO图
   - 横轴：实现程度
   - 纵轴：客户满意度

5. 确定优先级
   - 必须满足基本需求
   - 优化期望需求
   - 选择性添加兴奋需求
```

### 2.2 MoSCoW优先级法

**四级优先级**:

| 级别 | 英文 | 含义 | 占比 | 特征 |
|------|------|------|------|------|
| **M** | Must have | 必须有 | ≤60% | MVP核心，无则产品失败 |
| **S** | Should have | 应该有 | ~20% | 重要但非紧急，有替代方案 |
| **C** | Could have | 可以有 | ~20% | 期望功能，资源允许时实现 |
| **W** | Won't have | 不会有 | 记录 | 本次不做，移至待办列表 |

**优先级判断规则**:

```
判断Must have：
1. 没有这个功能，产品还能正常工作吗？→ 不能 = M
2. 这是核心业务流程必需的吗？→ 是 = M
3. 用户会因为这个功能缺失而投诉吗？→ 会 = M

判断Should have：
1. 很重要但有替代方案？→ 是 = S
2. 可以推迟到下个版本？→ 是 = S

判断Could have：
1. 用户期望但非必需？→ 是 = C
2. 资源充足时可以做？→ 是 = C

判断Won't have：
1. 本版本做不完？→ 是 = W
2. 需求不明确需要更多调研？→ 是 = W
```

### 2.3 RICE评分法

**计算公式**:
```
RICE Score = (Reach × Impact × Confidence) / Effort
```

**参数定义**:

| 参数 | 定义 | 单位/量级 |
|------|------|-----------|
| **Reach** | 影响用户数 | 用户数/季度 |
| **Impact** | 影响程度 | 3=巨大, 2=高, 1=中, 0.5=低, 0.25=微小 |
| **Confidence** | 确信程度 | 100%=高, 80%=中, 50%=低 |
| **Effort** | 工作量 | 人月 |

**示例计算**:

```
功能A：
- Reach: 10,000用户/季度
- Impact: 2 (高影响)
- Confidence: 80% (中等确信)
- Effort: 2人月
- RICE = (10000 × 2 × 0.8) / 2 = 8000

功能B：
- Reach: 5,000用户/季度
- Impact: 3 (巨大影响)
- Confidence: 50% (低确信)
- Effort: 3人月
- RICE = (5000 × 3 × 0.5) / 3 = 2500

结论：功能A优先级更高
```

## 三、需求结构化方法

### 3.1 用户故事地图

**理论基础**: Jeff Patton (2014)

**地图结构**:

```
┌─────────────────────────────────────────────────────────────┐
│  骨架层 (Backbone) - 用户目标序列                           │
│  ┌─────────┐   ┌─────────┐   ┌─────────┐   ┌─────────┐   │
│  │  注册   │ → │  浏览   │ → │  购买   │ → │  支付   │   │
│  └─────────┘   └─────────┘   └─────────┘   └─────────┘   │
├─────────────────────────────────────────────────────────────┤
│  活动层 (Activities) - 具体步骤                             │
│  ┌─────────┐   ┌─────────┐   ┌─────────┐   ┌─────────┐   │
│  │填写信息 │   │搜索商品 │   │加入购物车│   │选择支付 │   │
│  │验证邮箱 │   │查看详情 │   │选择规格 │   │确认订单 │   │
│  │设置密码 │   │对比商品 │   │修改数量 │   │完成支付 │   │
│  └─────────┘   └─────────┘   └─────────┘   └─────────┘   │
├─────────────────────────────────────────────────────────────┤
│  任务层 (Tasks) - 用户故事                                  │
│  ┌─────────┐                                                │
│  │故事1(P0)│ ← 按优先级垂直排列                             │
│  │故事2(P0)│                                                │
│  │故事3(P1)│                                                │
│  │故事4(P1)│                                                │
│  │故事5(P2)│                                                │
│  └─────────┘                                                │
├─────────────────────────────────────────────────────────────┤
│  发布切片 (Releases) - MVP定义                              │
│  ├─ MVP ──┤───── V1.1 ─────┤────── V1.2 ──────┤           │
│  核心故事    增强功能         优化功能                        │
└─────────────────────────────────────────────────────────────┘
```

**构建步骤**:

```
1. 定义用户画像
   - 目标用户是谁？
   - 用户目标是什么？

2. 构建叙事骨架
   - 用户完成目标的主要步骤
   - 按时间顺序排列
   - 形成"骨架层"

3. 分解用户活动
   - 每个骨架步骤下的详细活动
   - 形成"活动层"

4. 添加任务细节
   - 每个活动下的具体用户故事
   - 按优先级垂直排列
   - 形成"任务层"

5. 切割发布版本
   - 横向切割定义MVP
   - 后续版本逐步添加
   - 形成"发布切片"
```

### 3.2 JTBD框架

**理论基础**: Clayton Christensen (哈佛商学院)

**核心概念**: 用户不是购买产品，而是"雇佣"产品完成任务

**Job Statement格式**:

```
当 [情境/Situation] 时
我想要 [动机/Motivation]
以便于 [期望成果/Expected Outcome]
```

**示例**:

```markdown
Job 1:
当 早上匆忙出门上班时
我想要 快速获取咖啡因提神
以便于 保持清醒完成工作

→ 产品机会：即饮咖啡、咖啡胶囊、能量饮料

Job 2:
当 周末在家休息时
我想要 慢慢品味咖啡的香气和味道
以便于 享受悠闲时光

→ 产品机会：精品咖啡豆、手冲器具、咖啡品鉴课程
```

**JTBD访谈技巧**:

```
1. 关注"第一次"
   - "您第一次遇到这个问题是什么时候？"
   - "当时是什么促使您开始寻找解决方案？"

2. 探索"最后时间"
   - "您最近一次解决这个问题是什么时候？"
   - "当时您是怎么做的？结果如何？"

3. 追问"为什么"
   - "为什么这对您很重要？"
   - "这背后真正的目的是什么？"

4. 发现"挣扎时刻"
   - "在这个过程中最让您困扰的是什么？"
   - "您尝试过哪些解决方案？为什么放弃了？"
```

## 四、需求验证方法

### 4.1 MVP验证

**MVP类型选择**:

| 类型 | 特点 | 适用场景 | 验证周期 |
|------|------|----------|----------|
| **烟雾测试MVP** | 假门测试、着陆页 | 验证需求存在 | 1-2周 |
| **礼宾MVP** | 人工提供服务 | 验证解决方案 | 2-4周 |
| **原型MVP** | 低保真原型 | 验证交互流程 | 1-2周 |
| **功能MVP** | 核心功能版本 | 验证产品价值 | 4-8周 |
| **Wizard of Oz** | 前端真实后端人工 | 验证用户体验 | 2-3周 |

**MVP验证指标**:

```
核心指标：
1. 激活率 → 用户是否愿意开始使用
2. 留存率 → 用户是否愿意继续使用
3. 使用频率 → 用户是否频繁使用
4. NPS评分 → 用户是否愿意推荐

辅助指标：
- 注册转化率
- 功能使用率
- 用户反馈数量
- 客服工单数量
```

### 4.2 A/B测试

**测试流程**:

```
1. 定义假设
   问题：[当前问题]
   假设：[改进方案]
   预期：[量化结果]

2. 设计实验
   - 对照组(A)：当前方案
   - 实验组(B)：新方案
   - 样本量计算
   - 分流比例

3. 实施测试
   - 随机分流
   - 数据收集
   - 持续监控

4. 分析结果
   - 统计显著性检验
   - 效果量化
   - 决策建议

5. 做出决策
   - 全面推广
   - 继续优化
   - 放弃方案
```

**样本量计算**:

```
简单估算公式：
n = 16 × σ² / Δ²

其中：
- n = 每组样本量
- σ = 标准差
- Δ = 期望提升幅度

示例：
- 当前转化率：5%
- 期望提升到：6%
- σ ≈ √(0.05 × 0.95) ≈ 0.22
- Δ = 0.01
- n = 16 × 0.22² / 0.01² ≈ 77,440（每组）
```
