## 一、Design Thinking 设计思维

**理论基础**: IDEO, Stanford d.school

```

│                    Design Thinking 流程                      │

│                                                             │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐             │
│  │ 同理心   │ → │  定义    │ → │  构思    │             │

│  └──────────┘    └──────────┘    └──────────┘             │
│                                        │                    │
│                                        ↓                    │

│  │  测试   │ ← │  原型   │                               │
│  │  Test   │    │Prototype │                               │
│  │ 反馈收集 │    │ 快速验证 │                               │
│  └──────────┘    └──────────┘                               │

└─────────────────────────────────────────────────────────────┘
```

### 1.2 各阶段详解

#### 阶段一：同理心 (Empathize)

**目标**: 深入理解用户

**核心活动**:

| 活动 | 方法 | 输出 |
|------|------|------|
| 用户访谈 | 深度访谈、情境访谈 | 用户洞察 |
| 用户观察 | 伴随观察、影子跟踪 | 行为记录 |
| 同理心地图 | 感受/想法/行为/痛点 | 情感理解 |
| 用户旅程图 | 触点/情绪/痛点 | 体验全景 |

**同理心地图模板**:

```
┌───────────────────┬───────────────────┐
│      想/感受      │       听          │
│  (Thoughts &     │   (Hears)         │
│   Feelings)      │                   │
│                  │                   │
│                  │                   │
├───────────────────┤                   │
│       看          │                   │
│    (Sees)        │                   │
│                  │                   │
│                  ├───────────────────┤
│                  │   说/做           │
│                  │  (Says & Does)    │
│                  │                   │
├───────────────────┼───────────────────┤
│       痛点        │      收益         │
│     (Pains)      │     (Gains)       │
│                  │                   │
└───────────────────┴───────────────────┘
```

#### 阶段二：定义 (Define)

**目标**: 聚焦核心问题

**POV陈述格式**:

```
[用户] 需要 [需求]，
因为 [洞察]。

示例：
忙碌的职场人士 需要快速获取咖啡因提神，
因为 早晨时间紧张且需要保持清醒完成工作。
```

**HMW问题格式**:

```
How Might We... (我们可以如何...)

转换规则：
- 将POV中的"因为"部分转化为HMW问题
- 保持开放性，不预设解决方案
- 聚焦用户价值

示例：
HMW 帮助职场人士在短时间内获得提神效果？
HMW 让获取咖啡因的过程更加便捷？
HMW 提供替代咖啡因的提神方案？
```

#### 阶段三：构思 (Ideate)

**目标**: 生成多样化解决方案

**构思方法**:

| 方法 | 适用场景 | 参与人数 |
|------|----------|----------|
| **头脑风暴** | 快速生成想法 | 5-8人 |
| **反向头脑风暴** | 打破思维定式 | 5-8人 |
| **SCAMPER** | 改进现有方案 | 3-5人 |
| **思维导图** | 结构化发散 | 1-3人 |
| **最坏主意** | 消除恐惧心理 | 5-8人 |

**头脑风暴规则**:

```
1. 延迟判断 - 不评价、不批评
2. 追求数量 - 先多后精
3. 鼓励疯狂 - 越疯狂越好
4. 借力构建 - 在他人想法上延伸
5. 一人一话 - 一次只说一个想法
6. 可视化 - 写出来/画出来
7. 保持专注 - 一个主题
8. 时间限制 - 15-30分钟
```

#### 阶段四：原型 (Prototype)

**目标**: 快速验证想法

**原型层级**:

```
低保真 ←─────────────────────────────→ 高保真

草图 → 纸质原型 → 线框图 → 可交互原型 → 高保真原型
 │        │         │          │            │
1小时    2小时     半天        1-2天        3-5天
```

**原型设计原则**:

```
1. 快速制作 - 几小时而非几天
2. 低成本 - 材料简单、工具通用
3. 可抛弃 - 不舍不得扔掉
4. 聚焦核心 - 只验证关键假设
5. 可迭代 - 根据反馈快速修改
```

#### 阶段五：测试 (Test)

**目标**: 获取用户反馈

**测试方法**:

| 方法 | 适用阶段 | 时间 | 深度 |
|------|----------|------|------|
| **可用性测试** | 原型/成品 | 30-60分钟/人 | 深 |
| **A/B测试** | 成品 | 1-2周 | 广 |
| **游击测试** | 原型 | 5-10分钟/人 | 浅 |
| **远程测试** | 原型/成品 | 15-30分钟/人 | 中 |

**测试任务设计**:

```markdown
任务格式：
场景：[用户所处情境]
任务：[用户需要完成的目标]
预期：[用户应该完成的步骤]

示例：
场景：您刚买了一台新电视，想要设置并观看节目
任务：请完成电视的初始设置，并找到一个您想看的节目
预期：
1. 连接电源和天线
2. 开机并选择语言
3. 扫描频道
4. 浏览节目单并选择节目
```

### 2.2 各阶段详解

#### Discover 发现阶段

**目标**: 广泛探索，发现机会

**核心活动**:

```
1. 市场研究
   ├── 市场规模分析
   ├── 市场趋势分析
   └── 用户需求调研

2. 用户研究
   ├── 用户访谈
   ├── 用户观察
   ├── 问卷调研
   └── 数据分析

3. 竞品分析
   ├── 直接竞品
   ├── 间接竞品
   └── 潜在竞品

4. 技术趋势
   ├── 新技术调研
   ├── 技术可行性
   └── 成本估算
```

#### Define 定义阶段

**目标**: 聚焦问题，定义方向

**核心活动**:

```
1. 洞察提炼
   ├── 从数据中发现模式
   ├── 识别关键痛点
   └── 发现创新机会

2. 问题定义
   ├── POV陈述
   ├── HMW问题
   └── 设计挑战

3. 范围界定
   ├── In Scope (做什么)
   ├── Out of Scope (不做什么)
   └── MVP定义

4. 成功标准
   ├── 可量化指标
   ├── 验收标准
   └── 时间节点
```

#### Develop 开发阶段

**目标**: 发散方案，迭代优化

**核心活动**:

```
1. 方案构思
   ├── 头脑风暴
   ├── 设计冲刺
   └── 方案评估

2. 原型设计
   ├── 低保真原型
   ├── 中保真原型
   └── 高保真原型

3. 用户测试
   ├── 可用性测试
   ├── 迭代优化
   └── 方案定型

4. 技术验证
   ├── 技术可行性
   ├── 性能验证
   └── 成本核算
```

#### Deliver 交付阶段

**目标**: 完善方案，交付落地

**核心活动**:

```
1. 方案细化
   ├── 设计规范
   ├── 交互规范
   └── 技术规范

2. 开发实施
   ├── 开发排期
   ├── 进度跟踪
   └── 质量把控

3. 测试验收
   ├── 功能测试
   ├── 性能测试
   └── 用户验收

4. 发布上线
   ├── 发布计划
   ├── 灰度发布
   └── 效果监控
```

```
│  • 动效设计、微交互                                           │
├─────────────────────────────────────────────────────────────┤
│  骨架层 (Skeleton Plane)                                     │
│  ─────────────────                                           │
│  信息设计、交互设计、导航设计                                 │

├─────────────────────────────────────────────────────────────┤
```

### 3.2 各层设计指南

#### 战略层设计

**核心问题**: 我们要什么？用户要什么？

**设计输出**:

```markdown
## 产品目标
- 商业目标：[营收目标/市场份额/品牌价值]
- 品牌标识：[产品定位/品牌形象/差异化]

## 用户需求
- 目标用户：[用户画像]
- 核心需求：[用户想要解决的问题]
- 使用场景：[用户在什么情况下使用]

## 成功指标
- 业务指标：[GMV/DAU/转化率]
- 用户指标：[NPS/满意度/留存率]
```

#### 范围层设计

**核心问题**: 做什么？不做什么？

**设计输出**:

```markdown
## 功能规格
| 功能 | 优先级 | 描述 | 依赖 |
|------|--------|------|------|
| 用户注册 | P0 | 支持手机号/邮箱注册 | 无 |
| 搜索功能 | P0 | 支持关键词搜索 | 无 |
| 商品详情 | P0 | 展示商品信息 | 搜索功能 |

## 内容需求
| 内容类型 | 来源 | 更新频率 | 负责人 |
|----------|------|----------|--------|
| 商品信息 | SAP | 实时 | 运营 |
| 用户评价 | 用户 | 实时 | 运营 |
| 营销文案 | 运营 | 每周 | 运营 |

## Out of Scope
- 社交功能
- 直播功能
- 国际化
```

#### 结构层设计

**核心问题**: 如何组织？如何交互？

**设计输出**:

```markdown
## 信息架构
├── 首页
│   ├── 推荐商品
│   ├── 分类导航
│   └── 活动入口
├── 商品分类
│   ├── 一级分类
│   └── 二级分类
├── 商品详情
│   ├── 基本信息
│   ├── 规格选择
│   └── 评价信息
└── 我的
    ├── 订单
    ├── 收藏
    └── 设置

## 核心流程
购物流程：
首页 → 搜索/分类 → 商品详情 → 加入购物车 → 结算 → 支付 → 完成
```

#### 骨架层设计

**核心问题**: 如何呈现？如何操作？

**设计输出**:

```markdown
## 页面布局
- 首页布局：顶部导航 + Banner + 推荐区 + 分类入口
- 列表页布局：筛选区 + 商品列表 + 分页
- 详情页布局：商品图 + 信息区 + 操作区 + 推荐区

## 交互方式
- 手势交互：下拉刷新、上拉加载、左滑删除
- 操作反馈：加载动画、成功提示、错误提示
- 快捷操作：一键购买、快速加入购物车

## 导航设计
- 底部Tab导航：首页/分类/购物车/我的
- 面包屑导航：显示当前位置
- 返回导航：返回上一页
```

#### 表现层设计

**核心问题**: 看起来如何？感觉如何？

**设计输出**:

```markdown
## 视觉规范
- 配色方案：主色 #2563EB / 辅色 #22C55E / 中性色 #6B7280
- 字体规范：标题 24px / 正文 14px / 辅助 12px
- 图标风格：线性图标 / 圆角风格 / 2px线宽
- 图片风格：实景图 / 白底图 / 16:9比例

## 设计原则
- 一致性：相同功能使用相同视觉语言
- 清晰性：信息层级分明、重点突出
- 易用性：操作简单、反馈及时
- 美观性：符合品牌调性、视觉舒适

## 动效规范
- 过渡时长：150ms（快速）/ 200ms（常规）/ 300ms（慢速）
- 缓动函数：ease-in-out
- 触发条件：hover / click / scroll
```

## 四、产品规划方法

### 4.1 产品矩阵

**波士顿矩阵**:

```
        市场增长率
           高
            │
   Stars    │   Question Marks
  (明星产品)│  (问题产品)
   投资扩张 │   选择性投资
            │
────────────┼────────────  相对市场份额
            │
  Cash Cows │   Dogs
  (现金牛)  │  (瘦狗产品)
   维持收割 │   退出转型
            │
           低

策略：
- Stars → 投资扩张，保持优势
- Cash Cows → 维持现状，收割利润
- Question Marks → 选择性投资或放弃
- Dogs → 退出或转型
```

## 五、MVP设计方法

### 5.1 MVP定义

**MVP = Minimum Viable Product** 最小可行产品

**核心原则**:

```
1. 最小 (Minimum)
   - 最少功能集
   - 最小投入
   - 最快上线

2. 可行 (Viable)
   - 核心价值可验证
   - 用户愿意使用
   - 商业模式可行

3. 产品 (Product)
   - 不是原型
   - 不是Demo
   - 是真实产品
```

### 5.2 MVP功能选择

**功能筛选矩阵**:

```
                 用户价值
                   高
                    │
    必须包含        │     优先包含
    (Must Have)    │    (Priority)
    ───────────────┼──────────────
    可选包含        │     暂不包含
   (Optional)      │    (Defer)
                    │
                   低
    ────────────────────────
           实现难度
           低 → 高
```

### 5.3 MVP验证指标

```markdown
## 核心指标
| 指标 | 定义 | 目标值 | 测量方法 |
|------|------|--------|----------|
| 激活率 | 完成核心流程的用户比例 | ≥40% | 埋点统计 |
| 留存率 | 次日/7日/30日留存 | 次日≥30% | 埋点统计 |
| 使用频率 | DAU/MAU | ≥20% | 埋点统计 |
| NPS | 净推荐值 | ≥30 | 问卷调研 |

## 辅助指标
- 注册转化率
- 功能使用率
- 平均使用时长
- 用户反馈数量
```
