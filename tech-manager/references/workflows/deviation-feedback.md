# 实现偏差反馈流程

**何时使用**: 开发过程中或实现验证后，发现代码实现与 PRD 存在偏差，需要反馈给产品经理确认。

---

## 触发条件

以下任一情况出现时，启动偏差反馈：

| 触发场景 | 示例 |
|---------|------|
| 技术限制导致方案调整 | 某字段不支持实时计算，改为异步批处理 |
| 发现 PRD 逻辑缺陷 | 审批流转存在死锁、状态机不完整 |
| 接口与 PRD 字段不完全对应 | 合并/拆分字段、类型调整 |
| 实现验证发现功能偏差 | 验证报告中存在"不通过"项 |
| 新增技术约束 | 性能限制、安全约束、第三方依赖限制 |
| 开发过程中主动优化 | 简化流程、合并步骤、增加自动化 |

---

## 偏差识别流程

### 一、功能偏差识别

在实现验证阶段，逐项对照 PRD 功能清单检查实现情况：

```
功能偏差识别流程：
├── 对照 PRD 功能清单
│   ├── 逐项检查功能是否实现
│   ├── 检查实现方式是否符合 PRD 描述
│   └── 检查功能边界是否正确
│
├── 功能偏差类型
│   ├── 功能未实现：PRD 要求的功能完全缺失
│   ├── 功能部分实现：功能存在但不完整
│   ├── 功能超出 PRD 范围：实现了 PRD 未要求的功能
│   └── 功能实现方式偏差：实现方式与 PRD 描述不符
│
└── 输出偏差清单
    ├── 记录偏差项编号
    ├── 标注 PRD 功能 ID
    └── 描述偏差具体内容
```

**功能偏差识别示例**：

| PRD 功能 ID | PRD 要求 | 实际实现 | 偏差类型 |
|------------|---------|---------|---------|
| F0101 | 商品列表支持多条件筛选 | 仅支持名称筛选 | 功能部分实现 |
| F0102 | 商品详情显示库存预警 | 未实现预警显示 | 功能未实现 |
| F0103 | 商品编辑支持批量修改 | 实现了批量删除（PRD 未要求） | 功能超出 PRD 范围 |

### 二、接口偏差识别

对照 OpenAPI 接口规格文档检查接口实现：

```
接口偏差识别流程：
├── 对照 OpenAPI 文档
│   ├── 检查接口路径是否一致
│   ├── 检查请求参数是否完整
│   ├── 检查返回值结构是否匹配
│   └── 检查接口是否缺失
│
├── 接口偏差类型
│   ├── 参数变更：参数新增/删除/类型变更/必填性变更
│   ├── 返回值变更：字段新增/删除/结构变更
│   ├── 接口缺失：PRD 要求的接口未实现
│   ├── 接口新增：实现了 PRD 未要求的接口
│   └── 接口路径变更：路径与 OpenAPI 不一致
│
└── 输出接口偏差清单
    ├── 记录接口路径
    ├── 标注 OpenAPI 定义
    └── 描述偏差具体内容
```

**接口偏差识别示例**：

| 接口路径 | OpenAPI 定义 | 实际实现 | 偏差类型 |
|---------|-------------|---------|---------|
| GET /api/product/page | 返回 total、list | 仅返回 list | 返回值变更 |
| POST /api/product/create | 参数：name、price、stock | 参数：name、price（缺少 stock） | 参数变更 |
| GET /api/product/{id}/detail | OpenAPI 定义存在 | 接口未实现 | 接口缺失 |

### 三、数据模型偏差识别

对照 ER 图和 DML 脚本检查数据库实现：

```
数据模型偏差识别流程：
├── 对照 ER 图 + DML 脚本
│   ├── 检查表结构是否完整
│   ├── 检查字段是否完整
│   ├── 检查字段类型是否正确
│   ├── 检查约束是否正确
│   └── 检查索引是否完整
│
├── 数据模型偏差类型
│   ├── 字段变更：字段新增/删除/类型变更/长度变更
│   ├── 表结构变更：表新增/删除/关联关系变更
│   ├── 约束变更：唯一约束/外键约束/默认值变更
│   ├── 索引变更：索引新增/删除
│   └── 历史数据处理偏差：历史数据处理方式与 DML 不符
│
└── 输出数据模型偏差清单
    ├── 记录表名/字段名
    ├── 标注 ER 图定义
    └── 描述偏差具体内容
```

**数据模型偏差识别示例**：

| 表名 | ER 图定义 | 实际实现 | 偏差类型 |
|------|---------|---------|---------|
| product | 字段：stock_warning（库存预警阈值） | 字段缺失 | 字段变更 |
| product_category | 外键约束：category_id → category.id | 外键未创建 | 约束变更 |
| product | 索引：idx_name（商品名称索引） | 索引缺失 | 索引变更 |

### 四、业务规则偏差识别

对照 PRD 业务规则章节检查逻辑实现：

```
业务规则偏差识别流程：
├── 对照 PRD 业务规则章节
│   ├── 检查状态流转是否正确
│   ├── 检查校验规则是否完整
│   ├── 检查计算规则是否正确
│   ├── 检查权限规则是否正确
│   └── 检查异常处理是否完整
│
├── 业务规则偏差类型
│   ├── 逻辑简化：复杂规则被简化实现
│   ├── 规则缺失：PRD 要求的规则未实现
│   ├── 优先级调整：规则执行顺序与 PRD 不符
│   ├── 边界处理偏差：边界场景处理与 PRD 不符
│   └── 异常处理偏差：异常场景处理与 PRD 不符
│
└── 输出业务规则偏差清单
    ├── 记录规则编号（BR-xxx）
    ├── 标注 PRD 规则描述
    └── 描述偏差具体内容
```

**业务规则偏差识别示例**：

| 规则编号 | PRD 规则描述 | 实际实现 | 偏差类型 |
|---------|-------------|---------|---------|
| BR-001 | 库存低于阈值时自动预警 | 手动触发预警 | 逻辑简化 |
| BR-002 | 商品删除需检查订单关联 | 未实现关联检查 | 规则缺失 |
| BR-003 | 价格修改需审批后生效 | 直接生效 | 优先级调整 |

## 流程步骤

```
1. 识别偏差
   ├── 开发过程中发现 → 开发者向技术经理上报
   └── 实现验证中发现 → 验证报告中的"不通过"项

2. 偏差分类
   ├── 技术限制型 → 技术上无法按PRD实现，需调整方案
   ├── 逻辑缺陷型 → PRD 存在逻辑矛盾或遗漏
   ├── 优化建议型 → 技术侧发现更优实现方式
   └── 约束新增型 → 实现中发现新的限制条件

3. 评估影响
   ├── 对用户体验的影响（高/中/低/无）
   ├── 对功能完整性的影响
   ├── 对后续迭代的影响
   └── 是否有替代方案

4. 确定严重等级
   └── 根据偏差分级标准确定 P0/P1/P2/P3

5. 生成偏差反馈报告
   └── 使用模板 → @../change-templates.md#实现偏差反馈

6. 提交给产品经理确认
   ├── 偏差内容 + 影响评估 + 建议方案
   └── 等待产品经理决策

7. 记录决策结果
   ├── 接受偏差 → 更新PRD对应章节
   ├── 坚持PRD → 开发者按PRD重新实现
   └── 协商调整 → 产品+技术共同确定新方案
```

## 偏差分类详解

### 技术限制型

```
场景：PRD要求实时计算库存，但数据量大无法在200ms内完成
偏差：改为定时任务每5分钟同步，页面展示"约X件"
影响：用户看到的库存可能有5分钟延迟
建议：接受异步方案，页面标注"库存仅供参考"
```

### 逻辑缺陷型

```
场景：PRD定义的审批状态机缺少"撤回"到"草稿"的流转
偏差：开发时发现用户提交后无法撤回修改
影响：用户体验缺失
建议：补充撤回流程（已提交→撤回→草稿）
```

### 优化建议型

```
场景：PRD要求手动选择收货地址，但可以自动定位
偏差：开发时增加了自动定位+手动修改的组合方式
影响：用户体验提升，无负面影响
建议：接受优化方案
```

### 约束新增型

```
场景：使用的第三方支付SDK不支持PRD要求的"组合支付"
偏差：仅支持单一支付方式
影响：MVP阶段组合支付不可用
建议：MVP先上线单一支付，后续迭代支持组合支付
```

## 偏差分级标准

### P0 严重偏差

**定义**：无法实现核心功能，阻塞项目进度。

**判定条件**（满足任一即为 P0）：
- PRD 核心功能（P0 优先级）无法实现
- 技术方案完全无法支撑业务需求
- MVP 核心功能缺失
- 数据安全或系统稳定性存在严重风险

**处理要求**：
- 必须立即上报产品经理
- 阻塞开发进度，等待决策
- 产品经理需立即确认（当日）

**示例**：
- 订单创建功能无法实现
- 支付接口无法对接
- 核心数据表结构无法创建

### P1 重要偏差

**定义**：影响用户核心流程，但可通过替代方案缓解。

**判定条件**（满足任一即为 P1）：
- PRD 重要功能（P1 优先级）实现偏差
- 用户核心操作流程受影响
- 关键业务规则无法正确执行
- 重要接口或字段缺失

**处理要求**：
- 提交偏差反馈报告
- 可先处理其他任务，等待确认
- 产品经理需 1 个工作日内确认

**示例**：
- 商品搜索功能性能不达标
- 审批流程缺少关键节点
- 重要字段类型与 PRD 不符

### P2 一般偏差

**定义**：功能简化或体验差异，不影响核心流程。

**判定条件**（满足任一即为 P2）：
- PRD 一般功能（P2 优先级）实现偏差
- 非核心功能简化实现
- 用户体验细节差异
- 辅助功能缺失

**处理要求**：
- 提交偏差反馈报告
- 可按技术方案先行实现
- 产品经理需 3 个工作日内确认

**示例**：
- 导出功能格式与 PRD 不符
- 列表排序方式简化
- 提示文案与 PRD 不符

### P3 轻微偏差

**定义**：样式或文案差异，不影响功能使用。

**判定条件**（满足任一即为 P3）：
- UI 样式与 PRD 线框图有差异
- 文案与 PRD 描述有差异
- 交互细节优化
- 非功能性改进建议

**处理要求**：
- 提交偏差反馈报告，记录即可
- 按技术方案先行实现
- 产品经理下次迭代评审时确认

**示例**：
- 按钮颜色与线框图不符
- 提示文案措辞调整
- 表格列宽调整

### 处理时效汇总

| 等级 | 定义 | 产品经理响应时效 | 开发处理 |
|------|------|----------------|---------|
| P0 | 核心功能无法实现 | 立即确认（当日） | 阻塞等待 |
| P1 | 影响用户核心流程 | 1 个工作日内 | 可先做其他任务 |
| P2 | 功能简化或体验差异 | 3 个工作日内 | 按技术方案先行 |
| P3 | 样式或文案差异 | 下次迭代评审 | 按技术方案先行 |

## 产品经理决策流程

产品经理接收偏差反馈报告后的处理流程：

```
产品经理决策流程：
├── 1. 接收偏差反馈报告
│   ├── 来源：tech-manager 提交
│   ├── 格式：偏差反馈报告模板
│   └── 内容：偏差描述、分类、等级、影响、建议方案
│
├── 2. 理解偏差分类
│   ├── 技术限制型 → 技术上无法按 PRD 实现
│   ├── 逻辑缺陷型 → PRD 存在逻辑矛盾或遗漏
│   ├── 优化建议型 → 技术侧发现更优实现方式
│   └── 约束新增型 → 实现中发现新的限制条件
│
├── 3. 评估产品影响
│   ├── 用户体验影响 → 高/中/低/无
│   ├── 业务价值影响 → 是否影响核心功能
│   ├── MVP 影响 → 是否影响 MVP 范围
│   └── 用户接受度 → 用户是否能接受变化
│
├── 4. 产品决策
│   ├── 接受偏差 → 同意技术侧方案，更新 PRD
│   ├── 坚持原需求 → 说明理由，技术侧需重新实现
│   └── 协商调整 → 与技术侧共同确定新方案
│
├── 5. 更新 PRD
│   ├── 标注偏差决策结果
│   ├── 更新相关章节内容
│   └── 添加修订记录
│
└── 6. 生成决策通知
    └── 输出偏差决策通知给 tech-manager
```

### 决策选项说明

| 决策 | 适用场景 | 后续行动 |
|------|---------|---------|
| **接受偏差** | 技术限制型/优化建议型，用户体验影响低 | 更新 PRD，技术侧按新方案继续 |
| **坚持原需求** | 业务价值影响高，核心功能不能妥协 | 说明理由，技术侧需优化方案 |
| **协商调整** | 双方都有合理诉求，需共同确定方案 | 与 tech-manager 讨论，确定折中方案 |

### 决策判断矩阵

| 偏差类型 | 用户体验影响 | 业务价值影响 | 推荐决策 |
|---------|-------------|-------------|---------|
| 技术限制型 | 低 | 低 | 接受偏差 |
| 技术限制型 | 高 | 高 | 协商调整（可能坚持） |
| 逻辑缺陷型 | - | - | 接受偏差（补充逻辑） |
| 优化建议型 | 低/无 | 低/无 | 接受偏差 |
| 优化建议型 | 与设计冲突 | - | 坚持原需求或协商 |
| 约束新增型 | 低 | 低 | 接受偏差（延后） |
| 约束新增型 | 高 | 高 | 协商调整（更换方案） |

## 偏差追踪机制

### 偏差状态流转

```
偏差状态流转图：

上报 → 评估 → 决策 → 执行 → 验证
  │      │      │      │      │
  │      │      │      │      └── ✅ 已解决
  │      │      │      │
  │      │      │      └── 执行中（开发者按决策执行）
  │      │      │
  │      │      └── 决策完成（产品经理已决策）
  │      │
  │      └── 评估完成（tech-manager 已评估影响）
  │
  └── 已上报（tech-manager 已提交反馈）
```

### 状态定义

| 状态 | 定义 | 负责角色 |
|------|------|---------|
| 已上报 | 偏差反馈报告已提交 | tech-manager |
| 评估完成 | 影响评估已完成，等待决策 | tech-manager |
| 决策完成 | 产品经理已做出决策 | product-manager |
| 执行中 | 开发者按决策执行 | 开发者 |
| 已解决 | 偏差已处理并验证通过 | tech-manager |

### status.md 更新规范

**仅在协同开发模式下执行**（检测条件：项目根目录存在 `docs/status.md`）。

偏差反馈提交后，需更新 `docs/status.md`：

|------|------|---------|---------|------|
| 偏差反馈 | product | docs/reports/deviation-product-v1.md | product-manager | 待决策 |

## 协同模式下与 product-manager 的协作

### 协作链路

```
tech-manager（偏差反馈）
       ↓ 偏差反馈报告
       ↓ docs/reports/deviation-{模块}-v{版本}.md
product-manager（偏差处理）
       ↓ 评估产品影响
       ↓ 产品决策
       ↓ 更新 PRD
       ↓ 偏差决策通知
       ↓ docs/decisions/decision-{模块}-v{版本}.md
tech-manager（执行通知）
       ↓ 通知开发者执行
       ↓ 验证执行结果
```

### 文件传递

| 阶段 | 输出文件 | 接收角色 |
|------|---------|---------|
| 偏差反馈 | `docs/reports/deviation-{模块}-v{版本}.md` | product-manager |
| 偏差决策 | `docs/decisions/decision-{模块}-v{版本}.md` | tech-manager |
| 执行确认 | `docs/reports/execution-confirm-{模块}.md` | product-manager |

### product-manager 偏差处理流程

产品经理接收偏差反馈后的完整处理流程见 `@../../product-manager/references/workflows/deviation-handling.md`。

**关键协作点**：
1. tech-manager 提交偏差反馈报告 → product-manager 接收
2. product-manager 评估影响并决策 → 输出决策通知
3. tech-manager 接收决策通知 → 通知开发者执行
4. tech-manager 验证执行结果 → 反馈给 product-manager

### 协同模式状态同步

在协同开发模式下，偏差处理全程需更新 `docs/status.md`：

| 阶段 | 状态更新内容 |
|------|-------------|
| 偏差上报 | 添加"待处理项"，标注阻塞等级 |
| 决策完成 | 移除阻塞项，添加 tech-manager 执行任务 |
| 执行完成 | 移除待处理项，标注偏差已解决 |

详细规范见 `@../../team-collaboration/references/05-status-mechanism.md`。
