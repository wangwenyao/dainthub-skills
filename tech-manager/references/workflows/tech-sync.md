# 技术同步协调流程

**何时使用**: 前后端接口对齐、变更通知分发、技术方案评审。

---

## 流程步骤

```
1. 确定同步类型
   ├── 接口对齐 → 确保前后端接口定义一致
   ├── 变更通知 → 变更后通知相关方
   └── 方案评审 → 评估技术方案可行性

2. 接口对齐
   ├── 对比前端 API 调用与后端接口定义
   ├── 检查字段名、类型、枚举值一致性
   ├── 检查必填/可选字段
   └── 生成对齐报告

3. 变更通知分发
   ├── 识别变更影响方（后端/前端/测试）
   ├── 生成变更通知内容
   └── 使用模板 → @../change-templates.md#变更通知

4. 跟踪同步状态
   └── 记录各方确认状态
```

---

## 接口对齐检查清单

### 字段一致性

```
检查项：
□ 字段名一致（驼峰/下划线转换）
□ 字段类型一致（string/number/boolean/array/object）
□ 枚举值一致（状态码、类型值）
□ 必填/可选一致
□ 默认值一致
```

### 接口契约

```markdown
## 接口对齐报告

### 接口信息
- 接口路径: POST /api/orders
- 用途: 创建订单

### 请求参数对齐

| 字段 | 后端类型 | 前端类型 | 必填 | 状态 |
|------|---------|---------|------|------|
| productId | Long | number | 是 | ✅ 一致 |
| quantity | Integer | number | 是 | ✅ 一致 |
| remark | String | string | 否 | ✅ 一致 |

### 返回值对齐

| 字段 | 后端类型 | 前端类型 | 状态 |
|------|---------|---------|------|
| orderId | Long | number | ✅ 一致 |
| orderNo | String | string | ✅ 一致 |

### 对齐结论
✅ 接口已对齐，前后端可以并行开发
```

## 变更通知模板

### 后端变更通知前端

```markdown
## 接口变更通知

**变更类型**: 字段新增
**影响接口**: GET /api/users/{id}
**变更时间**: 2026-04-02

### 变更内容
返回值新增字段：
- `departmentName` (string): 部门名称
- `roleName` (string): 角色名称

### 前端影响
- 用户详情页：可展示部门、角色名称
- 用户列表页：筛选项可使用新字段

### 兼容性
✅ 向下兼容，原有字段不变

### 同步要求
请前端开发确认是否需要调整页面展示
```

### 前端变更请求后端

```markdown
## 接口调整需求

**需求来源**: 产品需求调整
**期望完成时间**: 2026-04-05

### 调整内容
GET /api/orders 接口返回值需新增：
- `customerName` (string): 客户名称

### 使用场景
订单列表页需要展示客户名称，避免二次查询

### 请后端确认
□ 是否可实现
□ 预计完成时间
□ 是否有技术风险
```
