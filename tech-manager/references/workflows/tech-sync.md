# 技术同步协调流程

**何时使用**: 前后端接口对齐、变更通知分发、技术方案评审。

---

## 流程步骤

```
1. 确定同步类型
   ├── 接口对齐 → 确保前后端接口定义一致
   ├── 变更通知 → 变更后通知相关方
   └── 方案评审 → 评估技术方案可行性

2. 接口对齐
   ├── 对比前端 API 调用与后端接口定义
   ├── 检查字段名、类型、枚举值一致性
   ├── 检查必填/可选字段
   └── 生成对齐报告

3. 变更通知分发
   ├── 识别变更影响方（后端/前端/测试）
   ├── 生成变更通知内容
   └── 使用模板 → @../change-templates.md#变更通知

4. 跟踪同步状态
   └── 记录各方确认状态
```

---

## 接口对齐流程

### OpenAPI 文档审查清单

```
审查项：
□ 接口路径是否符合 RESTful 规范
□ HTTP 方法是否正确（GET/POST/PUT/DELETE）
□ 接口描述是否包含功能ID和业务规则
□ 请求参数是否有完整的类型定义
□ 返回值是否有完整的 Schema 定义
□ 分页接口是否返回 list + total 格式
□ 错误响应是否使用标准模板
□ 是否有权限要求说明
□ 是否有性能要求说明
```

### 字段名称一致性检查

| 检查维度 | 后端命名 | 前端命名 | 转换规则 |
|---------|---------|---------|---------|
| 基本字段 | `productId` (驼峰) | `productId` (驼峰) | 直接对应 |
| 数据库字段 | `product_id` (下划线) | `productId` (驼峰) | 自动转换 |
| 时间字段 | `createdAt` | `createdAt` | 直接对应 |
| 关联字段 | `categoryId` | `categoryId` | 直接对应 |
| 冗余字段 | `categoryName` | `categoryName` | 直接对应 |

```
检查步骤：
1. 对比 OpenAPI Schema 与前端 TypeScript 类型定义
2. 检查命名风格是否统一（驼峰/下划线）
3. 检查字段是否有遗漏或多余
4. 检查字段顺序是否影响解析
```

### 枚举值对齐检查

```
检查项：
□ 枚举值定义是否一致（数值/字符串）
□ 枚举值含义是否一致（状态码、类型值）
□ 默认值是否一致
□ 前端展示映射是否正确

示例：
后端定义：status ENUM(0, 1, 2) // 0草稿 1上架 2下架
前端定义：status: 0 | 1 | 2    // 0草稿 1上架 2下架
前端映射：{ 0: '草稿', 1: '上架', 2: '下架' }
```

### 请求/响应格式对齐

| 格式类型 | 后端格式 | 前端格式 | 对齐要点 |
|---------|---------|---------|---------|
| 分页请求 | `pageNo, pageSize` | `pageNo, pageSize` | 参数名一致 |
| 分页响应 | `{ list, total }` | `{ list, total }` | 结构一致 |
| 时间格式 | `yyyy-MM-dd HH:mm:ss` | `Date` 或 `string` | 格式约定 |
| 金额格式 | `BigDecimal` | `number` | 精度处理 |
| 文件上传 | `MultipartFile` | `FormData` | 类型对应 |

```
请求参数对齐检查：
□ 路径参数（Path）是否一致
□ 查询参数（Query）是否一致
□ 请求体（Body）字段是否一致
□ 必填/可选标记是否一致
□ 参数校验规则是否一致

响应值对齐检查：
□ 成功响应字段是否一致
□ 失败响应格式是否一致
□ 空值处理方式是否一致
□ 时间字段格式是否一致
```

### 错误码对齐

| 错误码 | 后端定义 | 前端处理 | 对齐要点 |
|-------|---------|---------|---------|
| 400 | 参数校验失败 | 表单错误提示 | errors 字段格式 |
| 401 | 未登录 | 跳转登录页 | 自动处理 |
| 403 | 无权限 | 权限提示 | 提示文案 |
| 404 | 数据不存在 | 空状态展示 | 处理方式 |
| 500 | 系统异常 | 错误提示 | 日志记录 |

```
错误响应格式对齐：
{
  "code": 400,
  "message": "参数校验失败",
  "errors": [
    { "field": "name", "message": "名称不能为空" }
  ]
}

前端处理：
- code: 用于判断错误类型
- message: 用于全局提示
- errors: 用于表单字段错误提示
```

## 变更通知流程

### 变更通知模板

#### 后端变更通知前端

```markdown
## 接口变更通知

**变更类型**: 字段新增
**影响接口**: GET /api/users/{id}
**变更时间**: 2026-04-02
**变更版本**: v1.2.0

### 变更内容
返回值新增字段：
- `departmentName` (string): 部门名称
- `roleName` (string): 角色名称

### 前端影响
- 用户详情页：可展示部门、角色名称
- 用户列表页：筛选项可使用新字段

### 兼容性
✅ 向下兼容，原有字段不变

### 同步要求
请前端开发确认是否需要调整页面展示
确认截止时间：2026-04-05
```

#### 前端变更请求后端

```markdown
## 接口调整需求

**需求来源**: 产品需求调整
**期望完成时间**: 2026-04-05

### 调整内容
GET /api/orders 接口返回值需新增：
- `customerName` (string): 客户名称

### 使用场景
订单列表页需要展示客户名称，避免二次查询

### 请后端确认
□ 是否可实现
□ 预计完成时间
□ 是否有技术风险
□ 是否需要数据迁移
```

### 影响范围通知

```
影响范围识别：
1. 直接影响
   ├── 调用该接口的页面
   ├── 使用该字段的组件
   └── 相关的业务流程

2. 间接影响
   ├── 关联接口的变化
   ├── 数据库表的变化
   └── 权限配置的变化

3. 影响方识别
   ├── 后端开发：接口实现方
   ├── 前端开发：接口调用方
   ├── 测试：功能验证方
   └── 产品：需求确认方
```

### 版本兼容性通知

| 兼容类型 | 说明 | 通知内容 |
|---------|------|---------|
| 向下兼容 | 新增字段，原有不变 | 可直接升级，无需前端改动 |
| 不兼容变更 | 字段删除/类型变更 | 需要前端同步改动 |
| 废弃字段 | 字段标记废弃 | 提示前端迁移，设置废弃期限 |
| 版本过渡 | 新旧版本并存 | 提供版本切换方案 |

```
兼容性通知模板：

## 版本兼容性说明

**变更类型**: 不兼容变更
**废弃字段**: `oldField` (string)
**替代字段**: `newField` (object)
**废弃期限**: 2026-05-01

### 迁移指南
- 前端需将 `oldField` 替换为 `newField.value`
- 2026-05-01 后 `oldField` 将不再返回

### 过渡方案
- v1.1: 同时返回 `oldField` 和 `newField`
- v1.2: 仅返回 `newField`，`oldField` 废弃
```

## 技术方案评审流程

### 评审检查清单

```
完整性检查：
□ 方案是否覆盖所有需求点
□ 是否有遗漏的功能场景
□ 是否有未处理的边界情况
□ 是否有明确的验收标准

可行性检查：
□ 技术选型是否合理
□ 是否有技术风险点
□ 是否有性能瓶颈
□ 是否有安全风险

一致性检查：
□ 是否与现有架构一致
□ 是否与现有代码风格一致
□ 是否与现有接口规范一致
□ 是否与现有数据模型一致
```

### 方案一致性验证

| 验证维度 | 检查内容 | 验证方法 |
|---------|---------|---------|
| 架构一致性 | 是否符合分层架构 | 对照架构文档 |
| 接口一致性 | 是否符合 RESTful 规范 | 对照 OpenAPI 规范 |
| 数据一致性 | 是否符合 ER 模型 | 对照数据库设计 |
| 代码一致性 | 是否符合编码规范 | 对照代码规范文档 |

```
一致性验证步骤：
1. 提取方案中的关键设计点
2. 对照现有架构/规范文档
3. 识别不一致的设计点
4. 评估不一致的影响
5. 提出调整建议
```

### 性能/安全评估

#### 性能评估

```
性能检查项：
□ 接口响应时间是否满足要求（≤500ms P95）
□ 数据库查询是否有性能风险
□ 是否需要新增索引
□ 是否有 N+1 查询问题
□ 是否需要缓存策略
□ 是否有批量操作优化
□ 是否有分页限制（≤100条）
```

#### 安全评估

```
安全检查项：
□ 是否有权限校验
□ 是否有参数校验
□ 是否有 SQL 注入风险
□ 是否有 XSS 风险
□ 是否有敏感数据暴露
□ 是否有接口幂等性
□ 是否有并发控制
□ 是否有日志记录
```
