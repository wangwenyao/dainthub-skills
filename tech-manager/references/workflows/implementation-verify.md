# 实现验证流程

**何时使用**: 开发完成后对照 PRD 验证代码实现是否满足需求规格。

---

## 流程步骤

```
1. 获取验证输入
   ├── PRD文档（功能需求、业务规则、验收标准）
   ├── 接口文档（后端已实现的接口）
   └── 代码实现（前端页面、后端服务）

2. 功能完整性验证
   ├── PRD定义的功能是否都有对应接口
   ├── 前端页面是否覆盖所有功能入口
   └── 检查是否有遗漏功能

3. 接口一致性验证
   ├── 前端调用的接口是否在后端存在
   ├── 接口参数是否与定义一致
   └── 返回值是否被正确使用

4. 字段完整性验证
   ├── PRD定义的字段是否在接口中体现
   ├── 数据库字段是否与接口对应
   └── 前端表单字段是否与接口参数对应

5. 业务规则验证
   ├── 校验规则是否实现
   ├── 状态流转是否正确
   └── 权限控制是否到位

6. 测试质量验证（新增）
   ├── 测试覆盖率是否达标
   ├── 是否遵循 TDD 流程
   └── 边缘情况是否覆盖

7. 输出验证报告
   └── 列出符合项、缺失项、风险项
```

---

## 验证维度

### 功能完整性

```
验证方法：
1. 提取 PRD 功能清单
2. 检查后端接口覆盖情况
3. 检查前端页面覆盖情况

检查项：
□ 每个 P0 功能都有对应接口
□ 每个功能都有前端入口
□ CRUD 操作是否完整
□ 搜索/筛选/排序/分页是否实现
```

### 接口一致性

```
验证方法：
1. 提取前端 API 调用列表
2. 对比后端接口定义
3. 检查参数和返回值

检查项：
□ 前端调用的接口路径在后端存在
□ 请求参数名称、类型一致
□ 返回值字段名称、类型一致
□ 错误码处理一致
```

### 字段完整性

```
验证方法：
1. 提取 PRD 字段定义
2. 检查数据库表结构
3. 检查接口参数/返回值
4. 检查前端表单字段

检查项：
□ PRD 定义的字段在数据库中存在
□ 数据库字段在接口中有体现
□ 接口字段在前端有使用
□ 字段类型、长度、必填一致
```

### 业务规则

```
验证方法：
1. 提取 PRD 业务规则（BR编号）
2. 检查后端代码实现
3. 检查前端校验

检查项：
□ 字段校验规则（长度、格式、范围）
□ 状态流转条件
□ 权限控制逻辑
□ 异常处理逻辑
```

### 测试质量（新增）

```
验证方法：
1. 使用 test-generation skill 分析测试覆盖率
2. 检查测试文件是否存在且结构正确
3. 验证是否遵循 test-driven-development 流程

检查项：
□ Service 层单元测试覆盖率 ≥ 80%
□ 新增代码有对应测试文件
□ 核心业务逻辑有测试覆盖
□ PRD验收标准转化为测试场景
□ 边缘情况（空值、边界值、异常）有覆盖
□ 测试命名规范（{方法名}_{场景}_{预期结果}）

覆盖率标准：
| 模块 | 行覆盖率 | 分支覆盖率 |
|------|---------|-----------|
| Service | ≥ 80% | ≥ 70% |
| Mapper | ≥ 60% | ≥ 50% |
| Controller | ≥ 70% | ≥ 60% |
| 前端组件 | ≥ 70% | — |
```

## 与测试 Skill 协作

实现验证阶段调用测试能力：

| 验证场景 | 调用测试 Skill | 目的 |
|---------|---------------|------|
| 覆盖率分析 | `test-generation` | 分析代码覆盖率缺口 |
| 测试流程检查 | `test-driven-development` | 验证是否遵循 TDD |
| 边缘情况补充 | `test-generation` | 发现未覆盖的边缘场景 |

### 验证流程中的测试调用

```
Step 5（业务规则验证）完成后：
├── 若后端有新增Service方法
│   └── 调用 test-generation 分析 Service 覆盖率
│   └── 检查是否达到 80% 覆盖率标准
│
├── 若前端有新增组件/页面
│   └── 检查是否存在 __tests__/*.spec.ts 文件
│   └── 检查测试是否覆盖组件 Props/Events/状态
│
└── 若覆盖率不达标
    └── 输出缺失测试清单
    └── 建议开发者补充测试
```

```markdown
### 问题清单

| # | 类型 | 描述 | 影响范围 | 建议 |
|---|------|------|---------|------|
| 1 | 功能缺失 | 批量导出功能未实现 | 订单管理 | 补充接口 |

| 3 | 规则缺失 | 手机号格式未校验 | 用户注册 | 前端补充校验 |
| 4 | 测试缺失 | OrderService 覆盖率仅 45% | 订单模块 | 补充单元测试 |
| 5 | 测试缺失 | 结算流程无 E2E 测试 | 前端 | 补充 Playwright 测试 |

### 验证结论
⚠️ 部分通过，需解决 5 个问题后方可上线

### 后续行动

**开发补充**（开发遗漏，无需反馈产品）：
- [ ] 后端补充批量导出接口
- [ ] 后端接口新增 remark 字段
- [ ] 前端补充手机号校验
- [ ] 后端补充 OrderService 单元测试（覆盖率需达 80%）
- [ ] 前端补充结算流程 E2E 测试

**偏差反馈**（技术限制/PRD缺陷，需反馈产品确认）：
- [ ] [偏差项] → 启动偏差反馈流程 → `workflows/deviation-feedback.md`
```

## 快速验证检查表

```
□ P0 功能全部实现？
□ 接口与前端调用对齐？
□ 数据库字段与 PRD 一致？
□ 状态流转符合 PRD 定义？
□ 权限控制到位？
□ 必填字段有校验？
□ 异常场景有处理？
□ 分页/排序正常？

测试质量（新增）：
□ Service 层有单元测试？
□ Service 测试覆盖率 ≥ 80%？
□ 核心业务规则有测试覆盖？
□ PRD验收标准转化为测试？
□ 边缘情况（null/边界/异常）有测试？
□ 前端组件有测试文件？
□ 关键流程有 E2E 测试？

偏差反馈（新增）：
□ "不通过"项是否区分了开发遗漏 vs 实现偏差？
□ 实现偏差项是否生成偏差反馈报告？
□ 偏差反馈是否已提交产品经理？
```

## 与偏差反馈流程串联

验证完成后，对"不通过"项进行分类处理：

```
验证报告"不通过"项
    ├── 开发遗漏 → 直接分配开发者补充
    │   └── 功能未实现、字段遗漏、校验缺失、测试不足
    │
    └── 实现偏差 → 启动偏差反馈流程
        ├── 技术限制导致方案调整
        ├── PRD逻辑缺陷
        ├── 需求理解偏差
        └── 输出偏差反馈报告 → workflows/deviation-feedback.md
```

**判断标准**：
- 开发者**能按PRD完成**但没做 → 开发遗漏
- 开发者**无法按PRD完成**或**PRD本身有问题** → 实现偏差
