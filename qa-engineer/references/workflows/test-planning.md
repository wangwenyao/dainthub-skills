# 测试计划生成流程

**何时使用**: 编写测试计划，确定测试范围、优先级、资源分配。

---

## 流程步骤

```
1. 获取输入
   ├── PRD文档（功能清单 + 验收标准）
   ├── 功能优先级矩阵（P0/P1/P2）
   └── 技术方案（接口文档 + 数据库设计）

2. 分析测试范围
   ├── 提取 P0 功能 → 必测范围
   ├── 提取 P1 功能 → 重点范围
   ├── 提取 P2 功能 → 可选范围
   └── 排除 Out of Scope 功能

3. 定义测试类型
   ├── 单元测试覆盖范围（Service层核心方法）
   ├── 集成测试覆盖范围（Service + Mapper）
   ├── E2E测试覆盖范围（关键用户旅程）
   └── 性能/安全测试需求（如有）

4. 测试优先级矩阵
   ├── 功能优先级 × 风险等级 → 测试优先级
   └── 高风险 P0 功能 → 测试优先级最高

5. 资源与时间估算
   ├── 测试用例数量估算
   ├── 测试执行时间估算
   └── 测试资源分配

6. 输出测试计划文档
   └── test-plan-{模块名}-v{版本}.md
```

---

## 测试范围分析

### 功能优先级 → 测试覆盖

| 功能优先级 | 测试覆盖要求 | 覆盖率目标 |
|-----------|-------------|-----------|
| **P0** | 必测，100% 覆盖 | ≥ 95% 通过率 |
| **P1** | 重点测试 | ≥ 80% 覆盖 |
| **P2** | 可选测试 | ≥ 60% 覆盖 |

### 风险等级评估

| 风险因素 | 高风险 | 中风险 | 低风险 |
|---------|-------|-------|-------|
| 数据安全 | 涉及敏感数据 | 涉及业务数据 | 仅展示数据 |
| 业务核心 | 核心业务流程 | 辅助功能 | 查询展示 |
| 用户影响 | 所有用户 | 部分用户 | 极少数用户 |
| 技术复杂 | 新技术/复杂逻辑 | 标准实现 | 简单实现 |

### 测试优先级矩阵

| 功能优先级 | 高风险 | 中风险 | 低风险 |
|-----------|-------|-------|-------|
| **P0** | 🔴 必测优先级 1 | 🔴 必测优先级 1 | 🟠 必测优先级 2 |
| **P1** | 🟠 重点优先级 2 | 🟡 重点优先级 3 | 🟡 重点优先级 3 |
| **P2** | 🟡 可选优先级 4 | 🟢 可选优先级 5 | ⚪ 可跳过 |

## 测试类型定义

### 单元测试范围

```
覆盖范围:
├── Service 层核心方法
├── 复杂业务逻辑（状态判断、规则计算）
├── 工具类方法
└── 不覆盖: 简单 CRUD（由集成测试覆盖）

覆盖率目标:
├── Service ≥ 80%
├── 工具类 ≥ 90%
└── 参考: java-backend-dev/references/test-standards.md
```

### 集成测试范围

```
覆盖范围:
├── Service + Mapper 集成
├── 数据库事务正确性
├── 多表关联查询
└── 不覆盖: Controller 层（由切片测试覆盖）

覆盖率目标:
├── Mapper ≥ 60%
└── 关键 SQL 语句全覆盖
```

### E2E 测试范围

```
覆盖范围:
├── 关键用户旅程（登录 → 操作 → 结果）
├── P0 核心业务流程
├── 跨页面操作场景
└── 参考: workflows/playwright-testing.md

覆盖率目标:
├── P0 流程 100% E2E 覆盖
├── P1 流程 ≥ 80% E2E 覆盖
```

## 资源估算方法

### 测试用例数量估算

```
估算公式:
基础用例数 = 功能数 × 平均用例密度

用例密度参考:
├── P0 功能: 5-10 用例/功能（含边界值）
├── P1 功能: 3-5 用例/功能
├── P2 功能: 1-3 用例/功能

复杂度系数:
├── 状态流转功能 × 2.0
├── 多参数组合功能 × 1.5
├── 简单 CRUD × 1.0
```

### 测试执行时间估算

```
估算公式:
执行时间 = 用例数 × 平均执行时间

执行时间参考:
├── 单元测试: 0.5 分钟/用例
├── 集成测试: 2 分钟/用例
├── E2E 测试: 5 分钟/用例
└── 手工测试: 10 分钟/用例

复算因子:
├── 缺陷修复后重测 × 1.5（预留重测时间）
```

```markdown
| 测试类型 | 覆盖范围 | 用例数量 | 执行方式 |

## 四、资源与时间

### 4.1 测试用例估算

- 总用例数: 95
- 必测用例: 50
- 重点用例: 30
- 可选用例: 15

### 4.2 执行时间估算

| 测试类型 | 用例数 | 单用例时间 | 总时间 |
|---------|-------|-----------|-------|
| 单元测试 | 50 | 0.5min | 25min |
| E2E测试 | 10 | 5min | 50min |
| 手工测试 | 15 | 10min | 150min |
| **合计** | - | - | **225min + 重测预留** |

### 4.3 测试资源

| 角色 | 任务 | 时间投入 |
|------|------|---------|
| QA工程师 | 测试设计 + E2E执行 | 2天 |
| 开发 | 单元测试编写 | 1天 |

## 五、测试里程碑

| 里程碑 | 完成标准 | 预计日期 |
|--------|---------|---------|
| 测试设计完成 | 用例文档输出 | T+2 |
| 单元测试完成 | 覆盖率 ≥ 80% | T+3 |
| E2E测试完成 | P0 流程全通过 | T+4 |
| 测试报告输出 | 缺陷统计 + 结论 | T+5 |

---

## 六、风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| 测试时间不足 | P2功能未覆盖 | 降低P2测试密度 |
| 缺陷集中 | 重测时间增加 | 预留30%重测时间 |
| E2E环境不稳定 | 测试阻塞 | 准备备用环境 |

---

## 版本
- v1.0
- 创建: {日期}
- 创建人: QA工程师
```
